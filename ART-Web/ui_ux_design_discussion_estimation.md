# 見積もり機能 UI/UX デザイン検討会

**参加者:**
*   **UI/UXデザイナー**: ユーザー体験、視認性、操作性を重視。「Wow」な体験と使いやすさの両立を目指す。
*   **システムエンジニア (SE)**: 実装実現性、パフォーマンス、データ整合性を重視。
*   **設計士**: 実務での使い勝手、専門用語の正確さ、検算のしやすさを重視。

**ゴール:**
*   使いやすく、分かりやすい画面構成。
*   細かい数値（寸法、ホゾ、余裕、歩留まり）の一覧性が良い画面。
*   「Pro機能」としてのプレミアム感の演出。

## 1. 画面構成と配置場所の議論

**UI/UX**: 現在の画面は「左：パラメータ」「中央：プレビュー」「右：アクション」という構成です。見積もり情報はどこに置くべきでしょうか？

**設計士**: 設計中は常に「概算金額」が見えていてほしいですね。寸法を変えた瞬間に金額が変わるのが、このアプリの面白いところですから。でも、詳細な内訳（どの部材がいくらか）は、必要な時だけ見られれば良いです。

**SE**: 常に計算は走らせられるので、常時表示は可能です。現状の右パネル下部にある「PRO機能（UNLOCKしてね）」のエリアを、そのまま「見積もりサマリー」にするのが自然ですね。

**UI/UX**: 同意です。通常時は「合計金額」と「材積（m³）」だけをシンプルに表示しましょう。そこをクリック（または「詳細」ボタン）すると、詳細画面が展開する形はどうでしょう？

**設計士**: 詳細は一覧性が命です。Excelのような表形式で見たい。部材ごとに「仕上がり寸法」と「木取り寸法（余裕・ホゾ込み）」の両方が確認できないと、プロとしては不安で使えません。

**SE**: なるほど。仕上がり寸法（Editorで入力した値）と、計算用寸法（裏で計算した値）の対比ですね。これを右側の狭いパネルに押し込むのは無理がありそうです。

**UI/UX**: では、詳細ボタンを押したら、**中央のプレビューエリアの下半分から「詳細テーブル」がスライドアップ（ドロワー表示）**するのはどうでしょう？ プレビューを見ながら表も確認できます。

**設計士**: それはいい！ 図面を見ながら「あ、この框が高いんだな」と直感的に分かります。

## 2. 詳細画面（見積もり明細）の設計

**UI/UX**: 詳細画面のレイアウトを詰めましょう。ユーザー要望の「一覧性」を最優先します。

**SE**: 表示すべき項目は以下になります。多すぎませんか？
1.  部材名（縦框、上桟...）
2.  数量
3.  仕上がり寸法 (W x D x H)
4.  **計算用寸法 (W+余裕 x D+余裕 x H+ホゾ+余裕)** ← ここが重要
5.  単体体積
6.  合計体積
7.  単価 (円/m³) ※設定変更可能にしたい
8.  **原価**

**設計士**: 「計算用寸法」は必須です。ここが間違っていると全ての計算が狂うので。特に「ホゾ長」が含まれているかが一目で分からないといけません。

**UI/UX**: 横に長いテーブルになりますね。モダンなデータグリッド（DataGrid）を採用しましょう。
*   ヘッダー固定、スクロール可能。
*   「計算用寸法」は少し薄い色にするか、アイコンで「Calc」と明示して、仕上がり寸法と区別をつける。
*   重要な「原価」は太字で強調。

**SE**: 立米単価（Unit Price）は、プロジェクト全体で一律設定にするか、部材ごとに変えられるようにするか... 設定画面 (`settings.csv`) があるので、基本はデフォルト値参照で、この画面上でも「上書き」できるようにしましょうか。

**設計士**: はい、木材の種類によって単価は変わるので、画面上でサッと変えたいです。

## 3. PRO機能への誘導 (Unlock Flow)

**UI/UX**: 無料ユーザーにはどう見せるかですね。

**SE**: 現在の実装では、計算ロジック自体は動いていますが、表示を隠しています（ぼかしエフェクト）。

**UI/UX**: この「ぼかし」は続けて採用しましょう。「計算はできているけど見えない」というのが一番購買意欲をそそります。
*   **無料版**: 合計金額エリアに「¥ ???,???」と表示し、詳細ボタンには鍵アイコン。
*   **アクション**: クリックすると「Pro版で正確な原価計算をアンロック」というモーダルが出る。
*   （開発段階なので、今回はトグルスイッチで擬似的にUnlockできるようにします）

## 4. 最終決定デザイン案

### A. 常時表示エリア (右パネル下部)
*   コンポーネント: **SummaryCard**
*   内容:
    *   **Total Cost**: ¥ 12,500 (大きな文字)
    *   **Total Volume**: 0.018 m³ (小さな文字)
    *   **Status**: 「概算見積り (税抜)」などの注釈
*   アクション: カード全体がクリッカブル。「詳細を見る」ボタン。

### B. 詳細表示エリア (ボトムドロワー / オーバーレイ)
*   コンポーネント: **EstimationDetailPanel**
*   動作: 下から70%くらいの高さまでスライドアップ。×ボタンで閉じる。
*   内容:
    *   **ヘッダー**: 「材料費内訳」タイトル、CSV出力ボタン（将来用）。
    *   **設定**: 「基準単価: ¥[ 200,000 ] / m³」 ← ここで編集可能。
    *   **メイン表 (DataGrid)**:
        *   行: 縦框、上桟、下桟...
        *   列: 部材名 | 数 | 仕上がり(WxDxH) | **木取り(WxDxH)** | 体積 | 金額
    *   **フッター**: 合計金額（再掲）。

---

**システムエンジニアのメモ (実装方針):**
1.  `EstimationService` で詳細な計算結果オブジェクト（`EstimateResult`）を返す。
    *   Resultには `items: { part: string, rawDim: Dim, calcDim: Dim, volume: number, cost: number }[]` を含める。
2.  Store (`useDoorViewModel`) でこのResultを保持し、変更があるたびに再計算。
3.  UIは `SummaryCard` と `DetailPanel` に分割して実装。
