# AI Development Rules (Antigravity Guidelines)

このドキュメントは、本プロジェクトにおいてAIアシスタント（Antigravity）が開発を行う際に遵守すべきルールをまとめたものです。
AIモデルが変更された場合でも、必ずこのドキュメントを読み込み、一貫性のある開発を行ってください。

## 1. 開発環境の制約 (Windows)
- **コマンドチェーン禁止**: Windows環境（PowerShell/cmd）での互換性を保つため、`&&` 演算子は使用しないでください。コマンドは1行ずつ実行するか、適切なWindows用演算子を使用してください。
- **npmコマンド**: `npm` の代わりに、必ず `npm.cmd` を使用してください。
  - 例: `npm.cmd install`, `npm.cmd run dev`

## 2. アーキテクチャ (MVVM)
- **MVVMパターン**: メンテナンス性と移植性を高めるため、Model-View-ViewModel (MVVM) パターンを意識して実装してください。
  - **View**: UIコンポーネント (React JSX)。ロジックを持たず、ViewModelからデータを受け取って描画する。
  - **ViewModel**: カスタムフック (`use...`) など。Viewの状態管理、イベントハンドリング、Modelとの橋渡しを行う。テスト容易性を確保する。
  - **Model**: ビジネスロジック、データ構造、API通信。純粋なTypeScript関数/クラスとして実装する。

## 3. テスト駆動開発 (TDD)
- **テストファースト**: 新機能を実装する際は、まずテストコードを作成し、それが通るように実装を進めてください。
- **テストの網羅性**: 単体テスト（Unit Test）は、正常系だけでなく異常系や境界値も含めて、しっかりと記述してください。「とりあえず通ればいい」というテストは禁止です。
- **テストの維持**: 既存のテストがパスし続けることを常に確認してください。

## 4. 言語・ドキュメント
- **多言語対応**: システムは日本語と英語の両方に対応できる設計にしてください (`react-i18next` 利用)。
- **ドキュメント言語**: ドキュメント（Plan, Walkthrough含む）は必ず**日本語**で記述してください。これは絶対的なルールです。
- **英語バージョン**: 英語バージョンは最初は作成しないでください。ある程度システムができてきたとき、英語バージョンの作成を依頼します。

## 5. システム仕様
- **仕様書参照**: システムのUI/UX仕様、デフォルト設定、技術仕様については `SYSTEM_SPEC.md` を参照してください。

## 6. デプロイ手順
- **ビルド確認の徹底**: デプロイを実行する前に、必ず最新のビルドが完了していることを確認してください。
  - `npm.cmd run build` を実行
  - ビルドが完全に終了するまで待機（`Exit code: 0` を確認）
  - ビルド完了後に `.\upload.bat` を実行
- **例外**: ユーザーから「ビルド不要」または「アップロードのみ」と明示的に指示された場合は、ビルド確認は不要です。

## 7. コミュニケーション
- **質問への回答優先**: 「聞かせてください」「教えて下さい」などの指示があった場合は、開発を勝手に進めず、まずはその質問への回答に専念して止まってください。コードの修正などは、その後の明確な指示を待ってから実施してください。

## 8. サーバー起動
- **自動起動スクリプト**: サーバー起動が必要な状況（localhost接続エラー時など）では、手動でコマンドを打つのではなく、必ず **SVP ワークフロー (`/verify-start` または `./svp.ps1`)** を使用してください。
- **自律復旧**: SVPはゾンビプロセスの停止やポート解放待ちを自動で行います。"いちいち起動をお願いしない" かつ "環境の問題で止まらない" 開発を実現します。

## 9. ブラウザ検証の手順
AIが自律的にブラウザ検証を行う際の標準手順です。**プロジェクト固有の設定（ポート番号、フォルダパスなど）は `PROJECT_CONFIG.md` を参照してください。**

### 9.1 事前準備：プロジェクト設定の確認
検証前に、以下の情報を `PROJECT_CONFIG.md` または `package.json` から取得してください：
- **開発サーバーのポート番号** (例: 5173, 3000, 8080)
- **アプリケーションのルートディレクトリ** (例: `./src`, `./app`)
- **起動コマンド** (例: `npm.cmd run dev`, `npm.cmd start`)
- **成功メッセージのパターン** (例: `ready`, `started`, `listening`)

### 9.2 サーバー起動確認
**重要**: PowerShellの実行ポリシー制限により、`npm` コマンドは直接実行できません。**必ず `npm.cmd` を使用**してください。

#### Step 1: ポートの状態確認
```powershell
netstat -ano | findstr ":{PORT_NUMBER}"
```
- **`{PORT_NUMBER}`**: プロジェクト設定から取得したポート番号に置き換える
- **結果あり**: サーバー起動済み → Step 3へ
- **結果なし**: サーバー未起動 → Step 2へ

**例**:
```powershell
# ポート5173の場合
netstat -ano | findstr ":5173"

# ポート3000の場合
netstat -ano | findstr ":3000"
```

#### Step 2: サーバー起動
```powershell
# プロジェクトのルートディレクトリで実行
cd {APP_ROOT_DIR}
npm.cmd run {START_COMMAND}
```
- **`{APP_ROOT_DIR}`**: アプリケーションディレクトリ (package.jsonがある場所)
- **`{START_COMMAND}`**: 起動コマンド (通常 `dev` または `start`)
- **WaitMsBeforeAsync**: フレームワークに応じて調整
  - Vite/Webpack: 8000ms 以上
  - Next.js: 10000ms 以上
  - CRA: 15000ms 以上
- **成功確認**: コンソール出力に成功メッセージが表示されることを確認
  - Vite: `VITE v{version} ready`
  - Next.js: `ready - started server`
  - CRA: `webpack compiled successfully`

#### Step 3: ブラウザサブエージェントの実行
```
URL: http://localhost:{PORT_NUMBER}/
```
- **重要**: 接続エラーが発生した場合は、速やかに `./svp.ps1` を再実行して環境をクリーンアップしてください。
- タスク: 具体的な検証手順を明記
- Recording: 動作記録を保存

### 9.3 トラブルシューティング

#### エラー1: PSSecurityException
```
npm : ファイル ... を読み込めません ... PSSecurityException
```
**原因**: PowerShellの実行ポリシー制限  
**解決**: `npm` → `npm.cmd` に変更

#### エラー2: ERR_CONNECTION_REFUSED
```
net::ERR_CONNECTION_REFUSED at http://localhost:{PORT}/
```
**原因**: サーバー未起動  
**解決**: Step 1-2 を実行

#### エラー3: Port already in use
```
Error: listen EADDRINUSE: address already in use :{PORT}
```
**原因**: ポートが既に使用中  
**解決1**: 既存サーバーを使用（正常動作している場合）  
**解決2**: 既存プロセスを停止してから再起動

### 9.4 検証後の処理
- **サーバー停止不要**: 開発継続のため起動したままにする
- **スクリーンショット**: 重要な画面は自動保存される (`{artifact_dir}` 内)
- **Walkthrough更新**: 検証結果を walkthrough.md に記録

### 9.5 プロジェクト固有情報の参照先
- **このプロジェクト**: `PROJECT_CONFIG.md` を参照
- **別プロジェクト**: 各プロジェクトの設定ファイルを確認
