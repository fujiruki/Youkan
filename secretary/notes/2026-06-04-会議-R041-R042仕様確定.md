# 会議: R-041 (複数 Google カレンダー対応) と R-042 (表示範囲拡張＋ lazy load) 仕様確定

開催日: 2026-06-04
出席: 技術T、ビジネスB、UXU、リスクR（AI 4 専門家会議）
主宰: 指揮AI（晴樹さん代理）

## 1. 会議の定義

### テーマ
Youkan の Google カレンダー連携を拡張する 2 つの要望、R-041（複数カレンダー対応）と R-042（表示範囲拡張＋スクロール lazy load）の仕様を実装着手前に確定する。

### ゴール
4 専門家視点で論点・トレードオフ・リスクを洗い出し、Agent 委譲できる粒度の「確定仕様」を作成する。

### 前提条件（覆さない）
1. R-041 初回連携時: 全カレンダー ON（晴樹さん決定）
2. R-041 UI: カレンダー画面のヘッダー周辺に「表示するカレンダー」ボタン、押下でチェックリストがポップオーバー表示。設定画面には置かない（晴樹さん決定）
3. R-042 初期取得: `currentDate ± 6 ヶ月`（計 13 ヶ月）
4. R-042 lazy load: 端から 200px 手前で +3 ヶ月単位の追加読み込み
5. バックエンド技術: PHP 8 + SQLite3、composer 依存なし、curl 直叩き
6. フロント: React + TypeScript + Vite、`useExternalEvents` フック既存
7. Youkan は晴樹さん個人運用（個人スコープで最適化可、共有運用ではない）

## 2. ラウンド 1 — Divergence（独立発言、反論禁止）

### 1巡目

**技術T:**
> R-041 バックエンドは `GET /google/calendars` を新設し、Google `calendarList.list` を 1 リクエスト。レスポンスを `user_google_calendars` テーブルに upsert する。events 取得は **既存 `/google/calendar/events` を改修**、選択カレンダー全件に対して **直列ループ**（curl_multi は使わない、コード簡潔さ優先、カレンダー数は実用上 10 個未満想定）。各カレンダーへの `events.list` 呼び出しは平均 200ms 程度なので 10 個でも 2 秒以内。

**ビジネスB:**
> R-041 は Phase 3 本来の核なので、Phase 2 で 62 件取れていたところを **全カレンダー合算で 300〜500 件規模に増える** ことを想定。コストは API 呼び出しと DB 容量のみで、Youkan は 1 ユーザー運用なので無視可能なレベル。むしろ **「Youkan は晴樹さんのカレンダーすべてを一画面に集約する OS である」** という独自価値が立つ。広告塔として強い。

**UXU:**
> ポップオーバーの中身は **カレンダー名 + 色チップ + チェックボックス** の縦並びリスト。Google のカレンダー色（API レスポンスに `backgroundColor` フィールドあり）をそのまま反映、Youkan 側カスタム色は当面なし。クリック即反映（保存ボタン不要）、外側クリックで閉じる。**「表示するカレンダー」ボタンには ON 数 / 全数（例: 「📅 3/5」）をバッジ表示** すると現状把握が一瞬で済む。

**リスクR:**
> Google Calendar API の quota は 1 ユーザー 1 日 100 万クエリ（公式値）。1 ユーザー運用なら無問題。ただし R-042 の lazy load と R-041 の複数カレンダーを掛け合わせると **「13 ヶ月 × 5 カレンダー = 65 ヶ月分の events.list」** が初回ロードで発生する可能性 → ここが落とし穴。月単位で events を取得するのではなく **範囲指定 1 回 / カレンダー** にすれば 5 リクエストで済む。

### 2巡目

**技術T:**
> R-042 lazy load は **IntersectionObserver** で実装。grid / gantt / timeline で sentinel 要素を「先頭月の手前 200px」と「末尾月の先 200px」に配置、交差したら `loadMore('before' | 'after')` を呼ぶ。state は `useExternalEvents` を改修し、`fromDate / toDate` を Range 型にして「現在ロード済み範囲」を管理。キャッシュは月単位 key（`2026-05`, `2026-06`, ...）に分割し、再取得を抑制。

**ビジネスB:**
> R-042 は **「カレンダーを開いたらそこに 1 年見える、過去にも未来にも自然にスクロールできる」体験を提供する** ことが価値。ただし 1 年分を全部 DOM に並べると、ガント横スクロールで 400 日分のセル × タスク行で重くなる可能性。**段階的に増やす lazy load** を採用する晴樹さん前提が正しい。

**UXU:**
> 表示するカレンダーボタンの **位置はカレンダー画面のヘッダー右側、`完了を表示` トグル / `📊` ボタンの並び**。スマホでは `MobileBottomSheet`（R-033 既存）に統一して縦並びで開く。ポップオーバーは 280px 幅程度、最大 8 件くらいまでスクロールなし、それ以上はスクロール可。

**リスクR:**
> 新規カレンダー出現検知: 晴樹さんが Google でカレンダーを増やしたとき、Youkan 側にいつ反映されるか。**毎回 `calendarList.list` を叩く** とコスト増。**「表示するカレンダー」ボタンを開いた瞬間に list を refresh**（最後の refresh から 5 分以上経過していたら）するのが UX とコストのバランス良い。

### 3巡目

**技術T:**
> DB スキーマ詳細:
> ```sql
> CREATE TABLE user_google_calendars (
>   id INTEGER PRIMARY KEY AUTOINCREMENT,
>   user_id INTEGER NOT NULL,
>   calendar_id TEXT NOT NULL,          -- Google の calendarId (例: primary, ...@group.calendar.google.com)
>   summary TEXT,                       -- カレンダー名
>   description TEXT,
>   color_hex TEXT,                     -- Google の backgroundColor (例: #4285F4)
>   is_enabled INTEGER NOT NULL DEFAULT 1,
>   sort_order INTEGER NOT NULL DEFAULT 0,
>   last_synced_at INTEGER,
>   created_at INTEGER NOT NULL,
>   updated_at INTEGER NOT NULL,
>   UNIQUE(user_id, calendar_id)
> );
> ```
> カレンダーが Google 側で削除されたら、Youkan 側は `is_enabled = 0` に更新し DELETE はしない（履歴保持＋誤削除リカバリ）。

**ビジネスB:**
> R-041 / R-042 ともに **「複雑性を内部に閉じ込め、UI はシンプル」** が肝。チェックリストポップオーバーが 5 件超えるなら「色チップだけタイル状に並べる」コンパクト版も検討するが、晴樹さんは多くて 5-7 カレンダーだろうから縦リストで十分。

**UXU:**
> イベントチップ（カレンダー画面の予定表示）に **左側 4px の色帯** を追加。これで「どのカレンダーの予定か」が瞬時に分かる。タップして詳細モーダルを開いたとき、カレンダー名も表示する。

**リスクR:**
> ポップオーバーで OFF にしたカレンダーの **キャッシュ無効化**。`useExternalEvents` の `cache` Map は `from/to` で keying しているが、R-041 ではこれに「有効カレンダー集合の hash」を追加 key に組み込まないと、ON/OFF 変更後も古いキャッシュを返してしまう。

## 3. ラウンド 2 — Debate（穴指摘、賛成のみ禁止）

### 技術T → ビジネスB へ
> 「全カレンダー合算で 300〜500 件規模」と言うが、たとえば仕事カレンダーが 1 日 20 件の繰り返し予定（朝会・昼会等）を持つと簡単に 1000 件超える。**ガントの日付列 24px に最大 2 件 + `+N`** という R-039 仕様だと潰れる。R-041 ではチップ密度が変わる可能性を UX 側で再設計しないと破綻する。

### ビジネスB → 技術T へ
> 「直列ループで 10 個 2 秒」は楽観的すぎる。Google API のレスポンス時間は分散が大きい（200ms 〜 2 秒）。10 カレンダー直列だと最悪 20 秒。**せめて curl_multi で並列 5 接続** はやるべき。コード簡潔さよりも UX 影響が大きい。

### UXU → リスクR へ
> 「`calendarList.list` をボタン押下時に refresh」というのは、晴樹さんが「あ、カレンダー増やしたかな」と思って毎回ボタンを押すと毎回 API 呼ぶことに。**むしろ 24 時間に 1 回バックグラウンドで refresh** のほうが自然。新規追加は「次の日に気付く」程度で OK では。

### リスクR → UXU へ
> 「左 4px 色帯」は、ガントの日付セルが 24px しかない狭い空間では識別困難。**色帯ではなく、チップの背景色そのものをカレンダー色の薄い tint** にしないと一覧性が落ちる。アクセシビリティ（色覚多様性）も配慮し、色だけで識別させず必ずアイコン or テキストも併用。

### 技術T → リスクR へ
> 「キャッシュ key に有効カレンダー集合 hash」は理屈は合うが、これだと **チェックボックス操作のたびに全部再 fetch** で UX 重い。代案: events を **カレンダー別に分離してキャッシュ**し、UI 側で表示時にフィルタするだけにすれば、ON/OFF 切替は瞬時。fetch は新規カレンダー出現時 or TTL 切れ時のみ。

### UXU → ビジネスB へ
> 「ポップオーバー縦リスト」だけだと、晴樹さんが「全部 OFF」「全部 ON」を一発で切替できない。**「全選択 / 全解除」のクイックアクション** を追加するのが現実的。

### リスクR → 技術T へ
> DB スキーマで `sort_order` を入れているが、誰が並び順を決めるのか。Google の `calendarList` の `summaryOverride` や `position` を尊重するのか、Youkan 独自で並べ替え可にするのか。**Phase 1 では Google 順固定（calendarList の返却順）** にして、並べ替え機能は R-043 とかに切り出すべき。

### ビジネスB → UXU へ
> ガント・タイムラインで色を増やすと、Youkan のタスクとの色被りが発生する。たとえば Youkan で「会社」タスクが青系、Google の仕事カレンダーも青なら混在で見にくい。**「Google 予定の枠は必ず破線」など、Youkan タスクとの差別化要素を色以外で残す** べき。

## 4. ラウンド 3 — Convergence（採用 / 棄却 / トレードオフ）

### 採用される設計

| 領域 | 採用案 | 理由 |
|---|---|---|
| **R-041 バック取得方式** | カレンダー別に curl_multi で並列 5 接続、合算前にカレンダー別にキャッシュ | T と B の妥協点。10 カレンダーでも 2-4 秒で完了 |
| **R-041 DB スキーマ** | T 提案の `user_google_calendars` 採用、ただし `sort_order` は v1 では Google 返却順固定（並べ替え機能は R-043 へ） | R の指摘採用 |
| **R-041 イベントチップの色** | チップ背景を **カレンダー色の薄い tint（10% 不透明度）** + 左 3px の色帯 + 既存 📅 アイコン維持。色だけに依存しない | U と R の合意 |
| **R-041 Youkan タスクとの差別化** | Youkan タスクは「solid な背景＋ボーダーなし」、Google 予定は「tint な背景＋点線ボーダー」 | B 提案採用 |
| **R-041 UI 配置** | カレンダー画面ヘッダー右側、`完了を表示` の隣に「📅 N/M」ボタン。スマホは `MobileBottomSheet`、PC はポップオーバー | U 提案、前提条件と整合 |
| **R-041 ポップオーバー機能** | チェックリスト＋「全選択/全解除」、外側クリックで閉じる、クリック即反映 | U と U vs B の合意点 |
| **R-041 キャッシュ戦略** | events を **カレンダー別に分離キャッシュ**、UI フィルタは即時。`calendarList.list` は 24h バックグラウンド refresh＋ボタン押下時の手動 refresh | T、R、U の合意 |
| **R-041 削除カレンダー** | Google 側で削除 → `is_enabled = 0` + `deleted_at` 設定、行は残す | R 提案採用 |
| **R-042 取得戦略** | 初期 ±6 ヶ月を 1 リクエスト（カレンダーごと 1 回、月単位分割しない）、月単位 key でキャッシュ | T 提案 + R 指摘の融合 |
| **R-042 lazy load** | IntersectionObserver、sentinel が交差したら +3 ヶ月分追加 fetch。ロード中は次月以降のセルにスケルトン表示 | T 提案 |
| **R-042 DOM 上限** | 24 ヶ月（2 年）を超えたら反対側 3 ヶ月を unmount（仮想化）。**ただし v1 では DOM 制限なし、観測してから対応** | B（YAGNI） |

### 棄却される案

| 棄却 | 理由 |
|---|---|
| 直列ループでの events 取得 | パフォーマンス最悪値が許容外（B の指摘） |
| キャッシュ key に「有効カレンダー集合 hash」を組み込む | チェックボックス切替で重い再 fetch 発生（T の代案で解決） |
| カレンダー並べ替え機能 v1 から実装 | スコープ拡大、R-043 として後追い |
| 設定画面に「表示するカレンダー」管理 UI を追加 | 前提条件で「カレンダー画面のみ」、シンプル維持 |
| `calendarList.list` をボタン押下時に毎回 refresh | API 浪費 |

### 受容するトレードオフ

- **新規カレンダー出現検知の遅延 24h**: 即時性 vs API コスト・実装簡潔さ → 後者を選択
- **DOM 上限なし v1**: 仮想化のコスト vs 観測ベースの段階対応 → YAGNI 優先
- **ガント 24px セルでの色チップ視認性**: 縦並び 2 件まで＋ tint 背景で表現、より高密度な日には `+N` で集約
- **`sort_order` 機能なし v1**: Google 返却順固定で許容、後追い R-043 で対応

## 5. 最終結論

### 結論
**Yes（条件付き実装着手 OK）**

### 採用理由
- 前提条件 4 つは技術・UX・ビジネス・リスク全観点から実装可能と確認
- 主要トレードオフ（DOM 上限、並べ替え、新規検知遅延）は晴樹さん 1 ユーザー運用で許容範囲
- R-041 と R-042 は **データレイヤー（useExternalEvents 改修）で交差する** が、それ以外は独立しているため並列実装可能

### 想定リスクと緩和策

| リスク | 影響 | 緩和策 |
|---|---|---|
| 仕事カレンダーで 1 日 20 件超のチップが描画不能 | ガント 24px セルが破綻 | `+N` 表示の閾値を「上限 1〜2 件」に下げて視認性確保 |
| Google API のレスポンス分散で初回ロード 5 秒超 | UX 体感悪化 | curl_multi 並列 5 接続、ローディング表示明示、TTL 15 分キャッシュは維持 |
| lazy load の sentinel が見えない場合発火しない | 古いブラウザ等 | IntersectionObserver fallback として scroll listener も併用 or polyfill |
| キャッシュ無効化漏れ | 古いイベントが表示 | カレンダー別キャッシュ＋ events.list 結果に `etag` を含めて差分検知 |
| Youkan タスク色と Google 色の衝突 | 視認性低下 | Google 予定は点線ボーダー必須、Youkan タスクは実線で差別化 |
