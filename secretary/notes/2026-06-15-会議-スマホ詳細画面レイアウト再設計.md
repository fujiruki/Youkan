# 会議: スマホ詳細画面（DecisionDetailModal）レイアウト/UX再設計

日付: 2026-06-15
形式: AI専門家会議（技術/ビジネス/UX/リスク 4名・3ラウンド）

## テーマ・ゴール・前提

- **テーマ**: スマホ（特に横画面）でのアイテム詳細画面（DecisionDetailModal）のレイアウト/UX再設計
- **ゴール**: 横画面でカレンダーが読める＋全フィールドにアクセス＋判断ボタン常時押下可、を満たす実装可能な設計を1案に決める
- **前提条件**:
  - カレンダー（RyokanGridView）は無限スクロール用の独自 viewport が必須（ページscrollに埋め込めない）
  - 判断ボタン（今日やる/保留/いつかやる）は常時アクセス可能
  - デスクトップ `md:flex-row` レイアウトは壊さない
  - `useIsMobile`=768px、orientation判定は現状なし
  - 主因: 左カラム `h-[45vh]`（md分岐なし）が横画面（低いvh）で約162pxに潰れる（DecisionDetailModal.tsx:694）
  - 実装は sonnet Agent（SdDD）

## 現状の問題（実コードベース）

- モーダル: `w-[98vw] h-[calc(100vh-20px)] flex flex-col`（DecisionDetailModal.tsx:402）
- 中央エリア: `flex-1 flex flex-col md:flex-row`（691行）→ スマホは縦積み
- 左カラム（カレンダー）: `flex-none md:flex-1 ... h-[45vh] md:h-full`（694行）← **横画面で潰れる**
- メモ/Boost/目安期間: カレンダーカラム下部に `flex-none` 固定（719行）→ カレンダーをさらに圧迫
- 右カラム（見積/担当/サブタスク）: `w-full md:w-[320px]`（796行）
- フッター: `h-[64px]` 固定（946行）
- カレンダーの高さ連鎖: SideCalendarPanel(h-full) → DetailQuantityCalendar(flex-1) → RyokanGridView(`h-full overflow-auto`=スクロール実体)

## ラウンド1（Divergence・要点）

- 技術: ①orientation CSSで横画面を2カラム化（低コスト）②メモ等をカレンダーカラムから外し専有化③タブ化は state/保守コスト増・既存トグル干渉
- UX: ①横は「広い・低い」→2カラムが自然②ネストスクロールは最悪・スクロール領域は1画面1つ③判断ボタン常時可視・タブ移動という選択は判断疲れ哲学に逆行
- ビジネス: ①利用頻度次第で最小修正可②現場スマホ横の納期入力は価値大③CSS切替が最安・タブは今は過剰
- リスク: ①回転でUI急変の一貫性リスク②2カラムでも右フィールド列が長大スクロール③判断ボタンが埋もれない footer固定必須

## ラウンド2（Debate・要点）

- landscape2カラムでもフィールド列過多問題は残る（カレンダーのみ解決）
- タブは「混み具合を見ながら納期決定」の文脈を分断＝判断疲れに逆行
- CSS最小だけだと縦画面の貧弱さ（45vh＋メモ圧迫）が残る
- 目安期間ブレイクダウンは量感（青枠の根拠）と密接→完全分離は文脈断、折りたたみが妥当

## ラウンド3（Convergence）— 結論: **Yes（条件付き）**

### 採用案
1. **横画面（landscape）**: orientation / aspect-ratio ベースで `flex-row` 2カラムへ（カレンダー左 `h-full`、フィールド右 独立スクロール）。固定 `h-[45vh]` を廃止しカレンダーを中央エリア高さに追従させる。← 横画面カレンダー視認性の根本解決
2. **カレンダー専有化**: メモ/Boost をフィールド側へ移動。目安期間ブレイクダウンは「折りたたみ（デフォルト畳む）」でカレンダー直下に残し量感との文脈維持。カレンダーカラムの高さを最大化
3. **判断ボタン footer 常時固定**（既存維持）
4. **縦画面（portrait）**: カレンダーに十分な高さ（`flex-1`＋最低高さ確保）、メモ/フィールドはその下の独立スクロール領域に明確分離（縦並びだが各自bounded＝ネストscroll回避）
5. **Google予定の時刻非表示**（別途依頼）で情報密度UP。`ExternalEventChip` に `hideTime` prop追加 → 詳細カレンダー経路で `hideTime={true}`

### 棄却案
- フルなタブ/セグメント化: 文脈分断（判断疲れ）・保守コスト・既存トグル(FULL/COMPACT, VOL ONLY)干渉。将来余地として保留
- 何もしない/CSS最小のみ: 縦画面の貧弱さが残り不十分

### 想定リスクと対応
- 回転でレイアウトが変わる一貫性低下 → 横/縦で最適形が異なるため許容。アニメ無しで素直に切替
- メモ分離の文脈断 → 折りたたみ目安期間で補完
- フィールド列の長大スクロール → 独立スクロール領域として許容（ネストではない）。判断ボタンは footer固定で常時押下可
