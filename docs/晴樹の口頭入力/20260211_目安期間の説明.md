# 目安期間の算出ロジックと工数の関係性（詳細定義）

「目安期間」とは、アイテムに設定された**見積り工数（労働時間）**を、個人の**稼働可能時間（キャパシティ）**に基づき、カレンダー上の**基準日**から過去へ向かって展開した時間的な占有範囲のことです。

---

## 1. 原理原則と用語定義

### 基準日 (Reference Date)
計算の開始地点となる日付です。以下の優先順位で決定されます。
1.  **納期 (Due Date)**: アイテムに「納期」が設定されている場合は、その日が基準日となります。
2.  **マイ期限 (My Deadline)**: 「納期」が未設定（または不確定）で、「マイ期限」が設定されている場合は、その日が基準日となります。
※ 両方が未設定の場合は、目安期間の算出（青枠の表示）は行われません。

### 稼働日チェック (Working Day Check)
システムが過去へ遡る際、その日が「作業を進められる日」かどうかを判定する処理です。以下の**すべての条件**を満たした場合のみ「稼働日」とみなされます。
-   **休日判定のパス**: 土曜日・日曜日、およびシステム共通の祝日リストに含まれていないこと。
-   **キャパシティの存在**: その日の「個別キャパシティ（後述）」が **0分より大きい** こと。

### キャパシティ (Capacity)
特定の1日に割り当て可能な「作業時間（分）」のことです。
-   **基本設定**: ユーザー設定の `defaultDailyMinutes`（通常は480分＝8時間）が適用されます。
-   **例外設定**: 特定の日付に対して個別に設定された値（例：この日は午前中しか働けないため240分、など）がある場合、基本設定を上書きして適用されます。

---

## 2. 算出アルゴリズムの論理フロー

システムは以下の純粋な論理ループによって期間を決定します。

1.  **初期化**: `残工数 = 見積り工数`、`現在日 = 基準日`、`期間リスト = []` とする。
2.  **ループ条件**: `残工数 > 0` である限り実行する。
3.  **判定**: `現在日` に対して「稼働日チェック」を実行する。
    -   **稼働日ではない場合**: 何もしない（工数は減らさない）。
    -   **稼働日の場合**: 
        -   `現在日` を `期間リスト` に追加する。
        -   `当日の割当 = min(残工数, 当日のキャパシティ)` を計算する。
        -   `残工数` から `当日の割当` を差し引く。
4.  **推移**: `現在日` を1日前（`現在日 - 1日`）にずらしてステップ2へ。
5.  **終了**: `残工数` が 0 になった時点でループを抜け、`期間リスト` に含まれる日付をすべて「青枠」で描画する。

---

## 3. 目安期間の計算式（可視化の原理）

カレンダー上でユーザーが目にする「枠の幅（全日数）」は、論理的に以下の式で導き出されます。

$$目安期間（カレンダーの見た目） = \sum_{d \in WorkingDays} 1 + \sum_{d \in Holidays} 1$$

さらに具体的に、工数とキャパシティの関係に分解すると以下のようになります。

$$目安期間 = \text{ceil} \left( \frac{\text{見積り工数}}{\text{稼働日の平均キャパシティ}} \right) + \text{期間内に含まれる非稼働日の数}$$

### この数式が意味すること：
-   **工数が増える**と、分子が大きくなるため期間が伸びます。
-   **1日のキャパを減らす**と、分母が小さくなるため期間が伸びます。
-   **連休を跨ぐ**と、右側の項（非稼働日の数）が加算されるため、カレンダー上の見た目の幅が劇的に伸びます。

---

## 4. 設定数値が計算に与える影響

個人のカレンダー設定における各数値の影響は以下の通りです。

| 設定項目 | 具体的な数値 | 計算への論理的影響 | 視覚的結果 |
| :--- | :--- | :--- | :--- |
| **見積り工数** | 30分 〜 3日(1440分) | `残工数` の初期値を決定。 | 青枠が「埋めるべき総量」を決定する。 |
| **基本キャパ** | 480分 (8h) など | ループ1回あたりに差し引ける最大値を決定。 | 数値が小さいほど、青枠がより多くの日にまたがる。 |
| **特定日のキャパ** | 240分 (4h) など | その対象日だけ、工数の「消化効率」を下げる。 | その日1日分では工数が足りず、さらに前日へと枠が伸びる。 |
| **定休日/祝日** | 土日・振替休日など | 稼働日チェックを強制的に False にする。 | その日付のセルを「ジャンプ」して、さらに前日へ枠を伸ばす原因になる。 |

この論理構造により、ユーザーが「目安時間ボタン」を切り替えた瞬間に、上記すべての変数が再計算され、個人の現実に即した「着手すべき日」がカレンダー上に即座に可視化されます。
