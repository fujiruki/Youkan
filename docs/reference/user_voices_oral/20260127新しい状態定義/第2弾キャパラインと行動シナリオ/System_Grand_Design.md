# JBWOS Global System Grand Design (2026 Edition)

## 0. プロジェクトの真の目的 (Core Vision)
本システムは単なる「業務管理アプリ」ではない。
**「建具屋・藤田晴樹（および同様の悩みを持つ製造業経営者）」が、
終わりのないタスク量と判断の連続による精神的摩耗から解放され、
「人生のIntent（やりたいこと）」と「日々のReality（納期と生活）」を両立させるための
『能力拡張・精神防衛OS』である。**

---

## 1. システム全体構成 (Global Architecture)

システムは **「OSとしてのJBWOS」** を核とし、その上に **「拡張機能としてのプラグイン」** が接続される構造を持つ。

### 1-1. マルチテナント・ハイブリッド構造
ユーザーは以下の2つの顔（Context）をシームレスに行き来する。

| コンテキスト | 役割 | データスコープ |
| :--- | :--- | :--- |
| **Personal (個人)** | 人間としての「藤田晴樹」 | ・個人の生活タスク<br>・Intent（人生の目標）<br>・Daily Capacity（個人の体力） |
| **Company (会社)** | 経営者としての「株式会社デバッグ建具」 | ・顧客情報、案件データ<br>・見積、請求、資金繰り<br>・従業員管理 |

*   **UI表現**: ハンバーガーメニューで「Mode: Personal / Mode: Company」を明示。
*   **データ構造**: `Users` テーブルを中心に、`Tenants`（会社）と `Memberships` で紐付け。

## 2. JBWOS (Judgment-Based Work OS) の中核思想

「判断疲れ」を極限まで減らすための独自メカニズム。

### 2-1. 状態定義 (Status Definition)
`JBWOS_Intent統合_状態再整理_設計ブリーフ` に基づく最新定義。
**「進捗」ではなく「判断（今考えるか・否か）」**で分類する。

| ステータス | 役割 | 定義・振る舞い |
| :--- | :--- | :--- |
| **Inbox** | 判断待ち | 「考えながら入力しない」ための投げ込み口。判断不可ならとりあえずここ。 |
| **Focus** | 今日触る | 「必ず完了させる」ではない。**「今日意識と時間を向ける（触る可能性がある）」**タスクの集合。順序付きキュー。 |
| **Pending** | 保留 | 「今はやらない」と意図的に判断したもの。塩漬け。 |
| **Waiting** | 待ち | 他者・材料・承認待ち。自分の手は離れている。 |
| **Done** | 完了 | ただの出口。振り返り義務なし。 |

※ **Intent** はステータスではなく「フラグ（価値属性）」として直交管理する（後述）。
※ **Active (Now)** はステータスではなく「ポインター」としてサーバー永続化する。

### 2-2. Focusキューとキャパライン (The Defense Line)
「今日やりすぎ」を脳内計算ではなく視覚で防ぐメカニズム。

*   **Focus Queue**:
    *   件数制限なし（Userは減らせないため）。
    *   **Focus Order** による厳格な順序付きリスト。ドラッグ＆ドロップで入替・即時保存。
*   **Capacity Line (キャパライン)**:
    *   **ロジック**: 上から順に `estimateHours` を累積加算。
    *   **判定**: `Daily Capacity`（例: 8h）を超えた地点に**赤線（Overline）**を引く。
    *   **心理効果**: 「禁止」はしない。「はみ出ている現実」だけを可視化し、**「ラインより下は、今日できなくても仕方ない」**という諦めをシステムが肯定する。

### 2-3. ActiveTask (Now) の永続化
*   **定義**: 「今、この瞬間にやっていること」は常に1つ。
*   **実装**: `active_task_id` をUserテーブル等に保存。
*   **体験**: PCで作業中に離席し、移動中にスマホを開くと、同じタスクが「Now」として表示される。コンテキストの完全同期。

## 3. ユニーク機能群 (Key Features)

### 3-1. 量感カレンダー (Quantity Calendar)
「未来の埋まり具合」を直感的な「色（ヒートマップ）」で可視化するシステム。

*   **機能**: 日付セルに数字ではなく「色の濃さ」を表示。
    *   クリック時：その「濃さ」の原因となっている案件（納期）へ、**滑らかな曲線（Pressure Line）**を描画して可視化。
*   **目的**: 「○月○日は空いてますか？」の問いに、スケジュール帳を見ることなく「色の濃さ」だけで即答可能にする。

### 3-2. プロジェクトの有機的成長
*   **Entry**: ハードルは低く。Inboxに「〇〇邸の件」と1行書くだけでプロジェクト化。
*   **Breakdown**: 後から詳細画面で子タスクを足していく。
*   **Manufacturing Spec**:
    *   建具屋特有の「見積＝仕様書」文化を反映。
    *   「原価」「労務費」「利益率」をリアルタイムにシミュレーションしながら売値を決定する（**逆算見積もり機能**）。

## 4. 拡張性と未来 (Plugin & Future)

### 4-1. プラグインアーキテクチャ
「建具屋専用」で終わらせないための汎用設計。

*   **コア機能**: タスク、カレンダー、ユーザー管理。
*   **プラグイン**:
    *   **Tategu Estimator**: 建具専用の見積・木拾い計算。
    *   **Customer Log**: 顧客とのLINE履歴・工事写真管理。
    *   **Financial**: 締め日・繰越金管理・請求書発行。
*   **展開**: 他の製造業（板金、縫製など）向けには、プラグインの入れ替えだけで対応可能にする。

### 4-2. 思想的到達点
目指すのは、ただの効率化ではなく**「人間性の回復」**。
*   **Intentの保護**: 忙殺される日々の中で、「いつかやりたい（Intent）」が常にPanorama Viewで見守ってくれる。
*   **Realityの受容**: やりきれない量のタスクをキャパラインで可視化し、「できない自分」を責めずに「物理的に無理」と納得させる。

このシステム全体が、経営者・技術者としての誇りと、一人の人間としての安らぎを守る「城壁」となる。
