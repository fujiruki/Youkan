
# JBWOS 次世代タスク管理システム 設計思想・状態定義・UI仕様（開発AI向け完全資料）

## 0. この資料の目的
本資料は、JBWOS（仮称）におけるタスク管理システムの**思想・設計判断・状態定義・UI表現・期限設計・判断と作業の分離**を、
開発AIが**設計意図を誤解なく理解し、既存システムと照合しながら実装判断を行える**ようにすることを目的とする。

本資料は以下を含む。
- ユーザー（晴樹）の直接要望・理想像
- 現状システムの整理
- どこを・なぜ変更するのか
- 新しい状態モデルとその意味
- UI表現と内部状態の分離（MVVM前提）
- 具体例（失敗例含む）

省略は行わない。

---

## 1. ユーザーの理想と現実（一次情報）

### 1.1 理想の生活状態
- 寝る前に「明日やること」が決まっている
- 2〜3日後の見通しもなんとなく立っている
- 今週・来週の流れが把握できている
- 割り込みタスクにも対応できる余白がある
- 判断に疲れない

### 1.2 現実の問題
- 朝になって「今日何やるんだっけ？」と焦る
- 判断と作業が混ざり、疲れる
- 今日やろうと思ったができなかった → 自己嫌悪
- タスクは細分化されているが動けないことがある

本システムは「理想に無理に合わせる」のではなく、
**現実を前提にしながら理想に近づく補助線**として設計する。

---

## 2. 基本思想：判断と作業を混ぜない

### 2.1 判断の重さ（Decision Weight）
- 今日やるかどうか考えるしんどさ
- 優先順位をつけるコスト
- 不確実性の高さ

### 2.2 作業の重さ（Work Weight）
- 目安日数（時間）
- 集中力・体力
- 量感カレンダーで扱う対象

**この2軸は絶対に統合しない。**

---

## 3. 期限の定義（重要）

### 3.1 用語定義
- 納期：現実世界の締切（絶対）
- マイ期限：その手前で完了しておきたい自分用期限
- 目安日数：作業量（時間換算）

### 3.2 量感カレンダーの計算仕様
着色期間：  
（マイ期限 − 目安日数 − 1日）〜 マイ期限

#### 具体例
- 納期：1/31
- マイ期限：1/29
- 目安日数：1.5日（12h）
- 1日8h

表示：
- 1/28：4h
- 1/29：8h

**これは「作業開始日」ではなく、
「この期間までに判断を終えていないと危険」という可視化である。**

---

## 4. 永続（Persistent）の定義

### 4.1 永続とは
- 毎日やる
- 期限が消えない
- Doneにならない

例：
- 日記
- 毎日の片付け
- 習慣

### 4.2 扱い
- 通常タスクと同一テーブル
- 状態遷移は限定
- 量感カレンダーには原則影響しない

---

## 5. タスク状態モデル（内部）

### 5.1 状態一覧

| 状態 | 内部名 | 意味 |
|---|---|---|
| Inbox | INBOX | 未判断 |
| Pending | PENDING | 判断を意図的に延期 |
| Waiting | WAITING | 外部待ち |
| Focus | FOCUS | 判断完了・優先意識 |
| Done | DONE | 完了 |

### 5.2 心理的役割

| 状態 | 自分へのメッセージ |
|---|---|
| Inbox | 今は考えない |
| Pending | 今じゃない |
| Waiting | 待っていい |
| Focus | 今日はここを見る |
| Done | 終わった |

**Focusは「今日やるリスト」ではない。  
「今日やるかどうかの判断を終えた置き場」である。**

---

## 6. 状態遷移ルール（重要）

### 6.1 許可される遷移
- Inbox → Pending
- Inbox → Focus
- Pending → Focus
- Focus → Pending（現実対応）
- Focus → Done
- Any → Waiting

### 6.2 禁止・注意遷移
- Inbox → Done（原則禁止）
- Waiting → Done（確認なし禁止）

---

## 7. Focus の再定義

### 7.1 Focusとは何か
- 今日やる「可能性が最も高い」もの
- 判断を終えたもの
- 実行保証はしない

### 7.2 Focusに入れたができなかった場合
- 失敗ではない
- 判断は正しかった可能性がある
- 翌日、再判断するだけ

---

## 8. Inbox / Pending の設計意図

### 8.1 Inbox
- タイトルだけでもOK
- 納期があればバッジ表示
- 危険ではない

### 8.2 Pending
- 判断テンプレを通す
  - いつ判断するか
  - 判断条件
- 未判断が溜まりすぎた場合UIで注意

---

## 9. UI設計（MVVM前提）

### 9.1 内部状態とUI表示の分離
- 内部：State enum
- UI：色・バッジ・配置

### 9.2 アイテムカード
- 1行表示
- タップ＝詳細画面のみ
- 着手中/未着手は別操作（スワイプ等）

### 9.3 Focusビュー
- 親タスク名を右側に表示

### 9.4 パノラマビュー
- インデントで親子関係を可視化
- 忘れていない確認用

---

## 10. 親子タスク（プロジェクト）

- 同一テーブル
- 子がある＝プロジェクト
- 一覧で全階層を見られる

---

## 11. 現実対応設計

### 11.1 今日できなかった
- 状態は変えなくて良い
- 翌日に再判断

### 11.2 やりたいことをやってしまった
- 記録しない
- 自己否定しない

---

## 12. 失敗を前提にした思想

このシステムは、
- 完璧に守られる前提ではない
- 判断ミス・割り込み・疲労を前提とする

**続けられることが最優先である。**
