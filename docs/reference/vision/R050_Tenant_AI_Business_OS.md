- 会議による意思決定の詳細・棄却案・トレードオフは `docs/kaigi/2026-07-09-R050テナント型AI中枢設計.md` を参照。

## 目的

- Youkanを単なる個人タスク管理アプリではなく、**テナントアカウントを持つ会社・チームが、事業の中枢にAIを置いて業務を回すためのOS**に進化させる。
- 藤田建具店だけに閉じた専用機能ではなく、他の小規模事業者・職人業・店舗・制作会社・士業・地域企業などでも使える汎用的な仕組みにする。
- そのうえで、現在の藤田建具店にとってすぐ使いやすい形に落とし込む。

## 背景

- 現在のYoukanは、タスクを登録し、プロジェクト化し、ガント・量感カレンダー・Stream Viewで管理する土台はある。
- しかし、会社運営では「タスクを登録する」だけでは足りない。
- 実際には、案件ごとに複数工程があり、それを誰が、いつ、どの順番で、どの程度の負荷で行うかを決める必要がある。
- 晴樹さん個人の場合、タスク分解まではできても「どのタイミングでやるか」「誰に渡すか」「進んでいないときに誰が支援するか」を決め続けるのが難しい。
- これは藤田建具店固有の問題ではなく、小規模事業者全般にある「代表の頭の中に工程管理が集中している」問題である。

## 基本思想

- AIは人の上司として勝手に命令する存在ではない。
- AIは、案件・タスク・予定・担当者・キャパシティを横断的に読み取り、**次に必要な作業を人に渡せる形へ翻訳する中枢エンジン**とする。
- 最初は「AIが案を作る → 代表または管理者が承認する → 社員に共有する」方式にする。
- 将来的に信頼できる範囲だけ自動化する。

## 汎用ユースケース

- 建具屋: 案件→現場調査→見積→発注→製作→取付→請求
- 工務店: 案件→現調→職人手配→工程管理→施主連絡→請求
- デザイン会社: 案件→ヒアリング→初稿→修正→納品→請求
- 飲食店: 仕込み→予約→発注→当日運営→片付け→売上確認
- 士業: 相談→資料回収→書類作成→確認→提出→請求
- どの業種でも、Youkanは「誰が・いつ・何を・どの負荷でやるか」を管理し、AIが次の動きを提案する。

## 藤田建具店での具体ユースケース

- Beaverで案件を作成する。
- Beaverの案件種別・見積情報・工場時間・現場時間・取付時間・納期を元に、Youkan側に標準工程テンプレートを生成する。
- Youkanで工程を半日ブロック・小タスク・連続製作ブロックに分ける。
- AIが担当者候補を出す。
- 晴樹さんが確認して確定する。
- 社員には「今日やること」だけが分かりやすく共有される。

## Beaverとの責務分離

- Beaverは「案件・顧客・見積・売上・請求状態」の正本。
- Beaverには、案件名、顧客、現場、見積、売上、請求、入金、納期、案件種別、見積明細などを持たせる。
- Beaverにタスク管理機能を深く入れすぎない。
- Beaverは「この案件は何か」「お金はどうなっているか」「案件状態はどこか」を管理する。

## Youkanとの責務

- Youkanは「作業・担当・時間・キャパシティ」の正本。
- Youkanには、案件から生成された工程タスク、担当者、目安時間、目安日数、日付、半日ブロック、依存関係、状態を持たせる。
- Youkanは業種汎用のタスク・工程OSであり、建具専用にはしない。
- 建具向けロジックはプラグインまたはテンプレートとして実装する。

## 番頭・個人補佐AIとの責務分離

- 番頭さんは晴樹さん個人の補佐団の取りまとめ。
- 藤田建具店OS AIは事業側の案件・工程・担当・キャパ管理の中枢。
- 両者は情報共有するが、責務は分ける。
- 例:
  - 藤田建具店OS AI: 「梶本様案件の製作が2日遅れています。松井さんへ下作業を振る案があります」
  - 番頭さん: 「晴樹さんが疲れて進められていないので、今日は判断だけに絞る方がよい」

## 必要な画面1: 会社ディスパッチ・ダッシュボード

- 目的: 管理者が会社全体の作業状況を見る画面。
- 表示内容: 今日動く案件 / 遅れている案件 / 担当未定のタスク / 今日中に判断が必要なタスク / キャパ超過している担当者 / AIからの割り振り提案
- イメージ:
  ```text
  藤田建具店 今日の司令室

  ■ 要判断
  - 梶本様 建具2枚: 金物未確定。晴樹確認が必要
  - 長浜町 障子8枚: 当日返却不可。お客様へ翌日納品連絡が必要

  ■ 担当者別キャパ
  晴樹   520 / 420分  超過
  松井   210 / 360分  余裕あり
  宗三   180 / 240分  適正
  由依子  90 / 180分  余裕あり

  ■ AI提案
  - 障子剥がしは松井さんへ移す
  - 金物LINE確認は由依子さんへ依頼可能
  - 晴樹さんは仕様判断だけに絞る
  ```
- 注意: 会議の少数意見レポートで、この画面イメージ中の「超過」等の評価語が `Capacity_Management_Spec.md` の「審判しない（Non-Judgmental）」原則と文言レベルで矛盾すると指摘された。Phase2着手時に必ず再検討すること。

## 必要な画面3: AI割り振り提案画面（Phase3以降）

- 目的: AIが担当変更案を出し、管理者が承認する。AIは勝手に確定しない。
- 表示内容: 提案前の担当 / 提案後の担当 / 理由 / 工数 / リスク / 承認ボタン / 却下ボタン
- イメージ:
  ```text
  AI割り振り案

  案1: 長浜町 障子剥がし
  現在: 晴樹
  提案: 松井さん
  理由: 手順化済み。晴樹判断不要。松井さんの本日キャパに余裕あり。
  リスク: 建て付け調整判断が必要な場合は晴樹確認へ戻す。
  [承認] [却下] [修正]

  案2: 小松へ金物LINE確認
  現在: 晴樹
  提案: 由依子
  理由: 定型連絡。仕様確定済み。
  [承認] [却下] [修正]
  ```
- 会議決定: AI提案理由（reason）と社員向け引き継ぎメモ（handoff_note）はDBスキーマだけでなくAPIレスポンス・フロント型定義レベルで分離すること。承認は「コミット時再検証」と「承認疲れに対するサーキットブレーカー」を持たせること。承認書き込み経路はシステム権限直書きを禁止し、承認者本人のJWTでACLを通すこと。

## 必要な画面4: 社員向けToday画面（Phase6）

- 目的: 社員が迷わず作業できる画面。管理者向けの複雑な情報は見せない。「今日やる順番」「注意点」「完了ボタン」だけに絞る。
- 画面2（担当者別ビュー）とは統合しない（2026-07-09 kaigi2会議で明確に否決。反証: `docs/requests.md` の元フェーズ計画自体が画面2=Phase1・画面4=Phase6と分離しており、バックエンド配線の再利用という前提も `TodayController.php` に `assignees` とのJOINが存在しないことで崩れた）。
- 依存関係・ブロック分割ロジックの欠落が現場事故（部材未着のまま完了ボタンを押させる等）を招くリスクが会議のプレモーテムで指摘されている。Phase6設計の必須検討事項とすること。
- イメージ:
  ```text
  今日の作業 - 松井さん

  1. 長浜町 障子4枚 剥がし
     注意: 破れた紙はまとめて廃棄。桟を折らない。
     [完了] [困った]

  2. 梶本様 框戸 材料取り
     注意: 寸法指示書は作業台左。
     [完了] [困った]
  ```

- 目的: 旅館カレンダーの考え方を、担当者別の稼働表に展開する。
- 行: 担当者 / 列: 日付または半日コマ / セル: 割り当てタスク、キャパ色、超過警告
- イメージ:

  ```text
  晴樹      仕様確認   製作A     製作A     現調B

  由依子    連絡      請求確認   空き       空き
  ```

## 必要な画面6: プロジェクト内の担当割り当て編集（Phase2以降）

- 目的: 1案件の中で、どの工程を誰が持つかを編集する。既存のプロジェクト管理・フロー内タスク管理とつなげる。
- プロジェクト内タスクに対して、担当者・目安時間・依存関係・作業種別を編集できるようにする。
- 作業種別例: 判断 / 連絡 / 見積 / 発注 / 製作 / 現場 / 取付 / 請求 / 入金確認

## データ設計で追加または整理したい項目（Phase3以降・meta.dispatch案）

- `items.assigned_to` は既に存在するため活用する（Phase1で値域整理済み。§3.8参照）。
- `assignees` テーブルの仕様を明確化する。
- ユーザーアカウントと担当者マスタの違いを整理する。
- `items.meta` に以下を持てるようにする案（**Phase3着手判断の会議で改めて1案に統合設計する。今回は未採用**）:
  - `work_type`: 作業種別
  - `skill_required`: 必要スキル
  - `can_delegate`: 委任可能か
  - `requires_haruki_decision`: 晴樹判断が必要か（会議で汎用命名 `requires_owner_decision` 等への一般化が指摘されている。コストはほぼゼロなので命名決定は早期に行うべきという少数意見あり）
  - `block_type`: small_task / half_day / full_day / continuous_production
  - `source_system`: beaver / manual / ai / google_calendar
  - `source_case_id`: Beaver案件IDなど
  - `handoff_note`: 社員向け注意事項

## AIが考慮する要素（Phase3以降）

- estimated_minutes / work_days / due_date / prep_date / assigned_to / tenant_id / status / project_id / parent_id / item_dependencies
- 日別キャパシティ / Googleカレンダー予定 / 担当者のスキル / 担当者の出勤可否
- 作業が委任可能か / 判断が必要か / 現場場所 / 同日にまとめられる現場か / 外注待ち・材料待ちか

## AIがやってはいけないこと（Phase3以降・恒久ルール）

- 未承認で勝手に担当者を確定しない。
- 納期や顧客への約束を勝手に変更しない。
- 社員へ心理的圧力の強いメッセージを送らない。
- 「できるはず」とだけ判断して余白を潰さない。
- 製作ブロックを細切れにしない。
- 明細単位の取り付け時間をそのまま半日ブロック化しない。取り付けは訪問単位で合算する。

## 藤田建具店向けの重要ルール（Phase3以降・work_type外出し設計時の入力）

- 製作は連続ブロックとして扱う。
- 現場調査は原則半日ブロック。
- 取り付けは明細単位ではなく訪問単位で合算して半日/1日ブロック化する。
- 請求・発注・連絡は半日ブロックではなく小タスク。
- 障子張り替えなど標準化できる業務は社員に渡しやすくする。
- 組子細工・特殊案件・作家性の高い案件は晴樹判断が必要なプロジェクト扱いにする。
- 会議の横展開性専門家の指摘: これらのルールをPhase3実装時にコードへハードコードすると、後からテナント設定への外出し設計に作り直すコストが跳ね上がる。着手時点で最初から設定外出しの構造を選ぶべき。

## 実装フェーズ案（2026-07-09 kaigi2会議で更新）

- Phase 1: `assigned_to` を使った担当者別ビュー（画面2）。前提として管理者スコープ新設・`assigned_to`値域整理を実施（→ 着手中。`docs/SPEC/04_データ設計.md` 参照）
- Phase 2: 担当者別キャパ表示（既存 QuantityService の流用で低リスクと会議で判定）
- Phase 3: AI割り振り提案をMarkdown/JSONで出力（meta.dispatchスキーマ・dispatch_proposalsテーブル・承認ACLをこの段階で1案に統合設計）
- Phase 4: 承認した割り振り案をYoukanへ反映（承認のコミット時再検証・サーキットブレーカー必須）
- Phase 5: Beaver案件IDとYoukanプロジェクトを連携
- Phase 6: 社員向けToday画面（依存関係・ブロック分割ロジックの欠落に注意）
- Phase 7: 番頭・Googleカレンダー・Accessとの連携を拡張

## 成功条件

- 代表が全部覚えていなくても案件が止まらない。
- 社員が「今日何をすればいいか」を迷わない。
- AIが進捗停滞を検知して、代表または補佐役に支援依頼を出せる。
- 藤田建具店だけでなく、他テナントでも同じ仕組みを業種テンプレートとして使える。
- Youkanが「個人の判断疲れを減らすOS」から「小規模事業者の判断疲れを減らすOS」へ拡張される。

## 会議での少数意見（棄却されたが将来効いてくる可能性がある意見）

詳細は `docs/kaigi/2026-07-09-R050テナント型AI中枢設計.md` §6を参照。要約:

- 画面1・5は管理者限定、画面2はデフォルト本人限定というロール分離は、Phase2（キャパ表示）着手時に再検討すべき。
- AI提案のreasonと社員向けhandoff_noteは型レベルで分離すべき（テーブル分離だけでは不十分）。
- 承認は「コミット時再検証」と「承認疲れに対するサーキットブレーカー」がないと安全弁として機能しない。Phase4着手前に必ず設計に組み込む。
- work_type→block_type変換ルールの外出し設計は、Phase3でのハードコード実装時に構造だけは選んでおくべき。
- Phase1完了後、Phase3着手前に晴樹さん本人へ「判断疲れが減ったか」のヒアリングを実施すべき。
