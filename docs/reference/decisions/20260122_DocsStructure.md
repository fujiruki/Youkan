# AI会議_ドキュメント設計.md

## 0. 会議概要
**テーマ**: 持続可能でAIフレンドリーなドキュメント体系の再構築
**ゴール**: 「AIが初見で全体像を理解でき」「変更履歴が追え」「メンテナンスが楽」な構造の定義
**トリガー**: ユーザーからの「システム複雑化に伴うドキュメント管理の相談」

## 1. 前提・制約
1.  **AI-Native**: 新しいAIモデル（コンテキストリセット後）が自走できる情報を残す。
2.  **Human-Friendly**: 人間が見ても分かりやすく、検索性が高いこと。
3.  **Maintenance**: 更新漏れを防ぐ仕組み、または「腐らない」仕組みが必要。
4.  **History**: 「なぜその仕様になったか」の経緯（Context）が重要。

## 2. キャスティング
1.  **Arch (アーキテクト)**: 構造化、Git管理、トレーサビリティ担当
2.  **Ops (運用・保守)**: 更新コスト、陳腐化防止、ルール作り担当
3.  **Dev (開発者AI)**: 読みやすさ、コードとのリンク、コンテキスト理解担当
4.  **Librarian (司書)**: 分類、命名規則、検索性担当

---

## 3. ラウンド1：初期提案 (Divergence)

-   **Dev**: 正直、今の `docs/` はカオスになりつつある。一番困るのは「この機能、仕様書とコードどっちが正解？」という状態。**コードの近くにドキュメントを置く (Co-location)** のが良いのではないか？ 機能フォルダの中に `README.md` を入れるスタイルだ。

-   **Ops**: それは分散しすぎて全体像が見えなくなる諸刃の剣だ。管理側としては `docs/` に集約したい。ただし、現状の「適当にmdを置く」のは限界。**ADR (Architecture Decision Records)** を導入すべき。「決定」をイミュータブル（変更不可）なログとして残すんだ。仕様書を書き換えるのではなく、「変更決定」を積み上げる。

-   **Librarian**: 検索性を考えると、階層は浅い方がいい。さらに、AIが最初に読むべき**地図 (Map)** が必要だ。`docs/README.md` ではなく、`docs/SYSTEM_CONTEXT.md` のような、プロジェクトの「憲法・地図・用語集」をまとめたファイルがルートに欲しい。

-   **Arch**: 同意。構造としては「フロー（経緯）」と「ストック（現在の姿）」を分けるべきだ。
    -   **Stock**: 現在の仕様（随時更新）。例: `System_Specification.md`
    -   **Flow**: 決定履歴（追記のみ）。例: `ADR/001_Auth_Transition.md`, `AI会議_*.md`

---

## 4. ラウンド2：交差レビュー (Debate)

**Dev**: Opsの言うADRは魅力的だが、AI会議の議論ログ (`AI会議_*.md`) とどう使い分ける？
→ **Ops**: AI会議は「議論プロセス」で、ADRは「決定事項」。AI会議の結論部分だけを ADR として採番・保存するのが理想だが、手間なら **「AI会議ログ自体を準ADR」** として扱い、ファイル名に連番をつけるだけでもいい。`docs/decisions/005_Privilege_Model.md` のように。現状の `docs/AI_DEVELOP_RULES` もこの一部だ。

**Arch**: Devの「Co-location（コードの近く）」も捨てがたい。特にUIコンポーネントやプラグインの仕様は、`src/features/plugins/tategu/README.md` にあるべきだ。
→ **Librarian**: 折衷案で行こう。
    -   **全体アーキテクチャ・共通ルール**: `docs/` 直下
    -   **個別機能の詳細**: 各モジュールの `README.md`
    -   **決定ログ**: `docs/decisions/` (または `docs/history/`)
    そして、`docs/INDEX.md` でそれらへのリンク集を作る。

**Ops**: 「変更になりました」をどう見つけるか？
→ **Arch**: ファイルの**Frontmatter (YAMLヘッダ)** を活用しよう。`status: active | deprecated`, `last_updated: 2026-01-22`, `replaces: docs/old_spec.md` のようなメタデータを全ドキュメントに義務付ける。AIはこれを読んで情報の鮮度を判断できる。

## 5. 最終結論 (Convergence)

### 採用するドキュメント構造案

```
PROJECT_ROOT/
├── docs/
│   ├── 00_CONTEXT.md          # [最重要] AI/人間が最初に読む全体地図・用語集・思想
│   ├── 01_RULES/              # 開発ルール (憲法, OPS, Workflow)
│   ├── 10_ARCHITECTURE/       # システム全体の設計思想, DBスキーマ概要
│   ├── 20_DECISIONS/          # (New) 意思決定ログ (AI会議議事録, ADR) - 時系列順
│   │   ├── 20260122_AuthModel.md
│   │   └── ...
│   ├── 30_MANUALS/            # ユーザー向けマニュアル
│   └── ARCHIVE/               # 古くなった仕様書置き場
│
├── src/
│   └── features/
│       └── tategu/
│           └── README.md      # 機能固有の詳細仕様はコードの近くに配置
│
└── task.md                    # 直近の作業進行（現状維持）
```

### 運用ルール（ドキュメントエコシステム）

1.  **00_CONTEXT.md (The Map)**
    -   プロジェクトの目的、技術スタック、主要機能へのリンク、最新の主要な変更点（News）を記載。
    -   「迷ったらここに戻る」場所。

2.  **決定の保存 (ADR / Decisions)**
    -   仕様変更の議論（AI会議）は `docs/20_DECISIONS/` に保存。
    -   ファイル名は `YYYYMMDD_Theme.md`。
    -   重要な決定は、ここを見れば「なぜ変わったか」がわかる。

3.  **「更新」ではなく「追記・参照」**
    -   古い仕様書を直接書き換えて「前の仕様」を消滅させるのは避ける。
    -   大きく変わる場合は `v2` を作るか、DECISIONS に経緯を書いて古いファイルに `status: deprecated` を付与する。

4.  **Frontmatterの義務化**
    -   すべてのmdファイルに以下を付与。
    ```yaml
    ---
    title: ドキュメントタイトル
    status: active # active, draft, deprecated
    last_updated: 2026-01-22
    author: AI
    ---
    ```

### 次のアクション
1.  **Docs Refactor**: `docs/` フォルダを上記構造にリファクタリング。
2.  **CONTEXT.md Creation**: 全体地図を作成。
3.  **Migration**: 既存の `AI会議_*.md` を `20_DECISIONS` へ移動。

会議終了。
