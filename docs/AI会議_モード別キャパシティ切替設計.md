# AI会議_モード別キャパシティ切替設計

## 1. 会議の定義
- **テーマ**: 表示モード（個人/会社/全体）およびフィルタ状態に応じた「キャパシティ算出」の優先順位と切替ルールの策定
- **ゴール**: 
    - どの画面で、どのフィルタが効いているときに、どのキャパシティ（個人合計、会社個別、あるいはその合算）を表示・計算に使用するかを明確にする
    - `QuantityEngine` への入力パラメータの標準化
- **前提条件**:
    - AI開発憲法「現実の尊重」に基づき、ユーザーが「今、何の時間枠で考えているか」に即した表示を行う

---

## 2. キャスティング
1.  **技術・実装担当**: 
    - フィルタ状態（tenantId）を `QuantityEngine` にどう伝えるか。
    - 会社別設定が0の時に個人キャパにフォールバックすべきかを選択できるようにする。
2.  **UX・ユーザー視点担当**: 
    - 「会社Aのフィルタ」をかけている時、ユーザーは「会社Aの持ち時間」で見たいのか、「自分の全持ち時間」の中での会社Aの占有率で見たいのか。
3.  **ビジネス・マネタイズ担当**: 
    - 複数組織を掛け持つユーザーにとって、一日の総時間は有限。組織ごとの「枠」を超えた時の警告の重要性。
4.  **リスク・法（運営）担当**: 
    - 算出根拠が不明確だと、納期遅延やオーバーワークを招く。表示されている時間の「定義」をUIで明示すべき。

---

## 3. ラウンド構成

### 【ラウンド1】初期提案（Divergence）

**技術担当**: 
「現在の `QuantityEngine` は `tenantId` を渡すとその会社の枠だけを見ますが、`undefined` だと全合計を見ます。これを詳細画面（Modal）のコンテキストと連動させる必要があります。基本は『アイテムが属する会社のキャパ』をデフォルトにしつつ、全体の状況（全合計）も捨てがたい。」

**UX担当**: 
「ユーザーは『今、会社Aの仕事をしている』のであれば、会社Aに割り当てた時間内で終わるかを知りたいはず。しかし、もし会社Aの仕事が食い込んで『他の会社の時間が削られる』のであれば、それは一日の総量（Standard）で見せるべきです。カレンダーの背景は『全合計』、青枠（目安）は『そのタスクのコンテキスト』と分けるのが直感的です。」

**ビジネス担当**: 
「JBWOSの思想は『自分の現実』です。会社が期待している枠（Budget）と、自分が実際に使える時間（Reality）は別物です。まずは『自分の設定した枠』を最優先し、それが未設定の場合のみ会社平均やデフォルト値を使うべきです。」

**リスク担当**: 
「表示モードを明示すべきです。『会社Aのキャパで算出中』なのか『全活動時間で算出中』なのかを、警告メッセージの代わりにステータスバッジのように表示すれば、ユーザーの誤解を防げます。」

### 【ラウンド2】交差レビュー（Debate）

**UX担当 → 技術担当**: 
「『アイテムの tenantId』を常に使うと、プライベートなタスク（Private）の時に会社時間が計算に入らなくなります。現実にはプライベートの日でも会社時間は存在する（仕事はある）はずなので、計算の起点（Anchor）において『今日の全貌』が見えないのは不便です。」

**技術担当 → UX担当**: 
「それなら、`QuantityEngine` に `mode: 'context_only' | 'total_all'` といったフラグを導入し、カレンダーのフィルタ状態と同期させるのがスマートです。フィルタが『All』なら全合計、特定の会社ならその会社のキャパで青枠を出す、と。」

**ビジネス担当 → リスク担当**: 
「バッジ表示には賛成です。ただ、あまりに複雑な切替はユーザーを混乱させます。シンプルに『現在のカレンダーフィルタで見えているキャパシティ』を目安期間の計算にも使う、というルールで統一できないでしょうか？」

### 【ラウンド3】統合・収束（Convergence）

**ファシリテーターによる整理**:
1.  **算出原則**: 「現在カレンダーに適用されているフィルタ」の `tenantId` を計算の主軸とする。
    - **All フィルタ時**: 全組織の合計 + フリー時間（全キャパシティ）で算出。
    - **特定組織フィルタ時**: その組織に割り当てた時間のみで算出。
    - **未設定（NULL）時**: その組織に時間が割り当てられていない場合、警告（ガイド）を出す（前回結論通り）。
2.  **例外措置**: アイテム詳細画面では「そのアイテムが属する組織」のコンテキストが優先されるが、ユーザーが意図的にフィルタを切り替えている場合は、フィルタ状態を優先する。
3.  **デバッグの具体化**: 「今どのTenantIdで、一日のキャパを何分と判定したか」をログの1行目に必ず出す。

---

## 4. 最終結論

### 結論: フィルタ連動型コンテキスト算出
- **基本ルール**: `QuantityEngine` に渡す `tenantId` は、カレンダーの「現在のフィルタ状態」と連動させる。
- **詳細画面での挙動**: 
    - カレンダーフィルタが `All` の場合、アイテムの組織に関わらず「全キャパシティ」で目安期間を出す（一日の総稼働でいつ終わるか）。
    - フィルタが `特定組織` の場合、その組織の「割当枠内」でいつ終わるかを表示する。
- **実装改善**: 
    - `DecisionDetailModal` から `QuantityEngine` を呼ぶ際、`filterMode` に基づく `tenantId` パラメータの動的選択を実装する。
    - ログ出力を強化し、`Current Context: [TenantID/All], Daily Cap: [X] min` を明示する。

### 採用理由
- 「全体の中でいつ終わるか（スケジュール）」と「その仕事の枠内でいつ終わるか（コスト/予算）」の使い分けを、既存のフィルタUIで自然に行えるため。
- ユーザーの混乱を「表示されている背景（カレンダー）の計算根拠」と「青枠の計算根拠」を一致させることで防ぐ。
