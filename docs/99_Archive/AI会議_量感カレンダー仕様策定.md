# AI専門家会議：量感カレンダー（B）の仕様策定

## 1. 会議の定義

- **テーマ**: 量感カレンダー（B）の具体的仕様とシステム思想（集中と登録）への適合
- **ゴール**: 
    - フィルタリング基準の確定（ダッシュボードとの同期）
    - 個人モード/仕事モードにおける負荷計算ロジックの定義

- **前提条件**: 
    - AI_DEVELOPMENT_CONSTITUTION v4.2（日本語原則、可逆性、Deep Context）
    - JBWOS思想（「判断を求めない」「今の現実に寄り添う」）
    - TDD/MVVMアーキテクチャ

## 2. キャスティング

- **司会 (Facilitator)**: Antigravity

- **UX・ユーザー視点担当 (Expert 2)**: 心理学・デザイン研究者。「脳内のInbox」を視覚化し、心理的負荷を下げ、集中を促す体験を設計。

- **品質・検証担当 (Expert 4)**: QAエンジニア。ダッシュボードフィルタとの一貫性、エッジケース（未割当、期限切れ）の検証プロトコルを策定。

## 3. 議事録（順次更新）

### 【ラウンド1】初期提案（Divergence）

#### 技術・実装担当 (Expert 1)
「まず、ダッシュボードと同じフィルタ基準を適用するため、`ItemController` の `aggregated` / `dashboard` スコープで用いている SQL 抽出ロジックを `CalendarController` にも移植、あるいは Service クラスとして共通化すべきです。現在発生している `assigned_to` = null 問題は、まさにこの共通化の欠如から来ています。また、プロジェクト集中モード（ProjectFocused）時には、カレンダーもそのプロジェクト及びサブプロジェクトのアイテムのみを『加点』対象とする動的なクエリ切り替えが必要です。」

#### UX・ユーザー視点担当 (Expert 2)
「量感カレンダーの本質は『今日どれだけ頑張りすぎていないか』を直感することです。そのため、色の濃淡計算には『見積時間』だけでなく、ユーザーが脳内で感じている『圧迫感』も反映したい。具体的には、シングルクリックでの曲線アニメーションは、単なる指示線ではなく、その日の負荷がどのタスクから来ているかという『エネルギーの流れ』を表現すべきです。また、ダブルクリックでの一覧表示は、ダッシュボードに戻るのではなく、カレンダー上のオーバーレイとして表示することで、文脈（Context）を壊さない体験を目指すべきです。」

#### ビジネス・運用担当 (Expert 3)
「キャパシティ（MAX値）設定に関して、会社ごとの労働時間と個人の活動時間を完全に分離、あるいは統合して表示できる『トグル』が必須です。例えば『社内モード』時はその会社のタスクと定休日のみを反映し、背景色をグレーアウトする。一方『個人モード』ではすべての所属会社＋個人のタスクを統合した『総量』が見える必要があります。この権限と情報の切り替えは、新しい `membership` モデルに基づき、現在のコンテキストに合わせて正確に計算されるべきです。」

#### 品質・検証担当 (Expert 4)
「一貫性が最重要です。ヘッダーで『完了を表示』がオフならカレンダーからもそのアイテムの負荷（時間）は除外されるべきか？ ユーザーの意図としては『実態』を見たいはずなので、完了済みは負荷から抜くのが正解でしょう。また、期限が『マイ期限』なのか『絶対納期』なのかによって色がどう変わるか、あるいは『定休日』を跨いでタスクを配置した際の警告など、エッジケースのテストシナリオを初期段階で定義することを提案します。」

### 【ラウンド2】交差レビュー（Debate）

#### 技術・実装担当 (Expert 1) による Expert 2 への指摘
「Expert 2 が提案した『曲線アニメーション』は視覚的に素晴らしいが、1日に20以上のタスクがある場合、画面が曲線で埋め尽くされて視認性が著しく低下します（スパゲッティ状態）。また、0.5秒のアニメーションを繰り返すとユーザーの操作テンポを阻害するリスクがあります。さらに、カレンダー上のアイテムカードから『マスを光らせる』効果は、DOMの再レンダリング負荷が高まるため、Canvas描画あるいはCSSレイヤーの工夫が必要です。」

#### UX・ユーザー視点担当 (Expert 2) による Expert 3 への指摘
「Expert 3 の『トグルによる切り替え』は、逆にユーザーに『今どのモードか』を常に意識させるという認知負荷を強いています。システム思想である『判断を求めない』に反するのではないでしょうか。トグルで切り替えるのではなく、ヘッダーで選択されているプロジェクトや会社コンテキストに応じて、量感カレンダーが『空気を読んで（Context-Aware）』自動で重み付けを変えるべきです。」

#### ビジネス・運用担当 (Expert 3) による Expert 1 への指摘
「Expert 1 は SQL ロジックの共通化を提案しているが、個人アイテム（tenant_id IS NULL）と組織アイテムを単純に `OR` で結合すると、組織側の管理者が個人の未割当タスクの内訳まで見えてしまうプライバシー侵害のリスクがあります。カレンダー表示においても、『量は見えるが、具体名は伏せる』といったマスキング処理が、個人・仕事モードの境界線として必要です。」

#### 品質・検証担当 (Expert 4) による 全員 への指摘
「皆、『現在の負荷』に集中しすぎです。過去の『実績』とカレンダーの整合性はどうなるのでしょうか？ 例えば、完了したタスクが元々あったマスの色が薄くなるのか。あるいは『今日』を過ぎた過去のマスの色は消えるのか、実績として残るのか。ここが定義されないと、数日後の振り返り（振り返りもシステムの重要機能）でカレンダーが使い物になりません。」

### 【ラウンド3】統合・収束（Convergence）

#### 司会 (Facilitator) による論点整理
「活発な議論ありがとうございます。以下の3点が収束すべき重要ポイントです。
1. **プライバシーと透明性の両立**: 組織モードでは個人アイテムの『量』のみを合算し、詳細は秘匿する。
2. **パフォーマンスとUXのバランス**: 曲線は上位5件など代表的なものに絞るか、あるいはホバー時のみ表示する。
3. **Context-Awareな自動フィルタ**: トグルではなく、ヘッダーの『プロジェクト集中』状態をカレンダーの計算範囲（分母/分子）に自動反映する。

## 4. 最終結論

### 決定事項：量感カレンダー（B）の最終仕様

#### 1. フィルタリングと計算ロジック
- **ダッシュボード同期**: ヘッダーの選択状態（会社、プロジェクト、フィルタ条件）を完全に反映する。
- **負荷の定義**: `Σ(見積時間) / 日次キャパシティ`。
- **キャパシティ（MAX値）**: 
    - 「個人モード」＝ `個人キャパ + 全所属会社の基本キャパ合計`
    - 「社内モード」＝ `その会社のキャパのみ`
- **完了アイテムの扱い**: 基本的に「未完了」のみを負荷として計上するが、切り替えスイッチにより「実績（完了済み）」を重ねて表示できる予備設計とする。

#### 2. UXとビジュアルインタラクション
- **曲線アニメーション**: 指定した日をシングルクリックした際、その日の負荷を構成するトップ5件（時間順）に向けて曲線を表示する。それ以外の細かなタスクはまとめ表示、またはホバー時に個別に表示する（スパゲッティ化防止）。
- **光る強調**: アイテム一覧からアイテム名をクリックすると、そのアイテムが影響を与えている量感カレンダーの全てのマス（目安期間全体）が淡く発光する。

#### 3. プライバシーとマルチテナント対応
- **マスキング**: 「社内モード」において、他社のタスクや個人のタスクが同日に重なっている場合、その「量」は合計負荷に加算されるが、アイテム名や詳細は「他組織の予定」としてマスクされる。これによりプライバシーと物理的な空き時間の把握を両立する。

#### 4. 再利用性と構造
- **共通モジュール化**: `CalendarController` と `ItemController` のクエリロジックを Service クラスに抽出し、一貫性を保証。フロントエンドでは `reuseCalendarLayout` を作成し、納期選択等の別画面でも同一デザインを呼び出せるようにする。

### 採用理由
- 「判断を求めない」UXを追求し、モードに応じて自動で情報密度が最適化される仕様。
- 個人と組織の両立というJBWOSのコアバリューを、情報の秘匿と量の可視化でバランスさせた。

### 想定リスク
- リアルタイムでの複雑な集計クエリによるパフォーマンス低下（対処：ビューまたは集計テーブルの検討）。
- アニメーションの描画負荷（対処：Canvas/WebGLの検討）。
