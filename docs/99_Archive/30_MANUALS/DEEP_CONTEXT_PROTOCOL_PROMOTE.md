# AIがプロジェクトの記憶を失わないための全技術：Deep Context Protocol (DCP)

## 0. プロンプト (The Prompt)
この記事で紹介する手法を、あなたのAIモデル（ChatGPT, Claude, Gemini等）に一発で理解させ、運用を開始させるためのプロンプトです。プロジェクト開始時や、コンテキストがリセットされた時に貼り付けてください。

```markdown
# Role & Objective
あなたはプロジェクトの熟練アーカイブ担当兼エンジニアです。
これより**「Deep Context Protocol (DCP)」**に基づいて、情報の自律的な整理とコンテキスト維持を行ってください。

# 1. First Action: The Map
まず、プロジェクトルートにある **`docs/00_CONTEXT.md`** を熟読してください。
これが本プロジェクトの唯一の「正解」であり「地図」です。
ここには「現在地」「技術スタック」「解決すべき課題」が網羅されています。

# 2. Directory Structure & Rules
以下のディレクトリ構造と運用ルールを厳守してください。

- **`docs/00_CONTEXT.md`**:
  - 全体の地図。フェーズが進むたびにここを更新し、常に最新の状態に保つこと。
- **`docs/01_RULES/`**:
  - `DOCS_OPS.md` (文書管理ルール) や `AI_DEVELOPMENT_CONSTITUTION.md` (開発憲法) を格納。迷ったらここに戻る。
- **`docs/20_DECISIONS/`**:
  - 議論や設計決定のログ。ファイル名は `YYYYMMDD_テーマ.md` とする。
  - **重要**: 決定事項だけでなく、**「ユーザーの生の言葉」「捨てた案」「なぜそう決めたか」**という文脈（Deep Context）を要約せずに残すこと。
- **`docs/30_MANUALS/`**:
  - ユーザー向けマニュアルや、今回のような手法の解説書。
- **`docs/晴樹の口頭入力/`** (または `docs/OralInputs/`):
  - ユーザーのチャット発言や音声入力をそのまま保存する場所。ここにあるファイルは「加工されていない真実」として扱う。

# 3. Deep Context Philosophy (最も重要な指示)
あなたはドキュメントを作成する際、**「過度な要約」をしてはいけません。**
- ユーザーの比喩（例：「隠れ残業のような」）を「勤怠管理上の課題」と抽象化しないこと。
- ニュアンス、熱量、迷いをそのまま記録すること。
- それこそが、未来のあなたが読み返した時に最も価値のある情報になります。

以上を理解したら、`00_CONTEXT.md` の読み込みから開始してください。
```

## 1. はじめに：AIとの開発で起きる「文脈の断絶」

AIとのコーディングは快適ですが、唯一にして最大の欠点があります。
それは**「記憶喪失」**です。

長時間の議論の末にチャットのコンテキストがいっぱいになり、新しいスレッドを立てた瞬間、AIは別人になります。
「前回の続きですが」と言っても、AIは「前回の要約」しか知りません。そこには、あなたが共有した「熱量」や「微妙なニュアンス」はもう残っていないのです。

これを解決するために私たちが開発・実践しているのが、**Deep Context Protocol (DCP)** というドキュメント管理手法です。

## 2. Deep Context Protocol の全貌

DCPは単なるフォルダ分けではありません。「AIに文脈を再体験させる」ための保存技術です。

### 構成概要 (File Tree)

```text
ProjectRoot/
├─ docs/
│  ├─ 00_CONTEXT.md          <-- 【地図】全ての始まり、最新の正解
│  ├─ 01_RULES/              <-- 【憲法】開発ルール、運用ルール
│  ├─ 10_ARCHITECTURE/       <-- 【設計】システム構成図、DBスキーマ
│  ├─ 20_DECISIONS/          <-- 【歴史】思考プロセス、決定の経緯
│  ├─ 30_MANUALS/            <-- 【操作】手順書、解説書
│  ├─ 晴樹の口頭入力/         <-- 【原液】ユーザーの未加工の指示・想い
│  └─ ARCHIVE/               <-- 【過去】古くなった情報
└─ task.md                   <-- 【現在】直近のタスクリスト
```

### 各要素の役割

#### 1. The Map: `00_CONTEXT.md`
**「AIが新しくなったら、まずこれだけ読めばいい」**というファイルです。
膨大な仕様書を読ませるのではなく、このファイルにプロジェクトの目的、用語集、そして他ファイルへのリンクを集約します。AIには「地図のメンテナンス」を最優先義務とさせます。

#### 2. The History: `20_DECISIONS/` (ADR)
Architecture Decision Records の応用版です。
ファイル名は `YYYYMMDD_テーマ.md`（例：`20260122_セキュリティ権限設計.md`）。
ここでのルールは **"Deep Context"**。つまり、「結果」だけでなく「過程」を書くこと。
- ユーザーがポロッと言った「なんか使いにくい」という感覚
- AIが提案してボツになったA案、B案
これらを残すことで、将来のAIが「なぜこの仕様になったのか」を推論できるようになります。

#### 3. The Raw Voice: `docs/晴樹の口頭入力/`
ユーザーがチャットで入力した長文の要望や、音声入力の書き起こしを、**AIに解釈させずにそのまま**テキストファイルとして放り込む場所です。
AIの解釈が入ると、その時点でニュアンスが劣化します。原液を保存しておくことで、方針に迷ったときに立ち返る場所を作ります。

#### 4. The Laws: `01_RULES/` (DOCS_OPS)
「ファイル名は日付を含めない（検索性のため）」「更新したら必ず日付スタンプを押す」といった、ドキュメント自体の運用ルール（Ops）を定義します。AIはこのルールを参照して、自律的にファイルの整理整頓を行います。

## 3. なぜこれが画期的(Convenient)なのか

1.  **「引き継ぎ資料」を作る必要がない**
    常に `00_CONTEXT.md` が最新の状態にあるため、いつ誰が（人間でもAIでも）プロジェクトに参加しても、そこを見るだけで現状把握が完了します。

2.  **「なんでこうなってるの？」がなくなる**
    `20_DECISIONS` に「捨てた案」まで書いてあるため、コード上の奇妙な実装の理由（当時の制約やユーザーの強いこだわりなど）が明確になります。

3.  **AIの「知ったかぶり」を防げる**
    「口頭入力」フォルダにある生の言葉を参照させることで、AIが勝手に要件を丸めてしまうのを防ぎ、ユーザーの本来の意図に沿った提案を引き出せます。

## 4. 結論

AI時代のドキュメントは、「人間が読むためのメモ」から**「未来のAIに文脈をインストールするための記憶媒体」**へと進化する必要があります。

要約させない。綺麗にまとめすぎない。
**Deep Context Protocol** で、あなたのプロジェクトの「魂」を保存しましょう。
