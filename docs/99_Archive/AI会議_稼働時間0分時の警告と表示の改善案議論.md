# AI会議_稼働時間0分時の警告と表示の改善案議論

## 1. 会議の定義
- **テーマ**: 稼働時間0分設定時の動作仕様とUI（警告）の改善、およびハイライト不具合のデバッグ
- **ゴール**: 
    - 「設定漏れ」と「意図的な0分設定」の判別の要否と方法の決定
    - UIでの警告表示位置の修正案
    - ハイライトが1マスしか出ない不具合のデバッグポイント特定
- **前提条件**:
    - AI開発憲法に基づき「ユーザーの現実を尊重（Judgment-Free）」しつつ、システムの不備と誤解させない。
    - 画面レイアウトの整合性を保つ。

---

## 2. キャスティング
1.  **ファシリテーター**: 全体の進行と論点整理。
2.  **技術・実装担当**: `QuantityEngine` のロジックと、DB上のデータ構造（デフォルト値と未設定の区別）の観点。
3.  **UX・ユーザー視点担当**: 警告の表示タイミングと、カレンダーの見やすさ・重なり問題。
4.  **ビジネス・マネタイズ担当**: 業務支援ツールとしての信頼性、正確なスケジュール算出の重要性。
5.  **リスク・法担当**: カレンダー計算ミスによる納期遅延リスクの回避。

---

## 3. ラウンド構成

### 【ラウンド1】初期提案（Divergence）

**技術担当**: 
「現在、`users` テーブルの `non_working_hours` が NULL または空、かつ `standardWeeklyPattern` の合計が0の場合を一律で『設定なし』とみなして警告を出しているかもしれません。DB上で『明示的に0分と入力された』ことと『初期状態でNULL』であることを区別できれば、解決の糸口になります。また、ハイライトが1マスしか出ない件は、`remainingMinutes` が最初の1日で0以下になる計算ミス（多重加算の残りなど）を疑っています。」

**UX担当**: 
「画像を見る限り、警告のアラートが曜日ラベルに完全に重なっており、日付が見辛くなっています。表示位置をカレンダーの枠外、あるいはヘッダー付近へ移動すべきです。また『警告』と呼ぶとユーザーが否定されたと感じるため、『算出のための設定が不足しています』といったニュートラルなガイドに近い表現に変えたいですね。」

**ビジネス担当**: 
「土日を0にするのは普通ですが、全曜日が0だと『ツールとして機能しない』状態です。1週間すべての合計が0の場合のみ警告（ガイド）を出すのが現実的ではないでしょうか。1日でも稼働があれば、算出自体は（非常に長くかかっても）可能ですから。」

**リスク担当**: 
「目安期間が表示されないとユーザーは納期を誤認するリスクがあります。計算不能な場合は『目安が出せません』とハッキリ言うべきですが、意図的な0（祝日など）で計算が伸びる分には、その実態をそのまま青枠で示すのが正しい動作です。」

### 【ラウンド2】交差レビュー（Debate）

**技術担当 → UX担当**: 
「アラートをカレンダーの上に `absolute` で出すのではなく、カレンダーパネルの最上部に `static` 配置（押し下げる形）にすれば重なりません。ただ、縦幅が狭くなるのでコンパクトな設計が必要です。」

**UX担当 → ビジネス担当**: 
「『全曜日が0の場合のみ』だと、例えば『月〜金は0、土日だけ稼働』という特殊な設定をしたい場合も拾ってしまいます。システムが『デフォルト設定（480分など）』を使用しているのか『ユーザーの個別設定』を使用しているのかをステータスで示せれば、ユーザーは納得感が増すはずです。」

**ビジネス担当 → 技術担当**: 
「ハイライトが1マスしか出ない件、もしかしてキャパシティ計算時に『組織の合計』ではなく『個人のデフォルト値』が優先的に返ってきてしまい、かつそれが巨大な値（バグによる多重加算など）になっていないか再確認が必要です。」

### 【ラウンド3】統合・収束（Convergence）

**ファシリテーターによる整理**:
1.  **警告条件の緩和**: 「全曜日が0である」かつ「個人キャパシティ（dailyCapacity）もデフォルトのまま」の場合に限り、「ガイド」として表示する。
2.  **UI表示の改善**: `absolute` 配置をやめ、カレンダーの上部にスペースを確保して表示する、あるいはツールチップ形式にする。
3.  **デバッグ方針**: ハイライト1マスの謎を解くため、`QuantityEngine` 内の `calculateAllocationDays` と、 modal から渡される `capacityConfig` の中身を詳細に出力する。

## 4. 最終結論

### 結論: 条件付き修正
- **警告ロジック修正**: 「週の合計稼働時間が0かつデフォルト設定のみ」の場合に表示。意図的な休日設定（特定の曜日が0）では表示しない。
- **UI修正**: 警告メッセージの表示位置を調整（重なりを解消）。
- **デバッグ強化**: ユーザー提案の通り、各日の判定キャパシティと配分プロセスを詳細に追跡するログを仕込む。

### 採用理由
- ユーザーの意図的な休日設定（0分）を否定しないため。
- UIの視認性向上。
- ハイライト問題の根本原因の特定。

### 次アクション
1.  `DecisionDetailModal.tsx` の警告表示条件とCSSを修正（まずは計画）。
2.  `QuantityEngine.ts` と `DecisionDetailModal.tsx` に詳細なデバッグログ（キャパシティ設定、計算プロセス）を追加。
3.  ログの結果をユーザーと共有し、1マスしか表示されない原因を特定する。
