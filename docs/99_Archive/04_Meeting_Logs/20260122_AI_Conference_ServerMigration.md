# AI会議：サーバーサイド完全移行（クラウド化）とデータ権限

**日時**: 2026-01-22
**テーマ**: ローカルデータのサーバー移行（クラウド化）における設計思想の検証
**入力ソース**: [Migration_Plan_ServerSide.md](../05_Roadmap_and_Backlog/Migration_Plan_ServerSide.md)

**参加者**:
- **Facilitator**: 司会進行。
- **System Architect**: システム設計責任者。
- **Privacy Guardian**: ユーザーのプライバシーとデータ主権を守る担当。
- **UX Lead**: ユーザー体験（「PCでもスマホでも」）の実現担当。

---

## 1. 議論の背景と目的
ユーザーより「パソコンでもスマホでも同じものを操作したい」という要望があり、現在ブラウザ(Dexie)にある製造データ（Projects/Doors/Deliverables）をサーバー(SQLite)へ完全移行する計画を立てました。
この設計がプロジェクトの憲法（Constitution）や思想に合致しているかを確認します。

## 2. 議題

### 議題1: アーキテクチャ選定 (Online-First vs Offline-Sync)
**System Architect**:
現状のDexie(ローカルハヤイ)を捨て、APIベース(Online-First)に移行する提案です。
- **メリット**: スマホ・PC間の完全同期が即座に実現。実装コストが低い。
- **デメリット**: ネットがないと使えない。レスポンスがネットワーク依存になる。

**論点**:
「速さより迷わなさを優先する」という憲法に基づき、複雑な同期ロジック（Offline-Sync）によるコンフリクトの混乱を避けるため、シンプルなOnline-Firstを採用すべきか？

### 議題2: データ ID の移行戦略
**System Architect**:
Dexieは `Auto-Increment Number (1, 2, 3...)` を使用していますが、サーバーサイド分散環境では `UUID` が望ましいです。
移行時に既存データをどう扱うか？
- **A案**: 全てUUIDに変換し、リレーションを張り直す（安全だが大変）。
- **B案**: 既存データは文字列としての "1" を使い続け、新規はUUIDにする（混在）。

### 議題3: 画像データの扱い (Blob vs FileServer)
**System Architect**:
現在DexieにBlobとして保存されている写真を、サーバーのファイルシステム（`backend/uploads/`）に書き出す方式に変更します。
プライバシー観点で問題ないか？

---

## 3. 議論ログ

### 3.1 Online-First vs Offline-Sync

**UX Lead**:
「パソコンでもスマホでも」という要望が最優先です。ユーザーは現場（スマホ）で寸法を見たい、事務所（PC）で見積もりを作りたい、というシームレスな体験を求めています。
現在のDexie（ローカルDB）は爆速ですが、「あのPCじゃないとデータがない」というストレスはユーザーエージェンシー（自由）を阻害しています。よって、サーバー移行は必須です。

**Privacy Guardian**:
サーバーにデータを置くこと自体は、ユーザーが管理するサーバー（ConoHa上のインスタンス）であれば「データ主権」は守られます。外部のSaaSに預けるわけではないので、Constitutionの違反にはなりません。
ただし、**「ネットがないと何も見れない」**という状況は現場で困りませんか？

**System Architect**:
良い指摘です。しかし、Constitution 1.0 にある「速さより迷わなさを優先する」を適用しましょう。
Offline-Sync（PWAでローカルに書き込み、ネット復帰時に同期）は、技術的に非常に複雑で、「同期コンフリクト（PCとスマホで同時に編集した場合など）」によるデータ先祖返りや消失のリスクがあります。
これは「迷い」と「不利益」を生みます。
まずはシンプルな **Online-First** で実装し、Service Workerによる「Read-Onlyキャッシュ（直前に見たデータはオフラインでも見える）」程度に留めるのが、最も堅牢で安全です。今の日本の通信環境なら、完全オフラインの現場は稀です。

**UX Lead**:
同意します。変に同期ミスでデータが消えるより、確実にサーバーにあるデータが正（Single Source of Truth）である方が心の平穏に繋がります。

### 3.2 ID移行戦略 (Number vs UUID)

**System Architect**:
既存の数値ID（1, 2, 3...）はサーバーのDB（SQLite/MySQL）のAuto Incrementと競合しやすいです。特に将来的な分散環境を見据えるとUUIDが必須です。
Migration Toolで、既存データをアップロードする際に **「新しいUUIDを発行し、旧IDとの対応表を作って全リレーション（リンク）を書き換える」** べきです。

**Privacy Guardian**:
それは「不可逆」な変更になりませんか？
失敗したら元に戻せますか？

**System Architect**:
ローカルのDexieデータは消さずに残しておけばバックアップになります。
移行が成功し、ブラウザで確認が取れたら、Dexieを空にする（あるいはアーカイブ化する）手順にすれば安全です。

**結論**: **「全データUUID変換・リレーション書き換え」** を採用。

### 3.3 画像データの扱い

**System Architect**:
SQLiteに巨大な画像をBlobで入れるとパフォーマンスが死にます。
ファイルシステム (`/uploads`) に保存し、DBにはパスだけ書くのが定石です。

**Privacy Guardian**:
画像ファイルに直接アクセスされるリスク（Directory Traversalなど）に注意してください。
`.htaccess` で実行権限を切る、あるいはPHP経由でしか画像を表示できないようにする（認証チェック）のが理想ですが、まずはUUIDファイル名にして推測不可能にするレベルでスタートしましょう。

## 4. 最終結論

### 合意事項
1. **アーキテクチャ**: **Online-First API Architecture** を採用。
   - ブラウザのDB (Dexie) は廃止（キャッシュ用途のみへ）。
   - すべてのデータ操作はリアルタイムでサーバーAPIを叩く。
   - ネット接続必須とする（将来的にReadキャッシュ検討）。

2. **データ移行**:
   - クライアントサイドで移行スクリプトを実行。
   - 数値IDはすべて **UUID** に変換・再リンクする。
   - 画像Blobはサーバーへのファイルアップロードに変換する。

3. **セキュリティ**:
   - ファイルアップロード機能を追加するが、拡張子制限・サイズ制限を設ける。

### アクションアイテム
1. **Migration Planの確定**: 次のステップでこの設計通りに実装を進める。
2. **バックエンド実装**: テーブル作成、API作成。
3. **マイグレーションツール作成**: React上に「データ移行ボタン」を作る。

この設計は「ユーザーがどの端末からでも同じデータにアクセスできる（自由）」と「データの整合性を保つ（安全）」を両立しており、プロジェクト憲法に合致します。
