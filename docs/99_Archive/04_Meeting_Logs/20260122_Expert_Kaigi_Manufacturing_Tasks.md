## 1. 概念定義：製造業レイヤーの導入（Deliverables）

ユーザー（晴樹氏）の指摘により、データモデルを以下のように再定義する。
従来の「プロジェクト → 建具」という直接的な関係ではなく、間に「製造物（Products/Deliverables）」という抽象化レイヤーを挟む。

### 1-1. 3層アーキテクチャ
1.  **Project (案件)**: 顧客からの依頼単位。「箱」。
2.  **Deliverable (製造物・製品)**: 製造業として納品すべきモノ。
    *   共通属性: 製造原価、売値、納期、説明、製作時間、現場時間。
    *   JBWOS上の「タスク」の実体はこれになる。
3.  **Domain Item (建具・家具)**: その製造物の具体的な仕様。
    *   属性: 寸法、材料、加工方法、姿図データ。

### 1-2. データフロー
*   プロジェクト作成 -> `Project` レコード作成
*   建具追加 -> `Deliverable` レコード作成（「建具A」という名前で） -> 同時に `Door` レコード作成（詳細仕様）
*   GDB表示 -> `Deliverable` テーブルを参照してタスクリスト化する。

## 2. 要求定義

ユーザー（晴樹氏）からの要望は以下の2点に集約される。

### 2-1. プロジェクト＝親タスク (Project as Parent Task)
> 「製造業プロジェクトが作成されたら、そのプロジェクト名をタスクとして登録してほしい。そしてそのプロジェクトに関するタスク（制作とか見積もりとか取り付け）が作られるときには、必ずそのプロジェクト名のタスクを親タスクとして関連付けする。」

*   **現状**: プロジェクトは「箱」として存在し、建具データは `doors` テーブルに、タスクは `items` テーブル（GDB）に独立して存在していた（一部マージロジックはあるが）。
*   **新仕様**: プロジェクト作成と同時に、GDB上に「親タスク（Project Task）」を実体として生成する。後続のタスクは全てこの子タスクとなる。

### 1-2. 建具タスクの自動生成 (Automatic Door Task)
> 「タスク生成したものは、建具名を変更したら追従するとか、建具エディタで製造目安時間を入力したらそれが目安時間に適用される。というのもやりたい。」
> 「現在はプロジェクト内に建具Aを作っても『建具Aを製作する』などのタスクは自動追加されていません。追加されてほしい。」

*   **新仕様**: プロジェクト内で建具を追加した瞬間、GDB上に「製作タスク（Production Task）」を自動生成する。
*   **同期要件**: 建具エディタ（製造プラグイン側）での変更（名前、サイズ、見積時間）が、即座にGDB上のタスク情報に反映されること。

## 2. 設計：データモデルとリレーション

### 2-1. Project Task (Parent)

プロジェクト作成時に、`items` テーブル（またはGDB API経由）に以下のアイテムを生成する。

```typescript
interface ProjectTask extends Item {
    id: string; // "project-{projectId}" (Virtual) ではなく UUID (Real)?
    // -> Discussion: "プロジェクトID"と"タスクID"を分けるべきか？
    // -> 同一視するなら "project-{id}" をIDとして使うのが最もシンクロしやすい。
    
    title: string; // プロジェクト名
    status: 'inbox' | 'active' | 'hold' ...;
    type: 'project'; // 新設
    isContainer: true;
}
```

**決定事項**:
*   プロジェクト作成時、`items` テーブルへの登録は行わず、**`JBWOSRepository` が `local.projects` を参照して「仮想的な親タスク」として振る舞う** 現行方式を拡張・強化する。
*   理由：二重管理（ProjectテーブルとItemsテーブルの同期）はバグの温床になるため。Projectテーブルを正（Master）とし、GDBはそれを投影（Projection）する。

### 2-2. Door Task (Child)

建具作成時に、自動的にタスクとして認識される必要がある。

*   **Trigger**: `useJoineryViewModel.handleCreateDoor`
*   **Action**: 建具 `db.doors` に保存されると同時に、GDB上でもタスクとして見える必要がある。
*   **現状**: `JBWOSRepository.getGdbShelf` で `db.doors` をフェッチして混ぜている。
*   **課題**: 「名前の同期」「時間の同期」は、リロード時だけでなくリアルタイム性が求められるか？
    *   はい。エディタを開いて修正中は、タスク側も更新されてほしい（または保存時）。

### 2-3. Syncer (Observer Pattern)

エディタ（建具プラグイン）とGDB（JBWOS Core）の間でデータを同期する仕組み。

1.  **Entity**: `Door` (Local DB) <---> `Item` (GDB View)
2.  **Fields to Sync**:
    *   `Door.name` <---> `Item.title`
    *   `Door.estimatedWorkMinutes` <---> `Item.manHours (estimatedMinutes)`
    *   `Door.dueDate` <---> `Item.dueDate`

**同期の実装方針**:
*   データの実体は `doors` テーブルに持つ（Single Source of Truth）。
*   GDB側（`JBWOSRepository`）は、`items` テーブルだけでなく `doors` テーブルも「タスクの一種」として透過的に読み書きする。
*   ユーザーがGDB上でタスク名を変更した場合 -> `doors` テーブルの `name` を更新する。
*   ユーザーがエディタで建具名を変更した場合 -> `doors` テーブルの `name` が更新され、GDBはそれを読み込む。

## 3. 実装プラン

### 3-1. プロジェクトの親タスク化
*   [x] 既に `JBWOSRepository.getGdbShelf` で `project-{id}` としてプロジェクトを表示済。
*   [ ] **ペアレント機能の追加**: 新たに建具（タスク）を作成する際、所属プロジェクトのIDを `parentId` として持たせる必要があるかもしれないが、現状は `projectId` で紐づいている。
*   **確認**: タスクの親子関係（階層表示）をUIでどう表現するか？
    *   GDBではフラットに並ぶか、プロジェクトカードの中に含まれるか？
    *   要望：「必ずそのプロジェクト名のタスクを親タスクとして関連付けする」
    *   **UIイメージ**: Inboxやリスト上で、建具タスクは独立して存在しつつ、プロパティとして親プロジェクトを持つ。（フィルタリング可能）

### 3-2. 建具タスクの自動生成と同期
*   [x] 建具は既に `items` 相当としてGDBに表示されている（前回改修）。
*   [ ] **不足している同期項目**:
    *   `estimatedWorkMinutes` (製作目安時間)
    *   `status` (ステータス同期は確認済だが、ワークフロー同期が必要)
*   **アクション**:
    *   `Door` 型に `estimatedWorkMinutes` は追加済（`db.ts`）。
    *   エディタ画面（`DoorEditor` / `JoineryScheduleScreen`）でこの値を入力できるようにUIを追加。
    *   入力された値が `db.doors` に保存されることを確認。
    *   `JBWOSRepository` のマッパー (`convertDoorToItem`) で、この時間を `Item.manHours` にマッピングする。

## 4. 承認依頼事項

この設計方針（Single Source of Truthの実践）で進めます。
物理的に `items` テーブルにコピーを作成するのではなく、`projects` / `doors` テーブルを直接参照・更新する「仮想タスク」方式を維持・強化します。これにより「同期ズレ」を根本から防ぎます。

### 開発ステップ
1.  **エディタ改修**: 建具エディタに「製作時間（分）」入力欄を追加。
2.  **リポジトリ改修**: `JBWOSRepository` で時間を読み出し、GDBのタスク詳細に表示。
3.  **UI確認**: ダッシュボードで建具タスクを見た時、エディタで入力した時間が反映されているか確認。
4.  **親タスク表示強化**: GDB上で、各建具タスクに「親プロジェクト名」を明示的に表示（バッジ等）。
