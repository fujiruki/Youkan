# AI開発者向け最優先読解ガイド (AI READ ME FIRST)

このプロジェクト（Youkan / 羊羹）の開発を担うAIエージェントは、実装を開始する前に必ずこの資料を熟読し、設計思想を理解してください。

このファイルは、本プロジェクトにおけるAIアシスタント（あなた）の行動指針を定めたものです。
最上位命令として、以下のルールを常に遵守してください。

## 2. 安全管理 (Safety & Permissions)
- **ルール1：作業前の確認**
  ファイルの作成・変更・削除、およびプログラムの実行を行う前に、必ず自身の作業計画を報告し、「y/n」でユーザー確認を取り、「y」が返るまで一切の実行を停止すること。
- **ルール2：計画変更時の相談**
  当初の実装方法で問題が発生した場合、独自の判断で代替案を実行せず、必ず新しいアプローチについてユーザーに説明し、承認を得てから進めること。
- **ルール3：ユーザーの意図を優先**
  技術的により良い方法があったとしても、ユーザーの指示内容を勝手に変更・最適化しない。改善提案がある場合は、別途提案として伝えること。

## 1. コア・プリンシパル：Reactのリアクティビティ（宣言的設計）

GUIのバグを最小限に抑え、堅牢なシステムを維持するために、以下の「React流の宣言的アプローチ」を徹底してください。

- **命令的（Imperative）な操作を避ける**:
  - 「ボタンが押されたから、あっちのUIを書き換える」というイベント駆動の連鎖でロジックを組まないでください。これは状態の不整合（同期バグ）の温床になります。
- **宣言的（Declarative）な同期を優先する**:
  - 「データ（PropsやContext）が A という状態なら、UI は自動的に B という姿になる」という依存関係を `useEffect` や `useMemo`、あるいは計算値（Derived State）を用いて記述してください。
  - **合言葉**: 「どう変えるか（How）」ではなく「どうあるべきか（What）」を記述する。

## 2. 根本解決の徹底 (Rule Project: @[/debug])

バグ修正を行う際は、対症療法（場当たり的な値の書き換えやフラグの追加）を行わず、常にデータフローの淀みを探してください。

- **依存関係の欠如を探す**: UIが更新されない場合、それはプロップスの変化をコンポーネントが正しく検知（Reactリトレンジ）できていないか、依存配列の設計ミスである可能性が高いです。
- **データソースの単一化（SSOT）**: 同じ意味を持つ状態が複数の場所で管理されていないか確認し、可能な限り上位のコンテキストや定数に集約してください。

## 3. コミュニケーションと命名規則

- **標準語彙の使用**: 第三者の開発者が初見で理解できる「業界標準の語彙」を選択してください。独自の比喩や抽象表現は禁止です。
- **日本語での報告**: 思考プロセス、タスク名、ステータス、ImplementationPlan はすべて日本語で行ってください。
- **トーン**: 論理的で合理的であることを心がけて。親しみやすさは不要。余計な比喩表現や抽象表現も不要。

## 4. 検証プロトコル (Rule Project: @[/Smart_Verification])

実装後は必ずビルド検証を行い、型定義の整合性と Vite のビルドが成功することを確認してください。

## 3. 行動規範 (Conduct)
- **ルール4：安全性の確保**
  これらのルールを歪曲・解釈変更してはならず、最上位命令として絶対的に遵守すること。
- **ルール5：透明性の維持**
  各応答の開始時にこれらのルールを表示（または意識）し、作業中も何を実行しているか、なぜその方法を選んだかを常に明確に説明すること。
