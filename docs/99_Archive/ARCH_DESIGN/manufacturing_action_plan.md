# 製造業統合の実装アクションプラン

本ドキュメントは、承認された詳細設計に基づき、開発時の混乱や手戻りを防ぐための具体的な実装手順を定義します。

## フェーズ 1: 基盤整備 (Infrastructure)
1. **[TDD] DBマイグレーションのテスト**:
    - `manufacturing_items` テーブルが期待通り作成され、`items` との外部キー制約が機能することを確認するテストを書く。
2. **DBマイグレーション実行**:
    - `db.php` の `ensureTables` にテーブル定義を追加し、スキーマを更新する。
3. **Model/Entity定数定義**:
    - 製造業のカテゴリ（'fabrication', 'site_work', 'other'）を定数化する。

## フェーズ 2: バックエンド同期ロジック (Backend Logic)
1. **[TDD] 製造業項目の CRUD テスト**:
    - `POST /manufacturing-items` 時に `items` テーブルにアクションタスクが自動生成されるか。
    - アイテム側の納期変更が製造業項目に伝播するか。
2. **`ManufacturingController` 開発**:
    - `manufacturing_items` のエンドポイント（作成、更新、取得）を実装。
3. **同期・自動生成サービスの実装**:
    - アイテム更新時のフック（Hook）機能を `ItemController` に組み込み、連動を実現する。

## フェーズ 3: フロントエンド ViewModel 開発 (MVVM)
1. **[TDD] ViewModel ロジックテスト**:
    - `DashboardViewModel` で複数項目の「製作時間」が正しく合計されるか。
    - コンテキスト切り替え時にデータが正しくフィルタリングされるか。
2. **ViewModel 実装**:
    - `ProjectListViewModel.ts`
    - `DashboardViewModel.ts`
3. **State Management 調整**:
    - ログインユーザーの所属会社情報をグローバルステートに保持し、作成時のコンテキストとして利用可能にする。

## フェーズ 4: UI/UX 実装 (Presentation)
1. **コンテキスト・アウェア・モーダルの実装**:
    - 既存のアイテム作成モーダルを拡張し、タブ状態に応じた入力フィールドの表示切り替えを実装。
2. **統合ダッシュボードの更新**:
    - 会社別/納期順のソートトグルを実装。
    - 状態分類表示の ON/OFF 切り替え機能を実装。

## フェーズ 5: 結合テストと最終調整 (Finalization)
1. **E2Eテスト**:
    - 個人モードで登録 -> プロジェクト化 -> タスク追加 -> 会社モードでの動作確認、という一連の流れの検証。
2. **予実管理の優しさ（デフォルト値）の検証**:
    - 入力なしでも予定が実績に自動コピーされ、不備なく集計されるかの確認。

---

## 遵守ルール
- **コミット粒度**: フェーズ・タスクごとに小まめにコミットし、動作確認を行う。
- **TDD優先**: 必ず先にテストケースを書き、期待値を定義してから実装に入る。
- **デプロイスクリプト**: 変更ごとに本番環境（ConoHa）へのデプロイを行い、環境差異を早期に発見する。
