# JBWOS vs 建具プラグイン連携仕様 (Discussion Draft)

## 1. 現状の提案 (今回の実装予定だったもの)
*   **トリガー**: 建具(Door)の新規作成時。
*   **アクション**: JBWOSのInboxに「[建具名] 製作」というタスクを自動生成。
*   **データフロー**: 一方通行 (建具 -> JBWOS)。
*   **課題**: 建具側で修正してもタスクには反映されない。タスクを完了しても建具ステータスは変わらない。

## 2. 検討すべき連携レベル

### Level 1: 単純なタスク生成 (Fire & Forget)
*   建具登録 = 「やること」の発生、としてInboxに投げるだけ。
*   あとはJBWOS側で自由に料理する。
*   **メリット**: 疎結合。プラグインを切り離しやすい。
*   **デメリット**: 情報が同期されない。

### Level 2: データの双方向同期
*   **名称同期**: 建具名を直したらタスク名も変わる。
*   **ステータス同期**: JBWOSでタスク完了 -> 建具(Deliverable)も「完了」になる。
*   **期限同期**: 建具の納期を変えたらタスクの日付も変わる。

### Level 3: プロセス分解 (Manufacturing Plugin領域)
*   建具1つに対してタスク1つではなく、工程ごとにタスク化する？
    *   「[建具A] 木取り」
    *   「[建具A] 組立」
    *   「[建具A] 塗装」
*   これらはJBWOSのサブタスクとして扱うか、別々の独立タスクか。

### Level 4: コンテキスト連携
*   JBWOSのタスクをクリックしたら、即座に建具エディタが開く (Deep Link)。
*   JBWOSのタスクメモに、使っている材料や寸法が自動で転記される。

## 3. ユーザーへの確認事項 (Discussion Points)

1.  **「追加したら一覧にもタスクが増える」の真意は？**
    *   単に忘れないためのメモ代わり？
    *   それとも、JBWOS上で本格的に工程管理をしたい？

2.  **建具プラグイン切り離しの影響**
    *   「JBWOSから分離したい」という要望と、「密接に連携させたい」という要望のバランス。
    *   理想: 建具プラグインは独立して動くが、JBWOSが「インストール」されている場合のみ、アダプターを介してタスクを投げる（Event Busパターン）。

3.  **完了の定義**
    *   JBWOS上の「完了」は、その日の作業の完了か、その建具の完成か。
