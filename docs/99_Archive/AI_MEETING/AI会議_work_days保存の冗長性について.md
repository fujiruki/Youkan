# AI専門家会議：work_days 保存の冗長性について

## 1. 会議の定義
- **テーマ**: `work_days`（目安時間に基づく稼働日数）を、バックエンド（PHP/SQLite）とフロントエンド（IndexedDB）の両方で永続化することの是非。
- **ゴール**: 冗長化が本当に必要か、それとも技術負債（二重管理）になるのかを多角的に議論し、最終的な保存方針を決定する。
- **前提条件**: 現状、建具アイテム（Door）は Jw_cad プラグインとの連携を考慮して IndexedDB に保存されているが、共通アイテム（General）は PHP 経由で SQLite に保存されるハイブリッド構成である。

## 2. キャスティング
1.  **システムアーキテクト（技術・実装担当）**: データの単一性（Single Source of Truth）を重視。同期の複雑さを懸念。
2.  **JWCADプラグイン・スペシャリスト（ローカル連携担当）**: プラグイン側からの読み込み要件（オフライン/高速レスポンス）を代表。
3.  **UX・状態管理スペシャリスト（ユーザー視点担当）**: 画面遷移時の瞬時表示（キャッシュ）と操作のレスポンスを重視。
4.  **データ整合性・品質・リスク担当**: データの不整合や、フィールド増加に伴うマイグレーションコストを指摘。

## 3. ラウンド構成

### 【ラウンド1】初期提案（Divergence）

- **システムアーキテクト**:
    「基本的に、同期が必要なフィールドを増やすのは反対だ。`work_days` は `estimatedMinutes` から算出可能な計算値であるなら、保存すら不要かもしれない。もし保存が必要なら、クラウドを主とし、IndexedDB は揮発性のキャッシュに留めるべきだ。」

- **JWCADプラグイン・スペシャリスト**:
    「Jw_cad プラグイン側が、オフライン時に建具の納期や工数を確認するニーズがある。プラグインが IndexedDB を直接見ている、あるいはローカルブリッジ経由で読み込む場合、IndexedDB 側に最新の `work_days` がないと、プラグイン側で再計算が必要になり、ロジックの二重実装（PHP側とプラグイン側）が発生してしまう。」

- **UX・状態管理スペシャリスト**:
    「ダッシュボードを多用するユーザーにとって、PHP サーバーからのレスポンスを待たずに、IndexedDB から読み込んだ推定値を即座に表示できるメリットは大きい。UIのガクつきを防ぐための『一時的な記憶』としては IndexedDB は非常に有用だ。」

- **データ不整合リスク担当**:
    「クラウドとローカルで値がズレた際、どちらを信じるべきか？ `work_days` のような『計算に基づき、かつ手入力も許可する』値は、不整合が起きた際にユーザーが混乱する。冗長化するなら、明確な同期優先順位（例：常にクラウドが上書き）が必要だ。」

### 【【ラウンド2】交差レビュー（Debate）

- **システムアーキテクト → スペシャリスト達**:
    「プラグイン側での計算重複を避けるために IndexedDB に保存するというのは一見合理的だが、そもそも `work_days` が『工数 ÷ 480分』という単純な計算結果でしかないなら、保存する手間の方が大きいのではないか？」

- **JWCADプラグイン・スペシャリスト → アーキテクト**:
    「ユーザーは詳細画面で `work_days` を手入力（Dirtyフラグ）で上書きすることができる。つまり、単純な計算結果（Derived Value）ではなくなっている。この『手入力されたこだわり（意図）』をプラグイン側でも尊重するには、保存が不可欠だ。」

- **データ不整合リスク担当 → UX担当**:
    「キャッシュとしての IndexedDB は認めるが、今回の実装は `JBWOSRepository` という永続化層の仕様として組み込まれている。これは『一時的な表示用記憶』を超えて、二つのマスターデータが存在することになりかねない。特に建具（Door-）アイテムにおいてクラウド側への保存漏れが起きないか？」

### 【ラウンド3】統合・収束（Convergence）

- **ファシリテーター**:
    「議論を整理します。`work_days` は単なる計算値ではなく、**ユーザーが手入力で調整可能な永続化プロパティ**へと進化しています。そのため、保存自体は必須です。」
- **採用案**:
    - **ハイブリッド永続化の継続**: 建具アイテム（JWCAD連携分）については、オフラインでのプラグイン参照を保証するため IndexedDB とクラウドの両方に保存する。
    - **クラウド主権の確立**: クラウドが接続可能な場合、IndexedDB は常にクラウドのデータ（SQLite）で更新（リフレッシュ）される「ミラー」として振る舞う。
- **弃却案**:
    - 「IndexedDB のみを正解とする」案は、複数デバイス利用時の同期性を損なうため棄却。
    - 「保存せず毎回計算する」案は、ユーザーの手入力調整（Dirty機能）を破壊するため棄却。

## 4. 最終結論

### 結論
**条件付き Yes（冗長化は『目的』ではなく『Offline/Bridgeアクセシビリティのための中間実装』として継続）**

### 採用理由
1.  **ユーザーの入力意思の尊重**: 詳細画面で調整された `work_days` は「データの意図」であり、失われるべきではない。
2.  **プラグイン連携**: JW_cad というローカル環境のツールが Web アプリを介さずデータを確認する際の、ブリッジとしての役割を IndexedDB が果たしている。
3.  **パフォーマンス**: 巨大なタスクリストを瞬時にレンダリングするためのローカルソースとして機能する。

### 棄却理由
- 「純粋な冗長化（バックアップ目的）」は設計を複雑にするだけなので棄却。あくまで「アクセス経路の多様性（Web API vs Local Bridge）」への対応とする。

### 想定リスク
- クラウド保存失敗時に IndexedDB だけが更新された場合の不整合。→ エラー発生時の `refreshAll`（再取得による上書き）を徹底することで緩和する。

### 次アクション
1.  詳細画面カレンダーに、ユーザー要望の「フィルタ選択ボタン」を実装する。
2.  その際、アイテムの所属（個人/会社）に応じてデフォルトのフィルタを切り替える。
