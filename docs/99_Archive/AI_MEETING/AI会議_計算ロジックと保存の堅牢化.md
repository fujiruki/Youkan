# AI会議：量感計算ロジックの構造化と `work_days` 保存の堅牢化

## 1. 会議の定義
- **テーマ**: 量感計算ロジックの構造的改善と、`work_days` 更新時のデータ整合性確保。
- **ゴール**: 
  1. 重複した計算ロジック（ダッシュボード vs 詳細）の統一。
  2. 休日判定と稼働枠設定の干渉（背景色の矛盾等）の解消。
  3. 二重保存（SQLite/IndexedDB）による不整合を防ぐカプセル化設計。
- **前提条件**: JBWOS 憲法（オフライン優先、直感的な量感表示）を遵守すること。

## 2. キャスティング
- **司会 (Antigravity)**
- **専門家1（技術・実装）**: ロジックの共通化、カプセル化、型定義の厳格化を重視。
- **専門家2（ビジネス）**: 開発効率と長期的な保守コスト（技術負債の抑制）を評価。
- **専門家3（UX・ユーザー視点）**: 「なぜこの色なのか」が直感的にわかる表示の整合性を追及。
- **専門家4（リスク・整合性）**: SQLite と IndexedDB の不一致によるデータ損失やバグを防ぐ。

---

## 3. 議論（ラウンド構成）

### 【ラウンド1】初期提案
- **技術**: `QuantityEngine` に `internal_runAllocation` を用意。ダッシュボードも詳細画面もこれを通す。また、`work_days` 更新は `ViewModel` の単一メソッドに閉じ、そこで SQLite/IndexedDB 両方へのクエリを発行するべき。
- **ビジネス**: 共通化により、将来的な「稼働率計算アルゴリズム」の変更時の影響範囲を1箇所に抑えられる。これは必須。
- **UX**: 背景色のグレー（休日）が稼働枠設定があるのに表示されるのは混乱の元。「枠がある日は稼働日」というシンプルルールに寄せるべき。
- **リスク**: ネットワークが不安定な際、SQLite の更新が失敗したのに IndexedDB だけ更新される事態（またはその逆）が最も危険。アトミック性を意識したリトライまたは不整合検知が必要。

### 【ラウンド2】交差レビュー
- **技術 → UX**: 「枠がある日は稼働日」とする場合、カレンダー全体が白くなりすぎて祝日がわかりにくくならないか？
- **UX → 技術**: 祝日はアイコンや薄い境界線で示すべきで、大きな「塗り」としての背景色は稼働負荷（量感）に専念させるのが正しい。
- **リスク → 技術**: ViewModel のメソッドに閉じるだけでは不十分。Repository レイヤーで「二重保存」を強引に実行するのではなく、Syncキューのような仕組みを将来的に検討すべきだが、今は少なくとも「SQLite成功時のみIndexedDB」を徹底する。
- **ビジネス → 全員**: リファクタリングによる既存データへの影響（マイグレーション）はないか？ 
  - **回答**: ロジックのみの変更なので、データ移行は不要。

### 【ラウンド3】統合・収束
- **論点整理**: 
  1. `QuantityEngine` をステートレスなユーティリティから、より構造化されたロジックコアへ。
  2. 休日判定を「デフォルト設定」とし、個別設定（キャパシティ）を常に優先する。
  3. `work_days` 等のメトリクス更新を `JBWOSViewModel.updateMetrics()` に一本化。

---

## 4. 最終結論

### 結論
**採用**: 量感計算ロジックの共通化（Single Source of Truth）、および ViewModel による更新操作のカプセル化。

### 採用理由
- ダッシュボードと詳細画面での「色の不一致」を構造的に排除できる。
- 二重保存による矛盾のリスクを、操作窓口を制限することで最小化できる。

### 想定リスク
- 共通化により、万が一ロジックにバグが混入した場合、システム全体（カレンダー、ガント、詳細）に影響が波及する。→ ユニットテストの強化が必要。

### 次のアクション
1. `QuantityEngine` のリファクタリング（分配コアメソッドの抽出）。
2. `useJBWOSViewModel` に `updateItemMetrics` を実装。
3. `capacity.ts` とカレンダー表示コンポーネントでの休日色の扱いを整理。
