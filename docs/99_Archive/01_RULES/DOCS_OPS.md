# AI_DOCS_OPS v1.0
― ドキュメント維持・管理ガイドライン ―

## 1. 基本原則
ドキュメントは「コードの地図」であり、「過去の記憶」である。
コードが変更された瞬間、ドキュメントも更新されなければならない。
**「コードとドキュメントの乖離」はバグと同義である。**

## 2. ディレクトリ構造と格納ルール

| ディレクトリ | 内容 | 更新頻度 | 誰が書くか |
| :--- | :--- | :--- | :--- |
| `docs/` | ルート | | |
| ├── `00_CONTEXT.md` | **全体地図**。プロジェクトの目的、現状、主要リンク。 | 毎回確認 | Arch/Librarian |
| ├── `01_RULES/` | **法典**。開発ルール、ワークフロー、運用規定。 | 低 | Ops |
| ├── `10_ARCHITECTURE/` | **設計図**。DBスキーマ、データフロー、システム構成。 | 中 | Arch |
| ├── `20_DECISIONS/` | **判例集**。AI会議議事録、ADR。意思決定の履歴。 | 追記のみ | Facilitator |
| ├── `30_MANUALS/` | **取説**。ユーザー向けマニュアル。 | 高 | User/AI |
| └── `ARCHIVE/` | **倉庫**。古くなった仕様書、廃止された機能のドキュメント。 | - | - |
| `src/**/README.md` | **詳細仕様**。各機能の実装詳細、使い方、注意点。 | 高 | Dev |

## 3. 変更時の運用フロー

### ケースA: 機能の仕様変更・追加
1.  **AI会議/検討**: 方針が決まったら `20_DECISIONS/YYYYMMDD_機能名.md` を作成し、決定理由を保存。
2.  **地図更新**: もしその機能が主要なものなら、`00_CONTEXT.md` の機能一覧やステータスを更新。
3.  **詳細更新**: `src/features/機能名/README.md` を作成または更新。
    - **Note**: 既存の仕様書 (`10_ARCHITECTURE` 等) が古くなる場合は、古い箇所に「以下は古い。最新は `src/...` を参照」と追記するか、古くなったファイルを `ARCHIVE/` に移動する。

### ケースB: リファクタリング・構造変更
1.  **地図更新**: `00_CONTEXT.md` のディレクトリ構成図や技術説明を修正。
2.  **設計図更新**: `10_ARCHITECTURE/` 内の関連図を修正。

### ケースC: ドキュメント自体の陳腐化発見
1.  削除はしない。
2.  `ARCHIVE/YYYYMMDD_OriginalName.md` に移動する。
3.  または、ファイルの先頭に `> [!WARNING] Deprecated: This document is managed in [New Path]` と追記する。

## 4. Frontmatter (属性情報) の記述
すべてのMarkdownファイル (`DECISIONS`, `ARCHIVE` 除く) に以下を記述し、情報の鮮度を担保する。

```yaml
---
title: ドキュメントのタイトル
status: active # active (有効) | draft (下書き) | deprecated (非推奨)
last_updated: 2026-01-22
owner: 役割名 (例: Arch, Biz)
---
```

## 5. 検索性と文脈理解を高めるコツ
- **User Nuance (Deep Context)**:
    - 決定事項だけでなく、ユーザーが用いた**具体的な比喩や表現**（「脳内のInbox」「隠れ残業のように」等）をそのまま記録する。
    - AIによる過度な要約・抽象化は禁止。「なぜ」の文脈こそが資産である。
- ファイル名には「日付」ではなく「内容」を入れる（`20_DECISIONS` は除く）。
- ファイル内の見出し (`#`) を適切に使い、AIが目次だけで内容を推測できるようにする。
- 重要な用語は `00_CONTEXT.md` の用語集セクションにリンクする。
