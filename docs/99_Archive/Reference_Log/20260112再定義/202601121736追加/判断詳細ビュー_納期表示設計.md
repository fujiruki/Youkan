# 判断詳細ビューにおける納期情報の扱い（最終定義）
最終更新: 2026-01-12 (JST)

本ドキュメントは、
「納期は判断材料として重要だが、圧として表示してはならない」
という思想を、**UI・体験・実装ルール**として最終固定する。

これは、現実（相手・納期がある仕事）と本システムを
**無理なく接続するための重要な補足憲法**である。

---

## 1. 問題意識（なぜ必要か）

これまでの設計では以下が明確だった。

- Todayには判断のみを出す
- 納期はRDDとして内部処理する
- 納期を前面に出さない

しかし、それだけでは次の疑問が残る。

> 判断するときに  
> 「その仕事の納期がいつか」を  
> どこで、どうやって確認するのか？

この点を曖昧にすると、
- 納期を見ずに判断する危険
- 逆に、納期を常時表示して圧を生む危険
の両方が発生する。

---

## 2. 結論（先に固定）

> **納期は「判断詳細ビュー」にのみ表示される。**

- Todayの表面：表示しない
- ExecutionBlock：表示しない
- History：表示しない
- **判断カードを「開いたとき」だけ表示する**

これが唯一の正解である。

---

## 3. UI構造（確定）

### 3.1 Today画面（折りたたみ状態）

```
◯◯建具｜製作開始を判断する
```

- 日付
- 残日数
- 締切表現

**すべて表示しない。**

---

### 3.2 判断詳細ビュー（展開後）

判断カードをタップ／展開すると、
初めて以下の情報が見える。

```
【判断対象】
◯◯建具｜製作開始を判断する

▼ 判断材料
・取付日：10/30
・想定製作日数：5日
・材料リードタイム：3日
・バッファ：2日

（参考）
・RDD_start：10/23
```

#### 表示ルール
- 文字色：通常テキスト
- 赤・強調・警告なし
- 「あと◯日」表現は禁止

---

## 4. RDDの扱い（重要）

- RDDは**内部ロジック**
- ユーザーに見せるのは任意（上級者向け）
- 見せる場合も：
  - 注釈扱い
  - 意味説明付き
  - 期限表現は禁止

---

## 5. なぜこの構造が現実的なのか

### 常時表示しない理由
- 見るたびに焦る
- 他案件との比較が起きる
- 判断が歪む

### 完全に隠さない理由
- 現実の仕事では納期確認は必須
- 判断材料が不足する

👉  
**「自分から見に行く情報」だけにすることで、  
圧と情報を分離している。**

---

## 6. 判断体験の具体例（固定）

### 朝（Today）

```
今日の判断
1. 建具A｜製作開始を判断する
```

→ この時点では焦らない

---

### 判断詳細を開く

```
取付日：10/30
（あ、思ったより近いな）
```

→ 冷静に現実を把握できる

---

### 判断

- Yes：今の条件なら引き受けられる
- Hold：条件次第
- No：今回は無理

👉  
**納期を見た上で断る、が自然にできる**

---

## 7. 思想テスト追加項目（必須）

### I. 納期圧分離テスト

- 納期がTodayの一覧に表示されていないか
- 残日数・締切表現が前面に出ていないか
- 納期情報が判断詳細に閉じているか

YESが一つでもあれば思想違反。
