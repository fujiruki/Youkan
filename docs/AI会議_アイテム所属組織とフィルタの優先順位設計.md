# AI会議_アイテム所属組織とフィルタの優先順位設計

## 1. 会議の定義
- **テーマ**: カレンダーの「フィルタ状態」と「アイテム自体の所属組織」が競合または共存する場合のキャパシティ（目安期間）算出ルールの再定義
- **ゴール**: 
    - ユーザーが詳細画面（Modal）で目安期間を見たとき、どのコンテキスト（自分の総時間、アイテムの会社の時間、現在の表示フィルタ）を基準とした計算結果を表示すべきかを決定する
- **前提条件**:
    - AI開発憲法「 Judgment-Free」: ユーザーが複数の仕事をどう調整しているかの「現実」を正しく映し出す
    - ユーザーの追加指摘: 「アイテム自体がどの会社に属しているか」を考慮に含める

---

## 2. キャスティング
1.  **ファシリテーター**: 前回の結論（フィルタ優先）と今回の修正案（アイテム優先）のバランス調整。
2.  **技術・実装担当**: 複数テナントを跨ぐ計算（合計値 vs 組織別値）の実装可能性。
3.  **UX・ユーザー視点担当**: ユーザーが「3日間（24h）」というボタンを押した際、何を期待しているか。
4.  **ビジネス担当**: 「会社Xの仕事」をしている時に、自分の「他の会社の持ち時間」を使って計算されることの是非。

---

## 3. ラウンド構成

### 【ラウンド1】初期提案（Divergence）

**技術担当**: 
「現在は `tenantId` が一つしか渡せませんが、`item.tenantId` と `currentFilterId` の両方を考慮する必要があります。もし『会社A』のタスクを『全体（All）』で見ているなら、会社の枠（Budget）に余裕がなくても、自分の総時間（Standard）に空きがあればハイライトを出すべき、という解釈もできます。」

**UX担当**: 
「ユーザーはアイテムの所属組織を知っています。もし『会社A』のタスクなら、その日のキャパのうち『会社Aに割り当てた枠』の中でいつ終わるかを見たい。しかし、もしカレンダー全体が『会社B』でフィルタリングされているなら、そもそもそのタスクの青枠を表示すべきか？という矛盾も生じます。」

**ビジネス担当**: 
「理想は二段構えです。基本は『アイテムが属する組織のキャパ』で計算し、その組織に割り当てられた枠（Reality）の中で見積もる。ただし、カレンダーのフィルタが『All』の時は、その仕事が自分の人生のどの時間を占有するかを示すために『全キャパ』で計算すべきです。」

**リスク担当**: 
「『アイテムの会社』を無視して『All』の巨大なキャパで計算してしまうと、実際には会社から許可された作業時間を大幅に超えてしまうリスクに気づけません。常に『アイテムの所属組織のキャパ』をベースにし、不足分を警告する形が安全です。」

### 【ラウンド2】交差レビュー（Debate）

**UX担当 → ビジネス担当**: 
「『アイテムの組織優先』に固執しすぎると、フリーランスが自分のプライベート時間を削って仕事をする『現実』を表示できなくなります。アイテムの所属組織に枠がない場合でも、『自分のフリー時間』を使えば算出可能です。」

**技術担当 → リスク担当**: 
「技術的には `DailyCapacity = Max(アイテムの会社枠, (Allフィルタ時のみ) 自分のフリー時間含めた全量)` といったフォールバックが可能です。これなら『組織を尊重しつつ、自分の全時間も活用する』ことができます。」

**ビジネス担当 → UX担当**: 
「ユーザーの指摘は『アイテムの組織を考慮に入れて』ということなので、
1. アイテムが組織に属しているなら、その組織の枠を最優先で使う
2. もし組織フィルタがかかっているなら、そのフィルタを尊重する
という、より『狭い方』に合わせるのがスケジュール調整としては正しいですね。」

### 【ラウンド3】統合・収束（Convergence）

**ファシリテーターによる整理**:
1.  **優先順位の決定**: 
    - **原則**: 「アイテムが属する組織」のキャパシティを算出のベースとする。
    - **フィルタの役割**: カレンダーのフィルタがその組織と一致している、または `All` である場合に表示する。
    - **フォールバック**: もし特定の組織フィルタがかかっている場合は、その組織の枠のみで計算する。
2.  **算出コンテキストの動的切替**:
    - `DecisionDetailModal` では、`QuantityEngine` に渡す `tenantId` を `(カレンダーフィルタがAll ? undefined : (フィルタID || アイテムのテナントID))` とする。
    - ただし、アイテムの組織とフィルタが不一致（フィルタが会社A、アイテムが会社B）の場合は、そのタスクをカレンダーから消去するか、あるいは計算対象外とする。

---

## 4. 最終結論

### 結論: アイテム属性優先＋フィルタ制限型の算出ロジック
- **詳細画面のキャパシティ採用ルール**:
    1.  **カレンダーフィルタが `All` の場合**:
        - アイテムの `tenantId` に設定されている時間が「0より大きい」なら、その「組織枠」を優先して計算（消費）する。
        - アイテムの `tenantId` の設定が「0」または「未設定」の場合、「一日の全稼働（Standard合計）」を代替として使用し、警告（ガイド）を表示する。
    2.  **カレンダーフィルタが `特定組織` の場合**:
        - アイテムがその組織に属しているなら、その組織の枠で算出。
        - 異なる組織のアイテムなら、計算結果を空（青枠なし）にするか、フィルタ組織の枠をそのまま借りる。
- **UI表示の改善**:
    - 警告メッセージで「現在、どの会社の枠（あるいは自分の全時間）を使用して目安を出しているか」を補足する。

### 採用理由
- アイテムの所属組織を尊重しつつ、カレンダーの全体表示（All）時には「自分の24時間」の中での配置を可能にするため。
- ユーザーが「所属組織を考慮してほしい」と意図した「仕事の枠組み」の認識をシステムに取り込む。
