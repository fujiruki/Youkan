# 建具システム開発AI向けブリーフ：判断OS（JBWOS）統合設計
（2026-01-11 / Draft v1.0）

本資料は「建具システム」を、ユーザー（晴樹）が**時間管理から解放され、判断を終わらせる**ための統合システムへ進化させるための設計ブリーフである。  
実装担当AIは、本資料を**一次仕様（憲法 + 製品仕様 + 禁止事項 + 理由）**として扱い、局所的な便利機能の追加で思想を破壊しないこと。

---

## 0. 背景：なぜこの思想が必要か（ユーザーの現実）

### 0.1 ユーザーの仕事特性（高不確実）
- 日付が揺れる（大工工程・現場進行に依存）
- 割り込みが多い（打合せ・電話・現場）
- 正確な時間割が破綻する（製作は1～1.5日/枚、取付は6枚/日など“単位”はあるが予定化が破綻しやすい）
- 現場ごとに条件が違い、判断が常に発生する（寸法取り複数回、未確定修理、材料待ち）

### 0.2 現状のツール体験の失敗パターン
- Googleカレンダー：リマインダー・入力容易性は強いが、長期/不確実/多品種には登録が破綻し「入れない→見ない→忘れる」
- 紙手帳：書く気持ちよさはあるが、リマインダーがなく「見返す習慣」がないと無力
- 印刷ツール（週/月）：余白は増えたが、結局「開く習慣」「どう書けばよいか」で停止（億劫状態）
- Notion：プロジェクトと制作物は入れたいが、結局“使うための判断”が増え、未整備に

### 0.3 主要な痛み（核心）
- 「書いたっけ？」「どの日だ？」が頻発＝**判断の未完了が積み上がっている**
- 完璧な判断欲が強く、情報収集するほど決めきれず進まない
- 結果：タスクが増えて疲れる（判断疲労、罪悪感、回避）

---

## 1. ゴール（このシステムが作るべき状態）
**ユーザーを「時間管理」から解放し、「状態管理」へ移行させる。**  
成功条件は生産性ではない。  
**「今日はもう判断しなくてよい」とユーザーが言える日が増えること**が唯一の成功である。

---

## 2. 憲法（破ってはいけない設計原則）
この章は“最優先”。機能提案・UI提案より上位。

### 2.1 憲法レベル（必守）
1. 人は一日に正しい判断を何十回もできない（判断回数を減らすことが主目的）
2. 仕事は進まない前提（遅れ検知は追い打ちになるだけ）
3. 未完了の「判断」が最も疲れる（ToDo増殖よりも判断未完が悪）
4. 納期は事実だが、行動判断の主軸にしない（内部画面に日付圧を持ち込まない）
5. 目的は進捗管理ではなく「判断の完了」（正しさの最適化をしない）

### 2.2 永久禁止（思想違反）
- 遅延アラート、赤色警告、期限超過表示
- 完了率/達成率/%バー/ランキング
- 工数積算の可視化（分・時間の積み上げ）
- AIによる最適スケジュール提案（従えなかった自分を責める構造を作る）
- Inbox状態のままカレンダー登録（判断前を時間世界に送るのは禁止）

---

## 3. アーキテクチャ：3レイヤー構成（優先順位あり）
優先順位は **Layer1 ＞ Layer2 ＞ Layer3**。  
これが逆転すると、ユーザーは再び壊れる。

### Layer1：判断管理（Internal View / 主役）
- 目的：今日の迷いを消す。判断回数を減らす。停止を肯定する。
- 時間軸：持たない（＝日付を見せない）

### Layer2：現実翻訳（Bridge）
- 目的：判断結果だけを、必要最小限の形で時間世界（Googleカレンダー等）へ“そっと置く”
- ここは「デジタルはダメ」を回避する装置。精度を上げない。

### Layer3：対外説明（External View / 従属）
- 目的：顧客・元請けへの説明責任を果たす
- 注意：ユーザーの精神衛生には有害なので隔離する（別画面）

---

## 4. Layer1：判断管理ボード（Decision Board）仕様（完全版）
**時間軸を持たず、判断状態（バケツ）へ振り分ける。**

### 4.1 バケツ（確定）
| バケツ | 意味 | 心理・制約 |
|---|---|---|
| Inbox | 未判断（とりあえず置き場） | 溜めてOK。着手を強制しない。10件超でも「今日は1件だけ出せばOK」|
| Waiting | 外部待ち（寸法/承認/材料） | 止まってるのは自分のせいではない。安心して放置できる |
| Ready | 今日向き合う覚悟を決めたもの | **最大2件**。3件目はブロック。ここ以外見ない |
| Pending | 意図的放置（塩漬け） | 普段は折りたたみ。視界ノイズを消す |
| Done | 判断完了（出口） | 履歴・振り返りはしない。次を勧めない |

### 4.2 カードでできる操作（3択固定）
カード選択時、選択肢はこの3つのみ：
1) 仮決めする → Ready  
2) 待ちにする → Waiting（理由1行）  
3) 忘れる → Pending  

※「詳細に分解」「工程を全部書く」「正しい順序決め」などを促すUIは禁止。

### 4.3 「完了と停止の肯定」（重要UI）
- Ready→Done時の表示文（デフォルト固定）：  
  **「今日は、ここまでで十分です。」**
- Readyが空になったら：  
  **「今日はもう、やるものはありません」**  
  Inboxへ誘導しない（停止を肯定する）

---

## 5. Layer2：Bridge（判断→Googleカレンダー）完全版仕様（要点）
Googleカレンダーは管理ツールではなく、**判断結果の保管棚**としてのみ使用する。

### 5.1 連携原則（必守）
- 判断前（Inbox/Waiting/Pending）は送らない
- **Ready→Done になったものだけ**が対象
- 作業単位ではなく「作業ブロック（まとまり）」のみ登録
- 精度を上げない（時間帯：午前/午後/終日のみ）
- 通知しない（リマインド常にOFF・変更不可）

### 5.2 連携UI
ボタン名：**「この判断を、カレンダーに置く」**  
サブテキスト：  
「予定を管理するためではありません。判断を終えた証を残します。」

### 5.3 カレンダーイベント仕様
- タイトル：`【判断済】◯◯（作業ブロック）`
- 時間：午前/午後/終日（デフォルト終日）
- 説明：  
  「これはスケジュール管理用の予定ではありません。判断が完了した作業ブロックの記録です。詳細→Internal View」
- 通知：なし（強制）
- 色：非警告色（赤・オレンジ禁止）

### 5.4 逆流禁止（重要）
- カレンダーで日時変更しても内部状態は変化しない
- カレンダーで削除しても Done は維持
（外部を正にすると、内部が時間圧に支配されるため）

---

## 6. Layer3：External View（対外説明モード）仕様（隔離必須）
目的：顧客・元請けへ誠意ある説明を支援する。  
ただしユーザーが見るとプレッシャーになるため、**Internalと完全隔離**する。

### 6.1 機能
- ぼかしガント：日単位禁止。週/旬（上旬・中旬・下旬）のみ
- 状況説明テンプレ生成：謝罪・断定日付を避け、事実ベース

テンプレ例：
- Waiting：「必要情報/材料の確認待ち。確認が取れ次第次工程へ」
- Ready：「加工工程に入っており大きな問題なく進行」
- Inbox：「内容確認段階。全体状況を見て対応順を判断」

---

## 7. 思想違反テスト（よくある機能要求への判定）
以下は会議で頻出するため、初期から判定ルールとして固定する。

### 7.1 期限表示（Internal）
- 判定：違反  
- 理由：期限＝再判断トリガー。内部は“迷いを消す”ための日付禁止。

### 7.2 推奨順（Ready内の優先度）
- 判定：違反  
- 理由：最適化＝完璧主義刺激。Readyは覚悟の事実のみで十分。

### 7.3 進捗%・完了バー
- 判定：即違反  
- 理由：評価装置化し、罪悪感を増やす。

### 7.4 Inbox多すぎ警告
- 判定：違反  
- OK代替：「今日は1件だけ出せばOK」など安心誘導のみ。

### 7.5 Waiting長期放置通知
- 判定：違反  
- 理由：Waitingは自責を外す装置。通知は自責を呼ぶ。

### 7.6 AI最適スケジュール提案
- 判定：最重要違反  
- 理由：従えなかった自分を責める構造を作る（逆効果）。

---

## 8. 現状の「TODO＋横メモ」運用の評価（統合設計に活かす）
ユーザーは現在、TODOリストの横に常時メモ欄があるツールを使っている。
メリット：同時に視界にあり、書きやすく、TODO見返しのトリガーになる。
デメリット：時間が経つと検索以外に再集合できず「プロジェクト単位でメモを全部見る」ができない。

### 8.1 判定（結論）
- 短期：有益（探索コスト低い）
- 長期：有害になりやすい（判断材料が散らばり、再判断が増える）

### 8.2 JBWOS的な正しい位置づけ
- そのツールは**Inbox専用の捕捉装置（Capture）**として扱う
- 役割：思いついたことを即書く（判断しない）
- 必須ルール：**判断が終わったら、JBWOSへ1行で昇格**する  
  （原文メモを捨ててもよい。判断結果だけ残す）

---

## 9. 実装ロードマップ（統合→分離）
ユーザー方針：
1) まず建具システムに統合（自分の現実データで思想検証）
2) うまく回り、再現できたら建具部分をプラグイン化して汎用OS化

### 9.1 なぜ統合が先か（理由）
- 判断疲労は建具・現場・材料・取付など“実データ”付きで起きている  
  抽象化から入ると失敗する
- 思想の検証は「晴樹が壊れないか」でしか測れない（机上の正しさは無意味）
- 入口（判断材料）と出口（判断結果）が分断されると、未完了判断が再発する

### 9.2 分離してよい条件（判定）
- 「今日はこれで十分」と言える日が増えた
- 夜や休日の思考の回り続けが減った
- 完璧な判断欲が弱まった
（YESが2つ以上なら思想は本物）

## 10. MVP（最初に作るべき最小だが有効な統合）
※ここは「最小構成」だが、思想を壊さずに効く要素に限定する。

### 10.1 必須
- Internal View（Decision Board）
- バケツ（Inbox/Waiting/Ready/Pending/Done）
- Ready最大2件制限
- 3択操作（仮決め/待ち/忘れる）
- Doneメッセージ＆停止肯定
- Judgment→Googleカレンダー連携（判断済ブロックのみ・通知なし）

### 10.2 後回し
- External View（ぼかしガント、説明テンプレ）
- 高度な検索・分析
- 他アプリ連携（Notionなど）

## 11. 会議用：最重要の確認質問（仕様検討会議の入口）
実装AI側で仕様検討会議をする際、以下を必ず議題冒頭に置くこと。

1) この提案は「判断回数を減らす」か？（増やすなら棄却）
2) この提案は「罪悪感を増やさない」か？（増やすなら棄却）
3) この提案は「完璧主義を刺激しない」か？（刺激するなら棄却）
4) この提案は「見なくても困らない」か？（依存を作るなら棄却）

## 12. 付録：データモデルの最小拡張（例）
（既存のProject/Door/Item構造を維持し、Viewを変える方針）

```ts
type JudgmentStatus = 'inbox' | 'waiting' | 'ready' | 'pending' | 'done';

interface DoorOrItem {
  id: string;
  title: string;
  projectId?: string;

  // Judgment OS
  judgmentStatus: JudgmentStatus;
  waitingReason?: string;

  // Buffers（小さく表示。並び替えに使わない）
  weight?: 1 | 2 | 3;           // 1:Light 2:Medium 3:Heavy（時間ではない）
  proximity?: 'this_month' | 'next_month' | 'someday'; // 日付の代替ラベル
}
```

## 13. 最後に：この製品は何か（定義）
これはスケジューラではない。タスク管理でもない。

**「不確実な仕事をする人が、判断を終わらせるためのOS」**である。  
正しい計画を作らない。  
**これ以上考えなくていい状態**だけを作る。
