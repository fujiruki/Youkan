# Estimation & Calendar UI/UX Discussion

## 1. Estimation Aggregation: Hybrid Approach
ユーザーの要望「親タスクの目安とサブタスクの目安の関係性」に対する提案仕様です。

### コンセプト
**「トップダウン（概算）」から「ボトムアップ（積上）」へのシームレスな移行**

- **初期段階（概算）**:
    - プロジェクト作成直後など、詳細が決まっていない段階。
    - 親タスクに直接「目安日数：10日」のように入力可能。
    - これにより、「まずは枠だけ確保する」運用が可能。

- **詳細段階（積上）**:
    - サブタスクが追加され、それぞれに目安時間が入力された段階。
    - 親タスクの目安時間は、**サブタスクの合計値（Sum）** を正とする。
    - 建具製作においては「ドア3枚（2日×3）＝6日」のような加算的な工数計算が基本であるため、この方式が適合する。

### UI挙動
1.  **入力フィールドの切り替え**:
    - サブタスクが無い場合：通常の数値入力フィールド。
    - サブタスクが有る場合：自動計算された合計値を表示（Read-only または グレーアウト）。
        - *Option*: 手動入力値と合計値に乖離がある場合、「概算(10日) vs 積上(12日)」のように併記し、アラートを出すか、ユーザーに同期ボタンを押させる。

## 2. Calendar Visualization
親タスクとサブタスクのカレンダー上での表現方法について。

### コンセプト
**「Span (期間)」vs「Point (実施日)」**

- **サブタスク（Point/Small Span）**:
    - 具体的な作業単位。
    - カレンダー上には、具体的な「実施予定日（Doing Date）」や「My期限」に配置されるカードとして表示。
    - 完了チェックが可能。

- **親プロジェクト（Span/Container）**:
    - 複数のサブタスクを包含する大きな枠。
    - 通常のタスクカードとして特定の日付に置くのではなく、**「期間バー（ガントチャート風）」** や **「背景ブロック」** として表現する。
    - または、未配置のサブタスクがある場合、それらの合計ボリュームを持つ「未定ブロック」として仮置きエリアに表示する。

### 具体的なUI案
1.  **カレンダービュー**:
    - 親プロジェクトは「ヘッダー」または「背景色付きの帯」として、開始日〜納期（または現在〜納期）の範囲を可視化する。
    - その帯の上に、個々のサブタスクカードが配置されるイメージ。
2.  **Dashboard/Today**:
    - 「今日やること」には具体的なサブタスクのみ上がってくる。
    - 親プロジェクトは「コンテキスト（バッジ）」として表示されるのみ（現状の実装）。
