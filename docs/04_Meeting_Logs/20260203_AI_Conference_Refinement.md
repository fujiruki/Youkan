# AI専門家会議：JBWOS 技術設計の抜本的改善

**日時**: 2026年2月3日
**参加者**:
1.  **システムアーキテクト（SA）**: データ構造と長期的な保守性の責任者
2.  **リードデベロッパー（LD）**: 実装の容易さとフロントエンドのパフォーマンス（即時性）の責任者
3.  **UI/UXデザイナー（UX）**: ユーザー体験と名称の直感性、状態の透明性の責任者

---

## 議題1：`title` と `name` の混在について

**UX**: プロジェクトは `name`、タスクは `title` という現状は、ユーザーにとっても開発者にとっても混乱の元です。特に詳細画面のパンくずリストで両方が並ぶ際、コード上で条件分岐が発生するのは無駄です。

**SA**: 同感です。DB層でのリネームは影響範囲が大きいですが、APIの出力レベルで `title` に統一すべきです。プロジェクトもタスクも「決定事項のタイトル」という概念で統一しましょう。

**LD**: 実装上は、TypeScriptの型定義を `Item` プロトコルとして統一し、すべてのエンティティに `title: string` を持たせるのがスマートです。古い `name` への参照は非推奨（Deprecated）として段階的に廃止しましょう。

**結論**: **`title` に統一**。すべてのUI表示と型定義において `title` を正しく参照するようにリファクタリングする。

---

## 議題2：データの冗長性と不整合のリスク

**LD**: 今の `projectTitle` などの「重複持ち」は、オフライン性能や即時性のために導入されましたが、同期の不備で古い名前が残るバグが起きています。

**SA**: **「プロキシ・パターン」** を提案します。アイテム自体は `projectId` だけを持ち、ViewModelのSelectorが常に最新のプロジェクト一覧（別ストリーム）から `title` をマッピングして表示用の仮想プロパティを生成する仕組みです。

**LD**: Reactの `useMemo` でラップすればパフォーマンスへの影響は皆無です。これで「プロジェクト名を変えたのにタスク側の表示が変わらない」という不整合は物理的に消滅します。

**結論**: **重複プロパティ（`projectTitle`等）の廃止**。表示時にIDから動的に解決するアーキテクチャへ移行。

---

## 議題3：コンテキスト継承の自動化

**SA**: `throwIn` で毎回プロジェクトIDを手動で渡しているのが今の問題です。**「ActiveContext」** という概念を導入すべきです。

**LD**: ですね。UIの深層（全体一覧２のクイック入力など）が「今自分がどこにいるか」を知らなくても、`useContextService` から現在のプロジェクト情報を自動取得して `addItem` に注入する仕組みを作れます。

**結論**: **`ContextService` による自動注入**。UIコンポーネントが「どのコンテキストにいるか」を意識せずに正しい親子関係を持てるようにする。

---

## 議題4：ステータスの多重定義問題（ステートマシン）

**UX**: 今の `status: focus` が「今日やる候補（Candidate）」と「実行中（Execution）」の両方を指しているのは、ユーザーの心理モデル（決定→着手→完了）とズレています。

**SA**: **「Lifecycle（工程）」と「Engagement（熱量）」の2軸管理** にしましょう。
- **Lifecycle**:判断(Judgment) → 準備(Preparation) → 完了(Done)
- **Engagement**: 待機(Idle) → 焦点(Focus) → 実行(InProcess)

**LD**: ステートマシンライブラリ（XStateなど）を使うまでもなく、この2軸を明確にするだけで、「実行をやめたら準備に戻るのか、判断済みに戻るのか」という迷いがコード上から消えます。

**結論**: **内部状態の2軸化**。UIでは直感的なラベルを使いつつ、内部ロジックでは「どこにいるか（工程）」と「何をしているか（熱量）」を分離する。

---

## 議事録追記：ステータス2軸化の定義対応表（マッピング案）

現在の複雑なフラグ（status, isEngaged, is_today_commit等）を、以下の論理的な2軸に整理します。

| 現状の状態 / フラグ | 軸A：工程 (Stage) | 軸B：熱量 (Engagement) | UI/視覚表現 |
| :--- | :--- | :--- | :--- |
| **Inbox (インボックス)** | 1. 判断 (Judgment/Inbox) | 静止 (Idle) | リスト表示のみ |
| **Pending (保留/棚)** | 2. 進行 (Active) | 静止 (Idle) | 通常表示 |
| **Preparation (準備等)** | 1.5 準備 (Preparation) | 静止 (Idle) | カレンダー/準備リスト |
| **Candidate (今日やる候補)** | 2. 進行 (Active) | 注目 (Focus) | **青●** (status: focus) |
| **Commit (今日やる決定)** | 2. 進行 (Active) | 決定 (Commit) | リスト上位に固定 |
| **Execution (実行中)** | 2. 進行 (Active) | 燃焼 (Burning) | **青●** ＋ **明滅緑●** |
| **Done (完了)** | 3. 完了 (Done) | 終了 (Closed) | 取り消し線 |

この4点の改善により、JBWOSは「職人の道具」としてより信頼性の高い、メンテナンスしやすいシステムへと進化します。順次実装計画に組み込みましょう。
