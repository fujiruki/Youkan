# JBWOS Global Board UX改善会議議事録

**日時**: 2026/01/19  
**テーマ**: Global Board（トップ画面）の一覧性向上と「安心感」の設計  
**参加者**:
- **PM (Antigravity)**: 進行、ユーザー要望（一覧性・安心感）の代弁
- **Designer A (Information Architect)**: 構造・グリッドシステム専門
- **Designer B (Emotional UX)**: 心理的安全性・色彩設計専門

---

## 1. 現状の課題分析

**PM**: ユーザーから「一覧性を上げたい」「左右2段・3段などで整理しやすくしたい」という強い要望があります。現状のGlobal Boardは、Active → Preparation → Intent → Log と縦一列に並んでおり、以下の問題があります。
1.  **スクロールの強制**: 全体像を見るためにスクロールが必要で、「見えていない部分」への不安（抜け漏れ感）が生じる。
2.  **情報の等価性**: 「今すぐやるべきInbox」と「いつかやるIntent」が同じ幅・同じ重みで並んでおり、メリハリがない。

**Designer A**: 構造的にも、PCのワイド画面において縦積みレイアウトは画面スペースの浪費です。人間の視線はF型やZ型に動くので、重要な情報を「左上」から「面」として展開すべきです。

**Designer B**: 心理面でも、「Active（今）」と「Stock（未来・過去）」が混ざって見えるのがストレスですね。「これは今は考えなくていい」という領域を視覚的に分離することで、ユーザーは「今」に集中でき、安心感が生まれます。

## 2. 解決案の検討

### 案1: 3カラム・均等分割 (The Kanban)
- 左: Inbox/Active
- 中: Preparation
- 右: Intent/Log
- **評価**: 一般的だが、Inboxの領域が狭くなる。Inbox内のアイテム名が長い場合に見づらくなる恐れあり。

### 案2: 2カラム・非対称分割 (The Workbench)
- **左エリア (65-70%)**: **Active / Inbox**
    - 「作業机」。最も広く、視認性を高く保つ。
- **右エリア (30-35%)**: **Stocks (Preparation / Intent / Log)**
    - 「サイドシェルフ」。縦に積む。
    - 背景色を少し落とし、あくまで「控え」であることを示す。
- **評価 (採用)**: ユーザーの「左右で整理したい」という要望に合致しつつ、情報の重み付け（Active優先）も可能。

## 3. 具体的なデザイン方針

**Designer A (構造)**:
- `grid-cols-1 lg:grid-cols-3` をベースにします。
- **Main (Left)**: `lg:col-span-2`。ここに `BucketColumn (Active)` を配置。
- **Side (Right)**: `lg:col-span-1`。ここに `Preparation`, `Intent`, `Log` を `flex-col gap-4` で配置。

**Designer B (感情)**:
- 右カラム（Side）全体を少しグレーアウト（`bg-slate-50/50`など）し、境界線を設けることで、「ここにあるものは安全に保管されているが、今の集中を乱さない」という雰囲気を出しましょう。
- IntentやLogはデフォルトで折りたたむか、コンパクト表示にしても良いかもしれません（今回はまずレイアウト変更のみ）。

## 4. 決定事項（Implementation Plan）

1.  **レイアウト変更**: `JbwosBoard.tsx` のメインコンテナを Grid Layout (2カラム) に変更する。
2.  **レスポンシブ**: モバイル時は従来の縦積みに戻るようにする (`lg:` プレフィックスを使用)。
3.  **視覚的区分**: 右カラムに背景色やボーダーを調整し、Mainとの区分けを明確にする。

## 5. 第2ラウンド議論：ユーザーフィードバックを受けて

**PM**: ユーザーから「今の表示も嫌いではないので切り替えたい」「余白を極限まで削って一覧性を上げたい」「カラム分類ではなく、全部をバーンと見たいだけかも」という非常に鋭いフィードバックを頂きました。

**Designer A (構造)**: なるほど。「分類」よりも「俯瞰」が優先順位として高い可能性がありますね。
それなら、単なる2カラムではなく、**「モード切替」**を前提に設計し直しましょう。

### 検討案：View Modeの多様化

1.  **Story Mode (Current)**:
    -   現状の縦積み。物語のように上から下へ流れる。スマホや、一つ一つ吟味したい時に最適。残すべき。
2.  **Cockpit Mode (New 2-Column)**:
    -   先ほど決めた「ワークベンチ」。作業と保管を分ける。
3.  **Panorama Mode (All-in-One)**:
    -   **「画面全体に全部表示」**への回答。
    -   Active / Preparation / Intent / Log を、画面の幅いっぱいに `Grid` で敷き詰める。
    -   例えば4カラム（Active | Prep | Intent | Log）にするか、Masonry（レンガ積み）にするか。

**Designer B (感情)**:
    -   「不安」の正体は「隠れている情報」です。
    -   Panorama Modeでは、**文字サイズを小さく、余白（Padding/Margin）を限界まで削った「Super Compact」**なデザインにしましょう。
    -   情報はギュッと詰まっているが、整然としている。株価ボードや飛行機の計器類のような「高密度な機能美」を目指すべきです。

### 決定事項 (Revised Plan)

1.  **View Switcher (切替機能)の実装**:
    -   ヘッダーにビュー切替ボタンを設置（アイコンで表現）。
    -   **Standard (縦積み)** vs **Cockpit (2カラム)** vs **Panorama (全一覧)**。
    -   ユーザーの「とりあえず全部見たい」に応えるため、まずは**Panorama (4カラム Grid)**の実装を優先します。

2.  **Super Compact Style**:
    -   新しいビューでは、カードのパディングを `p-4` から `p-2` へ。
    -   フォントサイズを `text-sm` から `text-xs` へ。
    -   バケット間の隙間もなくし、画面を「埋め尽くす」感覚を作る。

**PM**: これにより、ユーザーは「気分別」に画面を選べます。不安な時はPanoramaで全貌を把握し、集中したい時はStandardやCockpitに戻る。これが最適解でしょう。
