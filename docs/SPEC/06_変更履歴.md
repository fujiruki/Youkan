## 2026-07-09: R-069 `/stocks` 未認証アクセス脆弱性の緊急修正

### 背景
R-050案の設計会議調査で、本番稼働中の `backend/index.php` に登録された `/stocks` 系ルート（`GET/POST /stocks`、`PUT/PATCH /stocks/{id}`、`POST /stocks/{id}/assign`）が `StockController.php` にディスパッチされる際、`authenticate()` が一切呼ばれず `tenant_id` によるスコープ分離も無いまま到達可能であることが判明した。特に `assign()` はリクエストの任意の `userId` を検証せず `items` テーブルへ直接 `INSERT` できるため、未認証の第三者が他テナントの `items` にレコードを書き込める状態だった。

### 調査結果
- フロントエンドで `/stocks` API を呼ぶのは `useStocks.ts`（呼び出し元は `DispatchScreen.tsx` のみ）のみ。`DispatchScreen.tsx` と `CompanyDashboard.tsx` はアプリのルーティング・画面遷移から一切参照されていない死蔵コードであることを確認済み（相互参照のみ）。
- したがって `/stocks` API は実際には稼働中のどの画面からも呼ばれておらず、実害のある機能を壊さずにルート自体を無効化できると判断した。

### 決定事項
| 種別 | 内容 | 備考 |
|:--|:--|:--|
| [修正] | `backend/index.php` から `/stocks` 系ルート登録（3ブロック）をコメントアウトし無効化 | `StockController.php` 自体は削除せず残置。将来再導入する場合は `authenticate()` と `currentTenantId` スコープを実装してから有効化する旨をコード内コメントに明記 |
| [追加] | `backend/tests/test_stocks_unauthenticated_access.php` を新規追加 | PHP内蔵サーバーを一時ディレクトリで起動し、Authorizationヘッダーなしで `/stocks` 系3エンドポイントを叩き 404 になることを検証する統合テスト |

### 影響範囲
- `backend/index.php`（`/stocks` 系ルート3ブロックの無効化のみ。他ルートは無変更）
- フロントエンド変更なし（`useStocks.ts`/`DispatchScreen.tsx`/`CompanyDashboard.tsx` は死蔵のまま残置、再利用時は要再設計）
- ブランチ: `fix/stock-controller-auth`（R-050 Phase1作業とは分離）
