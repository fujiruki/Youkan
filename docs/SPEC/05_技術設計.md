## 2. アーキテクチャ

### 2.1 3層積み上げ構造

```
┌─────────────────────────────────────┐
│  Domain Plugin (例: Tategu Plugin)  │  業種特化機能
├─────────────────────────────────────┤
│  Manufacturing Layer (Extension)    │  モノづくり共通機能
├─────────────────────────────────────┤
│  Youkan Core (Platform)             │  汎用タスク管理・意思決定OS
└─────────────────────────────────────┘
```

- **Youkan Core**: タスク管理、カレンダー、ユーザー管理。全ユーザー向け
- **Manufacturing Layer**: 製作時間、原価、現場工数。製造業共通
- **Domain Plugin**: 業種特化。建具のパラメータ、姿図描画、見積ロジック等

### 2.2 Manufacturing Bus

- 複数プラグイン（建具、家具、外注等）を束ねるバス
- アプリ起動時にプラグインが自動登録
- `ManufacturingBus.getSources()`で取得可能

## 5. 設計原則

### QuantityEngine（量感計算エンジン）の責務分離

- QuantityEngineの責務: 「与えられたアイテムリストに対して日別の負荷を計算する」
- フィルタロジックはViewModel層に統一し、QuantityEngineからは除去
- 「何を表示するか（フィルタ）」と「どう計算するか（エンジン）」は別の関心事

### ActiveTask（Now）の永続化

- 「今、この瞬間にやっていること」は常に1つ
- `active_task_id`をusersテーブルに保存
- PCで作業中に離席 → スマホで同じタスクが「Now」として表示（コンテキスト同期）

### リソース隔離ルール

複数アプリが`door-fujita.com`ドメインを共有するため、以下のリソースには`Youkan_`プレフィックスを付与:
- Cookie: `Youkan_session_id`（path=/contents/Youkan/）
- LocalStorage/SessionStorage/IndexedDB
- Cache Storage/Broadcast Channel

### リアクティブ伝播パターン

親→子孫カスケード操作は MVVM の各層で責務分離し、楽観的UI＋確定同期＋ロールバックの3段サイクルで実装する。

#### Model層（Backend / PHP）
- `ItemController::cascadeTenantToDescendants($rootId, $newTenantId): string[]`
- `ItemController::cascadeStatusToDescendants($rootId, $updatesSql, $updateParams): string[]`
- `ItemController::cascadeDeleteDescendants($rootId): string[]`
- いずれも `getAllDescendantIds()` を内部利用、トランザクションで保護
- レスポンスに `affectedDescendantIds` / `deletedDescendantIds` を含める

#### Repository層
- `ApiClient.updateItem/archiveItem/trashItem/restoreItem/destroyItem` の戻り値型を `{ success: boolean; affectedDescendantIds?: string[] }` に拡張
- `CloudYoukanRepository` / `YoukanRepository` は API応答をそのまま返す薄いレイヤ

#### ViewModel層
- `useYoukanViewModel`:
  - `refreshItems(ids: string[])`: 部分リフレッシュ（`refreshAll()` を撤廃）
  - `optimisticCascadeTenant(rootId, newTenantId)`: local state を即時更新＋ snapshot 返却
  - `restoreSnapshot(snapshot)`: ロールバック
- `updateItem` 等のサイクル:
  1. snapshot 取得
  2. 楽観的UI更新（親 + 子孫の local state 即時反映）
  3. API呼出
  4. 成功 → `refreshItems(affectedDescendantIds)` で確定同期
  5. 失敗 → `restoreSnapshot` でロールバック

#### View層
- ユーザー操作（パンくずクリック・右クリックメニュー等）から ViewModel メソッドを呼ぶのみ
- 子孫件数の視覚フィードバック（任意）は別タスク

#### メンテナンス則
- `refreshAll()` の追加使用は禁止（部分refreshの徹底）
- `is_project` 条件分岐の追加は禁止（親子関係で常に判断）
- バックエンドの巨大 `update()` メソッド肥大化を避け、ヘルパー抽出を継続

### Someday状態の技術仕様（R-028）

#### 型定義
- `JudgmentStatus = 'inbox' | 'waiting' | 'focus' | 'pending' | 'someday' | 'done'`
- 既存遺物 `judgmentStatus?: '...someday...'` を整理し、メイン `status` に統合
- `dueStatus?: 'someday'` は独立軸として別議論（本plan外）

#### キャパシティ計算除外
`QuantityEngine.ts` の集計関数で `someday` 状態のアイテムを集計対象外とする。
ガント・カレンダーの容量計算にも影響を与えない。

#### ビュー別表示制御
- パノラマ: 専用バケット（Pending と Log の間）
- 全体一覧: フィルタースイッチ（デフォルト除外）
- ガント・カレンダー: 強制フィルターアウト（表示オプションなし）
- 詳細モーダル: 状態切替ボタン群に追加

#### UI差別化
- アイコン: 💭（または `Cloud` lucide icon）
- カラー: `purple-400/50`（pendingのamber系、focusのblue/indigo と区別できる色）
- ラベル文言: 「いつかやる（自分で寝かせる）」
- pending ラベル: 「保留（外的要因待ち）」

#### マイグレーション不要
既存アイテムは現 status 値を維持。新値 `'someday'` はオプトインで使用される。

#### メンテナンス則
- 状態を6から7以上に増やさない（短期記憶限界）
- pending と someday の差別化UIを維持
- キャパシティ除外ルールを維持
- ガント・カレンダーでの someday 非表示を維持

### getGdbShelf レスポンス構造（R-029で修正）

`CloudYoukanRepository.getGdbShelf()` のレスポンス型は status 数と完全対応する6キー構造:
- `active`: status='inbox' or 'focus'
- `preparation`: status='waiting'
- `intent`: status='pending'
- `someday`: status='someday'  ← R-029で追加
- `log`: status='done'

ViewModel の `refreshGdb` は各 raw state に対して `shelf.<key>` から直接セットする。`activeRaw.filter(status==='someday')` のような派生フィルターは禁止（R-028 の暫定実装を是正）。

### 階層子孫取得の3軸評価ルール（R-029で修正）

プロジェクトフォーカス時の子孫取得 SQL は **id / project_id / parent_id の3軸**で初期行を評価する。

`backend/BaseController.php::getProjectDescendantIds()` の WITH RECURSIVE 初期行:
- 旧: `WHERE id = ? OR project_id = ?` （2軸、parent_id のみで紐づく子孫が漏れる）
- 新: `WHERE id = ? OR project_id = ? OR parent_id = ?` （3軸、3階層以上対応）

これにより、佐礼谷 > 2F組子製作（孫プロジェクト）> 製作（タスク）のような階層で、孫プロジェクトをフォーカスした時に子タスクが正しく取得される。

DFS 版 `ItemController::getAllDescendantIds()` は既に3軸（OR）で評価済み。

### 階層ラッパーの型分離方針（R-030）

`logic/hierarchy.ts` の `HierarchicalWrapper` は discriminated union として定義する。

#### 型定義
```ts
export type HierarchicalWrapper =
  | { id: string; type: 'item'; item: Item; project: Item | null; depth: number }
  | { id: string; type: 'header'; projectId: string; projectTitle: string; project: Item; depth: number };
```

#### 設計原則
- ヘッダー型は `item: Item` を**持たない**（型レベルで取り出しを禁止）
- ヘッダーが必要とする情報は `projectId` / `projectTitle` / `project` で持つ
- React key には `wrapper.id` を使う（`wrapper.item.id` 経由のアクセスは禁止、ヘッダーでは型エラー）
- `'header-${proj.id}'` 形式の `wrapper.id` で React key 衝突回避
- 階層 UI 拡張時はラッパー型に新 variant を追加（例: `'tenant_header'`）

#### 禁止事項
- `virtual-header-` プレフィックスの**生成も使用も禁止**
- 「item.id を加工して別の意味を持たせる」パターン禁止（意味の違う ID は別フィールドで持つ）
- レイヤー境界（API・URL・ナビゲーション）の ID は**常に Item の実ID**

#### 廃止された慣行
過去の R-Earlier では、ヘッダー Wrapper に `item: { ...proj, id: 'virtual-header-${proj.id}' }` という偽装IDを持たせていたが、
- レイヤー境界での剥がし忘れがフロー画面のコア機能崩壊（プロジェクトフィルタ・新規作成・依存関係）を引き起こした
- 12箇所で `.replace('virtual-header-', '')` を直書きする散在コードが生まれた

これらの問題を根本撤廃するため、R-030 で型分離方針を採用。

### ForAI機能・読み上げ機能（R-031）

#### 共通ユーティリティ

| ファイル | 内容 |
|:--|:--|

| `src/features/core/youkan/hooks/useSpeechSynthesis.ts` | React hook（再生中状態・seek・pause/resume） |
| `src/features/core/youkan/logic/forAiExporter.ts` | アイテム配列 → markdown 文字列変換 pure function |
| `src/features/core/youkan/logic/perspectiveLabel.ts` | filterMode + joinedTenants → 立場ラベル文字列 |

#### Web Speech API

- ブラウザ標準 `SpeechSynthesisUtterance` 使用、無料
- 言語: `ja-JP`
- iOS Safari ではユーザー操作起点でしか `speak()` できないため、ボタンクリック起点で発火
- `isSupported()` でフォールバック表示（Chrome / Safari は確実に対応）

#### 設計原則

- ForAI 出力は markdown 1本に集約。新形式は別関数（`forAiExporter.toJson()` 等）で
- Speech 状態は単一 hook（`useSpeechSynthesis`）で管理。Context 化は需要次第
- ボタン配置はヘッダー（PC）+ メニュー or フローティング（スマホ）の2系統
- Someday は ForAI/読み上げ対象外（夢リスト哲学、R-028 と整合）
- フィルタ変更時は Speech を自動 stop（リアクティブ）

#### 再利用する既存資産

| 資産 | 用途 |
|:--|:--|
| `useYoukanViewModel` | アイテム取得（`viewmodels/useYoukanViewModel.ts`） |
| `useFilter` | filterMode 取得（`contexts/FilterContext.tsx`） |
| `useAuth` | joinedTenants 取得（`features/core/auth/providers/AuthProvider.tsx`） |
| `compareFocusItems` | 各グループ内ソート（`logic/sorting.ts`）|
| `buildHierarchicalList` | ForAI のプロジェクト構造生成（`logic/hierarchy.ts`） |

#### メンテナンス則

- `refreshAll()` の追加使用は禁止（R-027 の部分 refresh 方針に従う）
- ForAI 出力は markdown 1本に集約。新形式追加は別関数で
- Speech は単一 hook で状態管理（Context は使わず単純化）
- Someday アイテムは ForAI/読み上げ対象外（夢リスト哲学）
- ボタン配置はヘッダー集約 → スマホはメニュー or フローティングのどちらかに分配（ヘッダーには置かない）
