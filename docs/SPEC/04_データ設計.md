**`capacity_profile` JSON:**

```json
{
  "standardWeeklyPattern": { "1": 480, "2": 480, "3": 480, "4": 480, "5": 480 },
  "exceptions": { "2026-02-09": 240 },
  "defaultCompanyWeeklyPattern": { "1": { "tenant-a": 240 } },
  "dailyCompanyExceptions": { "2026-02-09": { "tenant-a": 0 } }
}
```

- `standardWeeklyPattern`: このテナントへ提供する曜日別キャパシティ。未設定日は `daily_capacity_minutes` を使う

- 会社/チームスコープの量感分母は、`is_core=1` のメンバーについて上記ルールで日別キャパシティを算出し、合計する

### 3.5 item_dependencies テーブル

アイテム間の依存関係（順序）を管理する。フローView用。
`parent_id`（階層）、`project_id`（所属）とは完全に別概念。

| カラム | 型 | 説明 |
|:--|:--|:--|
| `id` | TEXT PK | UUIDv7 |
| `tenant_id` | TEXT | テナント隔離 |
| `source_item_id` | TEXT NOT NULL | 上流（先にやるタスク） |
| `target_item_id` | TEXT NOT NULL | 下流（後にやるタスク） |
| `created_at` | INTEGER | 作成日時 |

**制約:**
- `UNIQUE(source_item_id, target_item_id)` — 同じペアの重複登録を防止
- 循環参照（A→B→A）はアプリケーション層で防止
- `FOREIGN KEY` で items テーブルを参照

**フロー位置情報:**
- `items.meta` JSON に `{ "flow_x": 250, "flow_y": 100 }` として格納
- 新カラムは追加しない（既存の meta カラムを活用）

### 3.6 user_google_oauth テーブル（R-034 Phase 2）

ユーザーごとに Google OAuth のリフレッシュトークンと primary カレンダー情報を保管する。

| カラム | 型 | 説明 |
|:--|:--|:--|
| `user_id` | TEXT PK | users.user_id への外部キー（1 user 1 行） |
| `encrypted_refresh_token` | BLOB NOT NULL | AES-256-GCM 暗号化済みリフレッシュトークン |
| `primary_calendar_email` | TEXT | 連携した Google アカウントのメール（UI 表示用） |
| `primary_calendar_id` | TEXT | primary カレンダーの calendar_id |
| `last_sync_at` | INTEGER | 最終同期時刻（Unix epoch sec） |
| `created_at` | INTEGER NOT NULL | 作成日時 |

```sql
CREATE TABLE user_google_oauth (
  user_id TEXT PRIMARY KEY,
  encrypted_refresh_token BLOB NOT NULL,
  primary_calendar_email TEXT,
  primary_calendar_id TEXT,
  last_sync_at INTEGER,
  created_at INTEGER NOT NULL
);
```

**制約・運用:**
- 連携解除時は行ごと削除（`DELETE /api/google/oauth`）
- 暗号化キーは `.env` の `GOOGLE_OAUTH_ENCRYPTION_KEY`（Phase 2 着手時に保管場所を最終確定）
- アクセストークンは保管しない（リフレッシュトークンから都度発行し、メモリ内のみで使用）

### 3.7 external_events_cache テーブル（R-034 Phase 2）

Google カレンダーから取得したイベントのキャッシュ。TTL 15 分でリフレッシュ。

| カラム | 型 | 説明 |
|:--|:--|:--|
| `id` | TEXT PK | `"google:{event_id}"` 形式の合成キー |
| `user_id` | TEXT NOT NULL | users.user_id への外部キー（CASCADE DELETE） |
| `calendar_id` | TEXT NOT NULL | カレンダー ID（primary は primary_calendar_id と同値） |
| `event_id` | TEXT NOT NULL | Google 側の event id |
| `start_at` | INTEGER NOT NULL | 開始時刻（Unix epoch sec） |
| `end_at` | INTEGER NOT NULL | 終了時刻（Unix epoch sec） |
| `all_day` | INTEGER NOT NULL DEFAULT 0 | 終日フラグ（1=終日） |
| `title` | TEXT | イベントタイトル（**方針変更により保存 OK**。Youkan は晴樹さん専用） |
| `location` | TEXT | 場所（任意。詳細モーダルで表示） |
| `fetched_at` | INTEGER NOT NULL | このキャッシュ行を取得した時刻 |

```sql
CREATE TABLE external_events_cache (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL,
  calendar_id TEXT NOT NULL,
  event_id TEXT NOT NULL,
  start_at INTEGER NOT NULL,
  end_at INTEGER NOT NULL,
  all_day INTEGER NOT NULL DEFAULT 0,
  title TEXT,
  location TEXT,
  fetched_at INTEGER NOT NULL,
  FOREIGN KEY (user_id) REFERENCES users(user_id) ON DELETE CASCADE
);
CREATE INDEX idx_external_events_user_date ON external_events_cache(user_id, start_at);
```

**運用:**
- TTL: `fetched_at` から 15 分経過したら次回 `GET /api/calendar/grid` で再取得
- 手動更新（`POST /api/google/calendar/refresh`）は 60 秒クールダウン
- 連携解除時は `user_id` 一致行を全削除（FK の CASCADE で自動）
- Phase 3 で `calendar_id` ごとのテナント割当を追加（別テーブル `user_calendar_assignment` 予定）

### 3.8 assigned_to の値域整理（ID空間、R-050 Phase1）

`items.assigned_to` は単一カラムに2種類のID空間が混在している。discriminator列を追加するスキーマ変更は行わず、**アプリケーション層の解決ロジックで判別する**方針とする（2026-07-09 kaigi2会議で決定。詳細: `docs/kaigi/2026-07-09-R050テナント型AI中枢設計.md`）。

| 値の形式 | 参照先テーブル | 判定方法 |
|:--|:--|:--|
| `u_` プレフィックス付き文字列 | `users.id` | 文字列先頭が `u_` |
| プレフィックスなしの整数（文字列） | `assignees.id` | 上記に一致しない場合のフォールバック |
| どちらにも一致しない値（孤児データ） | なし | 自動修復しない。「未割当」表示＋ログ出力 |

- 実データ検証（2026-07-09時点）で、会社アイテム（`tenant_id` あり・`assigned_to` 非NULL）60件中58件（97%）が `u_` 形式であることを確認済み。既存の `LEFT JOIN assignees a ON items.assigned_to = a.id` はこの97%で担当者名を`NULL`にする表示バグを持つ。
- 解決方法: `backend/ItemController.php::getAssigneeEmail()` に既に実装済みの `u_` プレフィックス判定パターンを、一覧系クエリ（`TodayController.php` / `ItemController.php` の担当者名解決ロジック）へ横展開する。新規マイグレーションスクリプトは不要。
- 孤児データ（例: `u_697b2bfb28bee`）は自動修復・削除をせず、「未割当」表示とアプリケーションログ出力で可視化する。実数の棚卸し（`SELECT assigned_to, COUNT(*) FROM items WHERE tenant_id IS NOT NULL GROUP BY assigned_to` 相当）はPhase1実装時またはPhase2着手前に別途行う。

**担当候補の型（discriminator語彙）:**

フロント/バックエンドで担当候補の由来（`users` 由来 / `assignees` 由来）を区別する型を以下の名称で確定する。Phase3（AI提案設計）以降でこの語彙を再定義しない。

```ts
type AssigneeKind = 'user' | 'assignee';
```

- `'user'`: `assigned_to` が `u_` プレフィックス形式（`users.id` を参照）
- `'assignee'`: `assigned_to` が整数ID形式（`assignees.id` を参照）

## 4. アイテム状態定義（State Matrix）

### 4.1 基本ステータス（JudgmentStatus）

「進捗」ではなく**「判断（今考えるか・否か）」**で分類する。

| ステータス | 日本語 | 意味 | 表示場所 |
|:--|:--|:--|:--|
| `inbox` | 受信箱 | 未判断。判断待ち | Inbox（左カラム） |
| `focus` | 今日やる | 「今日意識と時間を向ける」と判断した状態 | Stream（中央カラム） |
| `waiting` | 待機中 | 他者待ち、または開始日待ち | Waiting（左カラム） |
| `pending` | 保留 | 「今はやらない」と意図的に判断した状態 | Pending（左カラム） |
| `done` | 完了 | 作業が終了した状態 | Log / Archive |

### 4.2 フラグとの組み合わせ

#### Inbox フェーズ

| ステータス | 組み合わせ | 意味 |
|:--|:--|:--|
| `inbox` | 標準 | 純粋な未処理アイテム |
| `inbox` | `interrupt: true` | 割り込み/緊急。Inbox最上部に強調表示 |

#### Focus フェーズ

| ステータス | 組み合わせ | 意味 |
|:--|:--|:--|
| `focus` | 標準 | 今日のタスクキュー |
| `focus` | `isEngaged: true` | 現在実行中（Active Focus）。FocusCardとして中央で最大化 |
| `focus` | `is_executing: true` | 実行中フラグ（Backend永続化）。isEngagedと同義 |
| `focus` | `isProject: true` | プロジェクト全体のFocus |

#### Waiting フェーズ

| ステータス | 組み合わせ | 意味 |
|:--|:--|:--|
| `waiting` | 標準 | 他者待ち（Delegated） |
| `waiting` | `prep_date`が未来 | 開始日待ち（Scheduled）。カレンダーに表示 |
| `waiting` | `prep_date`が過去/今日 | 開始可能（Ready）。Inbox/Focusへ移動候補 |

#### Pending フェーズ

| ステータス | 組み合わせ | 意味 |
|:--|:--|:--|
| `pending` | 標準 | いつかやる（Someday） |
| `pending` | `due_date`が未来 | 期限付き保留 |

### 4.3 特殊フラグ

| フラグ | 値 | 意味 |
|:--|:--|:--|
| `isProject` | true | プロジェクト化。子タスクを持てる |
| `projectId` | ID | 特定プロジェクトの子タスク |
| `tenantId` | ID | 特定の会社に所属 |
| `domain` | private/business | 公私区分 |

### 4.4 ライフサイクル（状態遷移）

```
inbox → focus      （今日やる）
inbox → waiting    （人に頼む / 開始日待ち）
inbox → pending    （今はやらない）
inbox → done       （もう終わった）

focus → done       （完了）
focus → inbox      （やっぱり保留解除）
focus → waiting    （待ちに変更）
focus → pending    （やっぱり保留）

waiting → inbox    （待ち解除）
waiting → focus    （待ち解除して即実行）
waiting → done     （待っている間に解決）

pending → inbox    （再検討）
```

### 5.2 認証

- JWT Bearer Token認証
- リクエストヘッダー: `Authorization: Bearer <token>`
- クエリパラメータ: `?token=<token>`（フォールバック）

### 5.3 アクセス制御

アイテムの可視性ルール:
1. **個人タスク**（tenant_id IS NULL）: created_by = 自分 OR assigned_to = 自分
2. **会社タスク**: 所属テナントのアイテム AND（created_by = 自分 OR assigned_to = 自分）
3. **プロジェクトフォーカスモード**: 上記に加え、focusしたプロジェクトとその子孫のアイテムも表示
4. **管理者スコープ**（R-050 Phase1・新設）: 所属テナントで `memberships.role IN ('owner', 'admin')` のユーザーは、そのテナント内であれば `assigned_to` を問わずアイテムを閲覧できる。既存の1〜3は変更しない（デフォルト表示は従来通り「自分が創建者 or 自分が担当者」のみ）。管理者が他者の担当分を見る場合は、一覧系エンドポイント（`/items`・`/today` 等）に明示的なクエリパラメータ（例: `assigned_to=<id>&scope=team`。パラメータ名は実装Agent確定）を指定した場合にのみ発動する。新しいロールモデルは作らず、既存の `memberships.role` 列を使う。用途は画面2「担当者別ビュー」（詳細: `docs/kaigi/2026-07-09-R050テナント型AI中枢設計.md`）。

  **実装上の注意（既存コードの不整合）**: `ItemController.php` の `isAdmin` 判定（`update()`/`archive()` 等）は現状 `role === 'admin'` のみを見ており `owner` を含まない。一方 `TenantController.php` は `role !== 'owner' && role !== 'admin'` で両方を管理者として扱っている。管理者スコープの実装時は `TenantController.php` 側のパターン（owner/admin両方を管理者とみなす）に合わせること。

## 補足: 状態・Today・プロジェクト文脈の直交性（2026-07-07）

`items.status = 'focus'` は永続化される状態値であり、Today キューへの所属そのものではない。Today キューは `todayCandidates` / `todayCommits` / `executionItem` として取得・構成される実行文脈である。

`focusedProjectId` / `activeProject` は、データ取得・表示・新規追加時の `project_id` 継承・量感計算の絞り込みに使うプロジェクト文脈である。`status = 'focus'` とは別概念であり、名称上の「focus」に引きずられて同一視してはならない。
