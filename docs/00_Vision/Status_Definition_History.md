# JBWOS ステータス定義書と変遷の歴史

このドキュメントは、JBWOS (Just Before Work OS) における「状態 (Status)」の厳密な定義、その存在理由、および現在の「5 Status Model (Haruki Model)」に至るまでの経緯をまとめたものです。

---

## 1. 現在のステータスモデル (Haruki Model)

2026年1月24日に制定された「Single Source of Truth」に基づく、**5つに固定された**ステータスです。これ以外のステータス（Sub-statusなど）をDBのカラムとして持つことは禁止されています。

### 哲学: "Decision Minimization" (判断の最小化)
- **目的**: ユーザーの「判断疲れ」「正解探し」「罪悪感」を排除する。
- **ルール**: 迷ったら `Inbox`。手を動かせるなら `Ready`。今はやめるなら `Pending`。これ以外の中間状態を作らない。

### 5つのステータス定義

| Status (Internal) | 表示名 | 定義・意味 | ユーザーのアクション / 判定基準 |
| :--- | :--- | :--- | :--- |
| **`inbox`** | インボックス | **「未整理・未判断」**<br>システムに発生した全ての情報の入り口。 | 「0秒で入れる」。中身を見てはいけない。判断してはいけない。とりあえず置く場所。 |
| **`waiting`** | 待ち | **「外部依存待ち」**<br>自分以外の要因で止まっている状態。 | 「あの人の返信待ち」「部品待ち」。自分が今すぐ取り掛かろうとしても物理的に不可能なもの。 |
| **`ready`** | 着手可 | **「実行可能」**<br>やろうと思えば今すぐできる状態。 | 邪魔が入らなければ今この瞬間から着手できるもの。<br>※「今日やる」かどうかは別（それはViewの問題）。 |
| **`pending`** | 保留 | **「今はやらない」**<br>意図的に先送りを決めたもの。 | 「今はやらない」と**判断した**もの。<br>Inboxのままだと「判断保留」だが、Pendingは「やらないという判断」積み。罪悪感不要。 |
| **`done`** | 完了 | **「終了・記録」**<br>役目を終えたもの。 | 完了し、アーカイブされるべきもの。 |

---

## 2. なぜこの定義が必要なのか？ (Why)

### ① 「状態」と「属性/ビュー」の分離
かつては「今日やる (Today)」や「明日やる (Tomorrow)」もステータスとして扱おうとしましたが、これは破綻しました。
- **修正**: 日付や優先度は **Flag (属性)** や **Derived (計算結果)** であり、Statusではありません。
- **例**: 「明日やるタスク」は、状態としては `Ready` です。属性として `due_date: tomorrow` を持つだけです。

### ② "Status Creep" (ステータスの肥大化) の防止
「確認中」「レビュー待ち」「修正中」…とステータスを増やすと、ユーザーは「これは確認中？それとも待ち？」と**分類に脳のリソース**を使い始めます。
JBWOSは作業をするためのOSであり、分類をするためのOSではありません。だから5つに固定し、それ以上の解像度は「横メモ」や「チャット」に任せます。

## 3. 定義の変遷と歴史 (History)

### Phase 1: 初期GTDモデル (2025以前)
一般的なTodoアプリ同様、`Todo`, `In Progress`, `Done` などのステータスが混在。
- **問題点**: 「着手したけど止まっている」ものと「まだ手をつけていない」ものの区別がつかず、リストがカオス化。

### Phase 2: Inbox Overload & "Stock" 概念の登場 (2026/01/18)
「自動生成でInboxに50個のタスクが入ると圧死する」という問題が発覚。
- **提案された解決策**: 「Stock (倉庫/Standby)」という、Inboxの前段階のバッファを作る案。
- **議論**: 「Inboxは判断の場だが、Stockは出番待ちの場」として区別しようとした。
- **結果**: 概念としては正しいが、システム上で `Status: Stock` を作ると、`Inbox` との境界線が曖昧になる懸念が残った。

### Phase 3: Haruki Modelへの収束 (2026/01/24)
PM、Architect、Userの議論の末、**「状態語彙を増やすと正解探しで止まる」**という結論に至る。
- **決定**:
    1.  Statusは5つに固定する。 (`Stock` や `Standby` というStatusは作らない)
    2.  `Standby` (出番待ち) という概念は、**「`Ready` 状態 + 期限/スケジュール」** で表現する。
    3.  `Pending` は「迷い」置き場ではなく、「やらない決断」置き場と再定義。

### Phase 4: 現在の "Standby" の扱い
ユーザーからの質問「InboxからStandbyへはどう動く？」に対する答えは以下の通り確立されました。

*   **Status**: `inbox` -> **`ready`**
*   **Flag/Attribute**: `due_date` (納期) や `estimated_minutes` (見積もり時間) が付与される。
*   **View**: ダッシュボードのInboxからは消えるが、**「Preparation (準備/出番待ち)」** ビュー、またはカレンダー上に現れる。

つまり、**「Standby」とはシステム用語(Status)ではなく、ユーザーに見せる「ビュー(View)」の名前**となりました。

---

## 4. 補足: "Derived" (導出状態)
DBには保存されないが、UI上で表現される状態です。

*   **`today_candidates`**: `Status: Ready` かつ 「今日やる候補（日付合致など）」
*   **`executing`**: `Status: Ready` かつ 「タイマー作動中」
*   **`overdue`**: `Status: Ready` かつ 「期限切れ」

これらは全て、根底にあるStatusは **`Ready`** です。
