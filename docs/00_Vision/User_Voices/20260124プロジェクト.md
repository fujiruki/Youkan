#例
タスクA：納期1/30、マイ期限1/29、目安日数　3日
タスクB：納期1/29、マイ期限1/27、目安日数　1.5日
社員X、Yの1日稼働時間：8時間
主力社員：X,Yの2名
それぞれの日のにX,Yは同様に8時間労働可能だとする。
上記の条件のとき、
タスクAの期間：1/29-3日 なので、　1/27（8時間）、1/28（8時間）、1/29（8時間）
タスクBの期間：1/27-1.5日なので、1/26（4時間）、1/27（8時間）
という負荷量。
この場合のそれぞれの日は
26日の負荷量：4時間
27日の負荷量：16時間（8+8）
28日の負荷量：8時間
29日の負荷量：8時間
となっていて、2人8時間＝16時間がそれぞれの日の最大なので、
26日の負荷率：4/16 = 25%
27日の孵化率：16/16 = 100%

# Status Model (Haruki) - Single Source of Truth

## 目的
- 判断疲れ/正解思考/罪悪感→回避を起こさない
- 割り込み時に「戻し先」で迷わせない
- 納期回答のストレスをInternalに持ち込まない

## 用語
- Status = ユーザーが手で選ぶ棚（保存される）。5つ固定。増やすの禁止。
- Flag = 内部条件（保存される）。UIはバッジまで。
- Derived = 計算で出る見え方（保存禁止）。
- View = 画面/フィルタ。Statusではない。

## 1) User-visible Status（固定5語彙）
| internal_key | UI表示名 | 目的(一言) | ユーザーが迷わない判定基準 |
|---|---|---|---|
| inbox   | インボックス | 未整理の受け皿 | 0秒で入れる場所。判断しない。 |
| waiting | 待ち         | 外部待ち/依存待ち | 自分が今すぐ動けない理由がある |
| ready   | 着手可       | 今やれる作業 | 今すぐ作業できる。Ready上限あり |
| pending | 保留         | 今はやらないが捨てない | “やらない”を選ぶ棚。罪悪感なし。 |
| done    | 完了         | 終了/記録 | 終わったもの。 |

## 2) Flags（Statusに昇格禁止）
| flag_key | 意味 | UI |
|---|---|---|
| has_deadline | 期限あり | バッジ |
| needs_decision | 判断待ち（情報不足） | バッジ |
| has_dependency | 依存あり | バッジ |
| is_interrupt | 割り込み由来 | バッジ |

## 3) Derived（保存禁止：状態にしない）
| derived_name | 算出 | 備考 |
|---|---|---|
| today_candidates | ready から抽出 | today_commitというStatusは禁止 |
| executing | timer_running 等 | executionというStatusは禁止 |
| overdue_like | deadline と稼働枠から算出 | 警告はExternalのみ |

## 4) Transitions（遷移：禁止も書く）
- inbox -> (waiting/ready/pending) は user 操作
- waiting -> ready は「待ち解除」で user/system
- ready -> done は完了で user/system
- done -> ready を許可するかは要件で固定（原則は戻し禁止）

## 5) Harukiルール（最重要）
- Status語彙は増やさない。新機能は Flag/Derived/View で表現する。
- Internal(GDB)には圧を入れない。納期根拠や警告はExternalに隔離。

# 1日に処理できる（したい）時間の総量について

個人事業主なら「ごちゃまぜでいいんだ！」という人もいるし「個人事業主だから労働12時間！でも生活も3時間で睡眠時間を確保したい」というニーズもある。

# 一般プロジェクトと製造業プロジェクト
ユーザーとしてログインしている場合、プロジェクト一覧の画面で、は一般プロジェクトと仕事プロジェクトで分けて表示できている。現在その画面での「新規プロジェクト」ボタンを押して現れるダイアログは製造業向けのような内容だ。現場名などがある。だから、これは製造業プラグインの影響があると、仕事プロジェクトの新規作成時には現在のダイアログが良いだろう。
逆に一般プロジェクトにおいては、現場名などは不要だ。タイトル、一言説明、タグ、色があればいいのではないだろうか。他の項目は、構造としては通常のタスクと同じなのでメモ欄もありますし。
上記のように思っているんだけど、この部分の構造についてどのようなUI/UXとすべきか、このシステム全体の思想に沿うようにしたい。それについてAI会議してください。



# デバッグが複雑な場合について（例えばフロントとバックエンド）
ブラウザでの確認ができるのもすばらしい。ただ、その方法は時間がかかるので、「このあたりに問題があるな」と思ったら、その問題がどのような原因からきているかを関連のコードや設計を論理的に調査し、根本的な解決策を探してほしい。そして、第三者目線で、全体を見渡してその解決策が正しいかを調査してほしい。それでもその解決策がいいなとおもったら実装して、最後にブラウザ確認をしてほしいんだ。

# プロジェクト新規作成モーダル
- [ ] このモーダルには何かプロジェクト作成時に外部プラグインから設定項目を追加できるようなフックのようなインターフェースがあってほしい。ワードプレスみたいな。この機能を活用して製造業プラグインなどが作られるようにしたい。
## 個人アカウントでログインしている場合
- [ ] 個人アカウントで開いた時、プロジェクトの新規作成モーダルには所属している会社リストをドロップダウン、もしくは会社数が四つ以下ならそのモーダル上にボタンとして会社名が表示されている。その会社名を押すと、その会社のプロジェクトとして新規作成できる。それら会社名の一つ目は「個人」である。  
- [ ] このモーダルを開いた元の画面が、プロジェクト一覧画面であれば、一番上の会社がデフォルトで選択されている。プロジェクト一覧の中で会社名の横の新規プロジェクトボタンを押して開いた場合はその会社名が選択されている。

## 会社アカウントでログインしている場合
- [ ] 会社アカウントで開いた時、プロジェクトの新規作成モーダルには所属している会社リスト表示しない。つねにその会社に所属するプロジェクトを作るモードである。
