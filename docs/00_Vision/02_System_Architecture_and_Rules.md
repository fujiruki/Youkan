# 02. システムアーキテクチャとルール (System Architecture & Rules)

## 1. 3層アーキテクチャ (The 3 Layers)
JBWOSは、ユーザーの生活を3つの異なるレイヤーで捉え、それぞれを意識的に分離して管理する。

### Layer 1: Decision (判断層) - The Core
- **役割**: 「今、何をすべきか」「何をしないか」を決める場所。
- **構成**: Inbox, GDB (Global Decision Board), Today (Commit)。
- **ルール**: 時間軸を持たない。「判断済み」か「未判断」かだけが存在する。
- **UI**: Internal View (後述)。

### Layer 2: Execution (実行層) - The Reality
- **役割**: 判断された結果（作業ブロック）を実際に消化する場所。
- **構成**: 現場作業、製作、図面作成などの実作業。
- **ルール**: ここでの進捗（何%終わったか）はDecision層には戻さない。「終わったか、まだか」の二値のみ。

### Layer 3: Life (生活層) - The Foundation
- **役割**: 仕事の前提となる生活（家事、休息、育児）を管理する。
- **構成**: チェックリスト（掃除、洗濯等）。
- **ルール**: Executionと競合しない。「仕事が進まなかった日」でも「生活は進んだ」という事実を残し、自己否定を防ぐためのバッファとして機能する。

## 2. 視点の完全分離 (View Separation)

JBWOSの最も重要なアーキテクチャ上の決定事項は、**「自分が見る画面」と「他人に見せるための情報の源泉」を完全に切り分ける**ことである。

### A. Internal View (自分用)
- **対象**: Global Decision Board (GDB)
- **目的**: 自分の精神衛生を守る。今日の迷いを消す。
- **表示するもの**:
    - Inbox (未判断の山)
    - Ready (今日やる覚悟を決めたもの **Max 2件**)
    - Waiting (他者待ち)
- **絶対に表示しないもの**:
    - 期限へのカウントダウン (例: "あと3日")
    - ガントチャート
    - 顧客への言い訳

### B. External View (説明用)
- **対象**: Project List / Schedule / Gantt
- **目的**: 顧客・元請けに対する説明責任を果たす。
- **表示するもの**:
    - 案件ごとの進捗状況
    - 納期（カレンダー日付）
    - 建具表、見積もり
- **運用ルール**: 
    - 能動的に「モード切り替え」ボタンを押さない限り表示されない。
    - ここでの日付操作はInternal Viewに「判断フック」としてのみ影響を与え、焦燥感（アラート）としては伝播しない。

## 3. 日付と期限の概念 (Dates & Deadlines)

JBWOSにおいて「日付」は単一の意味を持たない。2つの異なる性質として管理される。

### ① 納期 (Deadline) - Solid
- **意味**: 外部との約束。守らなければならない契約上の日付（取付日など）。
- **可視化**: カレンダー上で**「くっきりとした実線・文字」**で表示。
- **扱い**: 動かせない事実（Fact）。

### ② 備え完了目安 (Preparation Target) - Blurry
- **意味**: 自分の安心のために置いた仮のゴール。
- **可視化**: カレンダー上で**「ぼやけた色・面」**として表示。文字は出さない。
- **扱い**: いつでも動かしてよい。過ぎても失敗ではない。
- **重要思想**: これを「納期」と同列に扱って（DB制約など）はならない。

## 4. 技術アーキテクチャ (Tech Stack)

### 基本構成
- **Frontend**: React (Vite) + TypeScript
    - 徹底した **Optimistic UI (楽観的UI)** を採用。ユーザーの操作（ボタン押下）に対し、APIレスポンスを待たずにUIを即時更新し、「サクサク感」と「思考を止めない体験」を提供する。
- **Backend**: PHP 8 (Slim等) + SQLite
    - **Server Authority**: ロジックの正解（Today上限判定、RDD計算、データ整合性）はサーバーが独占する。
    - クライアントは「予測表示」を行うが、サーバーからのエラー時には静かにロールバックする。

### データモデル方針
- **Items テーブル**: 建具(Door)だけでなく、タスク、メモ、生活ログを統合管理する。
- **Projects テーブル**: 案件のコンテナ。
- **SQLite**: ファイルベースDBを採用し、AIによるバックアップ・移行・構造解析を容易にする（`verify_and_start.ps1` での完全性チェックが可能）。
