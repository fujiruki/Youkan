# イベント駆動設計 最終固定
## Decision確定 → Today反映フロー
最終更新: 2026-01-12 (JST)

本ドキュメントは、本システムにおける
**Decision確定からToday画面への反映まで**を、
イベント駆動設計として最終固定するものである。

以後、このフローを破る実装はすべて設計違反とする。

---

## 1. 基本思想（前提）

- 判断と実行は完全に分離する
- Todayは「判断結果の投影先」であり、判断の場ではない
- GDBはTodayの存在を知らない
- TodayはGDBを直接操作しない

連携は **イベントのみ** によって行う。

---

## 2. 登場するイベント一覧

### 2.1 DecisionConfirmed
Decisionが確定（Yes / No / 保留）した瞬間に必ず発火する。

```
event: DecisionConfirmed
payload:
  decisionId
  decisionType (Yes | No | Hold)
  rdd
  timestamp
```

---

### 2.2 DecisionDeferred
Decisionが保留にされた場合に発火。

```
event: DecisionDeferred
payload:
  decisionId
  holdReason
  nextReviewDate
```

---

### 2.3 TodayCommitGenerated
Today側で「今日の判断候補」として採用されたときのみ発火。

```
event: TodayCommitGenerated
payload:
  decisionId
  reason (rdd_reached | manual_override)
```

---

## 3. イベント発火責務の分離

| コンポーネント | 発火してよいイベント |
|---------------|---------------------|
| GDB           | DecisionConfirmed / DecisionDeferred |
| Today Engine  | TodayCommitGenerated |
| UI            | イベント発火禁止 |

UIは **購読のみ**。

---

## 4. フロー定義（時系列）

### ステップ1：GDBで判断が確定
- ユーザーがYes / No / 保留を決定
- GDBは即座に `DecisionConfirmed` を発火
- GDBはそれ以上何もしない

---

### ステップ2：Today Engineが購読
- Today Engineは全DecisionConfirmedを購読
- 以下の条件を評価する

```
if decisionType == Yes
and rdd <= today
then
  候補プールに入れる
```

※この時点では **Todayに表示しない**

---

### ステップ3：朝のToday生成時

- 1日の開始時、Today Engineが候補プールを評価
- 優先度ロジックにより最大2件を選出
- 選ばれたDecisionに対してのみ
  `TodayCommitGenerated` を発火

---

### ステップ4：Today画面への反映

- Today画面は `TodayCommitGenerated` を購読
- Commit枠に表示
- ユーザーは「承認 or 差し替え」のみ可能

---

## 5. 明示的に禁止するフロー

以下は**絶対禁止**。

- GDBがTodayを直接更新する
- Decision確定時に即Todayに表示
- Today画面からDecision状態を変更
- UI操作でイベントを直接発火

---

## 6. Lifeとの非連携ルール

- Lifeはイベントを発火しない
- LifeはDecisionConfirmedに影響しない
- Lifeの状態でTodayCommit選定条件を変えてはならない

---

## 7. 失敗しやすい誤実装パターン

### NG例1
- DecisionConfirmed → 即Commit表示  
→ 今日という時間軸がGDBに侵入

### NG例2
- Commit枠が3件以上  
→ 判断疲労を誘発

### NG例3
- Life未チェック → Commit生成抑制  
→ Lifeブロッカー化（思想違反）

---

## 8. 思想テスト（必須）

以下に1つでもYESなら不合格。

- GDBを操作した結果、Todayの表示が即変わるか？
- TodayからGDBのDecision状態を変えられるか？
- Lifeの状態がCommit選定に影響するか？

---

## 9. 最終固定宣言

> Decision → Today の連携は  
> **イベント購読のみで行う。**

> 状態の直接参照・更新・UI起点操作は  
> **すべて設計違反**とする。
