# 設計索引（Implementation Hand-off） v1.0
最終更新: 2026-01-12 (JST)

このドキュメントは、実装AI/開発者が「どこに何が定義されているか」を即座に引けるようにするための索引です。  
本文は **“決めるべき上位ルール” と “実装に必要な確定事項”** のみを収録し、議論ログは含めません。

---

## 0. システムの一文定義（憲法の核）
本システムは **「やること（TODO）」を管理しない**。  
**「決めるべき瞬間（判断対象）」だけを浮かび上がらせ、決めたら迷わせずに実行へ流す。**  
実行できなかった事実は残すが、失敗には変換しない。

---

## 1. レイヤ構造（全体像）
- **Decision Layer**: 決める（Inbox / GDB / Today）
- **Execution Layer**: 実行する（ExecutionContext / ExecutionBlock）
- **Life Layer**: 生きる（生活維持。判断から除外）

参照:  
- 2章（Decision）  
- 3章（Execution）  
- 4章（Life）  
- 5章（Life×Execution衝突ルール）

## 2. Decision Layer（判断の定義とルール）

### 2.1 定義
- **判断対象（Decision）**: 「今、Yes/No/いつ/どれ を決めないと前に進まない問い」
- **GDB（Global Decision Board）**: **判断対象のみ**が存在できる場所（作業・制作物・進捗は存在不可）
- **Inbox**: **未決断の判断対象**を放り込む場所（分類・優先・今日やる判断は不要）

### 2.2 自動分類（憲法条文として確定）
分類は人がやらない。システムが8割やる。

- **Today – Commit（最低取り掛かりライン）**
  - 条件: `Today >= RDD_x` かつ未解決
  - 原則最大2件
  - ユーザー操作: 「承認」または「1件差し替え」のみ
- **Today – Progress（進めたい）**
  - 条件: プロジェクト紐づき + `Today < RDD_x` など
  - 上限なし（デフォルト折りたたみ）
- **Today – Life（生活維持）**
  - 判断対象ではなく実行対象（チェックのみ、Commit枠消費なし）

- **迷いの排除**
  - 迷ったら Progress
  - Commitへの昇格は RDD 到達 or 明示的昇格のみ

- **朝の操作制限**
  - Yes/No 最大2回 + 入れ替え最大1回を超える設計は憲法違反

---

## 3. Execution Layer（実行の定義とルール）

### 3.1 概念
- **ExecutionContext**: 「今日はこの制作を進めるか？」という *実行への入口（判断の結果として起動）*
- **ExecutionBlock**: 自動生成される *消化単位（原則1日=1ブロック）*  
  - **判断対象ではない**  
  - Todayに直接出さない  
  - 選ばせない

### 3.2 Today 連携（重要）
- Todayに出るのは **制作単位で1件の問い**:  
  - 「◯◯制作を、今日は進めるか？」
- Activeな Context がある場合のみ、次の `pending` ExecutionBlock を **自動で1つ** 提示
- 実行完了: `ExecutionBlock.status = done`（次ブロックは自動で待機）
- 制作完了: 全ブロック done → Context 非アクティブ化 → Project done

---

## 4. Life Layer（生活タスクの扱い）
- 生活（掃除・洗濯・片付け等）は **判断対象ではない**
- Commit枠を消費しない
- 完了チェックのみ（未完了でも罪悪感・遅延を生成しない）
- Lifeは Execution の前提条件

---

## 5. Life × Execution の時間衝突ルール（確定）
- **衝突は失敗ではない**（物理条件）
- Life未完了かつ ExecutionContext Active の場合、Executionは **自動で保留**（何も起きない）
- ExecutionBlock 未実行は「事実」として残すが、遅延・警告に変換しない
- 「忙しかった日」などの判定は **ユーザーに入力させず**、システムが記録

---

## 6. RDD（Decision Deadline）算出ロジック（数式確定）

### 6.1 汎用（全プロジェクト共通）
- `RDD_start = DD − E`
- `RDD_Pi = RDD_start − Li`
- GDB出現条件: `Today >= RDD_x` かつ未解決

### 6.2 建具（ドメイン特化プラグイン）
- `DD = InstallDate（取付日）`
- `RDD_start = InstallDate − E_mfg`
- 材料（材料ごとのリードタイム）:
  - `RDD_material_decision = RDD_start − max(L_material_i)`
  - `RDD_material_order = RDD_material_decision`
- 見積:
  - `RDD_estimate = min(RDD_material_decision, RDD_start − Buffer_estimate)`
- 例外開始（見積未完で開始）:
  - `Decision(type='exception')` を必ず生成し、ログ化する（暗黙禁止）

## 7. DBスキーマ（建具RDD/実装基盤）

### 7.1 方針
- **RDDは保存しない**（算出結果）。DBは「算出に必要な事実」と「人が決めた判断」を保持。

### 7.2 テーブル一覧（最小）
- `tbl_Project`
  - `ProjectID (PK)`, `ProjectType`, `InstallDate`, `Status`
- `tbl_Production`
  - `ProductionID (PK)`, `ProjectID (FK)`, `E_mfg`, `StartApproved`, `StartApprovedAt`, `ExceptionStart`
- `tbl_Material`
  - `MaterialID (PK)`, `Name`, `LeadTime`, `IsStandard`
- `tbl_ProductionMaterial`
  - `ProdMatID (PK)`, `ProductionID`, `MaterialID`, `Required`, `DecisionDone`, `Ordered`, `OrderedAt`
- `tbl_Estimate`
  - `EstimateID (PK)`, `ProjectID`, `Completed`, `CompletedAt`, `Buffer_estimate`
- `tbl_Decision`
  - `DecisionID (PK)`, `ProjectID (FK)`, `Type (enum)`, `RelatedID`, `Resolved`, `ResolvedAt`
- `tbl_ExecutionContext`（論理概念でも可）
  - `ContextID`, `ProductionID`, `Active`, `CreatedAt`
- `tbl_ExecutionBlock`
  - `BlockID`, `ProductionID`, `DayIndex`, `Status`, `ActualDate`

### 7.3 RDD算出（View/Service）
- `v_DecisionRDD`（VIEW or Serviceで算出）
- GDB抽出（例）:
  - `Today >= RDD AND Resolved = false`

## 8. 制作開始Decision確定トリガ（自動処理）

### 8.1 トリガ条件
- `tbl_Production.StartApproved` が `false → true` に変化し、`StartApprovedAt` が埋まった瞬間（冪等）

### 8.2 自動処理（順序固定）
1. `tbl_Project.Status = 'execution'`
2. `E_mfg` 日数分の `tbl_ExecutionBlock` を生成（※日付割当なし）
3. 未解決 Decision は **必ず残す**（見積/材料判断/発注）
4. 見積未完なら `ExceptionStart = true` とし `Decision(type='exception')` を生成
5. `Decision(type='start')` を `Resolved = true` にする

### 8.3 禁止事項
- ExecutionBlock を GDB に出す（禁止）
- 開始時に未解決Decisionを自動完了（禁止）
- 見積未完開始をログ無しで許可（禁止）

## 9. UI/UX 最小仕様（実装指針）
- ユーザーは **放り込む（Inbox）だけ**
- 分類は自動（RDD/タグ/紐づけ）
- 朝の操作は最大 3アクション以内（承認、1差し替えなど）
- Today:
  - Commit（最大2）
  - Progress（折りたたみ）
  - Life（チェックのみ）
- 実行画面:
  - Active Context の次Blockを1つ提示（選択不可）

---

## 10. 実装チェックリスト（受け入れ基準）
- [ ] GDBに「作業」「制作物」「進捗」が直接並ばない
- [ ] Inbox投入時に分類判断が不要
- [ ] RDD到達で判断のみが自動浮上する
- [ ] Commitは最大2で、ユーザーは選ばず確認するだけ
- [ ] ExecutionBlockはTodayに直接並ばない（Context経由のみ）
- [ ] Life未完でExecutionが止まっても、遅延/失敗扱いにしない
- [ ] 見積未完開始は必ず例外Decisionとして残る
- [ ] RDDは保存せず算出（事実のみ保存）

---

## 11. 非目標（意図的にやらない）
- すべてのタスクに優先順位を付ける
- カレンダーに全作業を割り当てる
- できなかった日を“失敗”として可視化する
- 実行単位（Block）をユーザーに選ばせる

---

## 12. 変更禁止（思想のコア）
- 「GDB=判断対象のみ」から外れる変更
- 「分類はシステム責務」から外れる変更
- 「ExecutionBlockは判断対象ではない」から外れる変更
- 「Lifeを判断に戻す」変更

## 付録B: 最小データフロー
1. 判断対象を Inbox に放り込む
2. システムが RDD を算出し、到達した判断のみ GDB へ
3. Today-Commit は最大2（確認のみ）
4. 制作開始Decision 確定 → ExecutionBlock 自動生成
5. Today は制作単位の Context を提示 → 次Blockを1つ実行
6. Lifeが重い日はExecutionが進まなくても「事実のみ」残す
