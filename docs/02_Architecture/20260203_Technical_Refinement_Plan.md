# JBWOS 技術設計改善案：データの完全性と保守性の向上

システムが拡大する中で発生している「不整合のリスク」と「開発の無駄」を解消し、より堅牢なシステムにするための提案です。

---

## 1. プロパティ名の統一：`title` への集約
プロジェクトとアイテムの両方で「表示名」を扱う際、`name` と `title` が混在している現状を **`title`** に統一します。

- **理由**: このシステムは「自分の決定やタスク」を扱うものであり、「タイトル」という言葉の方が「見出し」としての意味合いが強く、UIのパンくずやリスト表示と親和性が高いためです。
- **実装**: DBスキーマおよびAPIレスポンスの `name` フィールドを `title` にリネーム、またはエイリアス（Alias）を設けてフロントエンドでは `title` しか見ないようにします。

## 2. 不整合を防ぐ「リアクティブ・マッピング」
フロントエンドで `projectTitle` などの冗長なデータを持つのをやめ、**「IDのみを持ち、表示時に最新のマスターから引く」** 仕組みを徹底します。

- **スマートな解決策**: 
    - フロントエンドのメモリ上に「プロジェクト情報のキャッシュ（Map）」を持ちます。
    - UIコンポーネントは `item.projectId` を渡すと、キャッシュから自動的に `project.title` を引いてくる「Selector（セレクタ）」を利用します。
- **メリット**: プロジェクト名が変更された瞬間、すべてのアイテムの表示名が即座に同期されます。

## 3. 継承ロジックの「サービス層」への集約
ViewModelごとに手書きしている継承ロジック（プロジェクトに属するなら会社も継承する等）を、リポジトリまたは専用の **`ContextService`** に移動します。

- **実装**: 
    - `ItemCreationContext` という概念を導入。
    - アイテム作成関数は「どのコンテキスト（どのプロジェクト内か）」を引数に取り、そのコンテキストが持つ `tenantId` などの情報を自動的に注入します。
- **メリット**: 新しいUI機能を追加しても、継承ルールを意識せずに「正しいデータ」が作成されるようになります。

## 4. ステータス設計の２層化（ステートマシン）
現在の `status` 文字列一つに頼るのをやめ、**「ライフサイクル」** と **「集中状態」** の２つの軸に分けます。

- **軸1: Lifecycle (進捗)**
    - `Inbox` (未判断) → `Active` (判断済/進行中) → `Done` (完了)
- **軸2: Engagement (熱量/集中)**
    - `Idle` (待機中) → `Focus` (今日やる候補) → `InProcess` (現在実行中)
- **メリット**: 「実行中のアイテムは、同時にActiveでもある」という状態を論理的に矛盾なく表現でき、「実行中のフラグを消したらどこに戻るべきか」という迷いが解消されます。

---

> [!IMPORTANT]
> これらの改善は、単なるコードの整理ではなく「間違いが物理的に起こり得ない構造（Correct by Construction）」への移行を意味します。
