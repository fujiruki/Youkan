# R-056〜R-058: アイテム詳細画面（DecisionDetailModal）パフォーマンス改善 設計書

作成: 2026-06-11（指揮AI: Fable / 実装: sonnet Agent）

## 背景・ユーザー報告

「アイテムの詳細画面においてその中のいくつものUIが動作が遅くて使いづらい」

## 調査結果（コード精査 + chrome-devtools 本番実測）

### 実測値（本番 door-fujita.com、CPUスロットリングなし）

| 計測項目 | 値 |
| :--- | :--- |
| INP（最遅インタラクション） | **331ms**（processing 17ms / **presentation delay 312ms**） |
| モーダル内 DOM ノード数 | **1,531**（ページ全体 2,959） |
| カレンダーセル数 | **161**（grid の最多子要素 162） |
| Layout 更新 | 52ms（283/2052 nodes） |
| モーダル内の 1 編集で発火する API | **POST 1 + GET 6**（items / today / memos / members / user/profile / projects の全量refetch） |

### 真因（3つの独立した課題）

**課題A（最重要）: 編集のたびに全量 refetch が走る**
`DashboardScreen.tsx:488-491` の `onUpdate` が `updateItem()` の直後に毎回 `handleRefresh()`（= `vm.refreshAll()` = 6 API 並列）を呼ぶ。
`useYoukanViewModel.updateItem`（useYoukanViewModel.ts:1102）は既に楽観的更新で VM の 6 リスト（gdbActive/gdbPreparation/gdbIntent/gdbSomeday/todayCandidates/todayCommits）を更新しているため refreshAll は冗長。
結果: 日付クリック・担当者変更・Boost 等のたびに 6 GET → VM 全リスト置換 → `unifiedAllItems` 再生成 → モーダル 1,531 ノード再レンダリング + QuantityEngine 全再計算。
コンソール実測でも `[ApiClient] Updating item` の直後に `[ViewModel] Fetched GDB Shelf` を確認。

**課題B: モーダル内のあらゆる state 変更で 161 セルのカレンダーが再レンダリングされる**
`DecisionDetailModal`（1,112行）はタイトル/メモの毎キーストローク（`setEditedTitle`/`setNote`）、日付入力フォーカス切替（`setActiveDateInput`）等で全体が再レンダリングされる。子の `SideCalendarPanel` → `DetailQuantityCalendar` → `RyokanCalendar` → `RyokanGridView` → `CalendarCell`(×161) は **一切 React.memo されていない**。さらに:
- `SideCalendarPanel` へ渡す `new Date(dueDate)` が毎レンダーで新インスタンス
- `onSelectDate` 等のコールバックがインライン生成（毎レンダー新参照）
- `RyokanCalendar` 内 `renderItemTitle` が毎レンダー再生成
- INP の presentation delay 312ms はこの巨大サブツリーの style/layout 起因（DOMSize insight でも grid が最多子要素として指摘）

**課題C: QuantityEngine のホットループに console.log・冗長計算**
- `QuantityEngine.calculateCapacityForDate`（QuantityEngine.ts:220, 239）に console.log があり、アイテム数 × 最大120日 のループ内で発火しうる
- `calculateVolume` は全アイテムに対し `allocateBackwardsCore`（最大120日 × `calculateCapacityForDate`）を実行。同一 (日付, tenantId) の capacity を何度も再計算しており、キャッシュがない（O(items×120) → O(distinct days×tenants) に削減可能）
- `DecisionDetailModal.allocationDetails` useMemo（DecisionDetailModal.tsx:126-169）の deps に無関係な `activeDateInput` が含まれ、日付欄フォーカス切替だけで再計算。console.log も 2 箇所残存

## 実装方針（3 並列、ファイル競合なし）

### R-056: 詳細モーダル編集時の全量 refetch 廃止

対象ファイル: `DashboardScreen.tsx` ほか DecisionDetailModal マウント箇所

1. `DashboardScreen.tsx:488-491` の `onUpdate` から `handleRefresh()` を削除（`await updateItem(id, updates)` のみ残す）
2. `onClose` / `onDecision` / `onDelete` の `handleRefresh()` は維持（閉じ際 1 回の整合性担保）
3. 他のマウント箇所も同パターンを確認し同様に修正:
   - `FlowScreen.tsx:1144` / `PanoramaBoard.tsx:353` / `SpeechView.tsx:282` / `ProjectRegistryScreen.tsx:387` / `VolumeCalendarScreen.tsx:253`
   - ※各画面の onUpdate 実装が「updateItem 相当 + 全量リフレッシュ」なら同じ削除。楽観的更新が無い画面（独自 state）の場合は壊さないこと
4. TDD: 「モーダル内 onUpdate 時に refreshAll が呼ばれない」「onClose 時に 1 回呼ばれる」テストを先に書く

リスク: サーバー側でしか計算されない派生値の画面反映が閉じるまで遅延する → onClose の refresh で最終整合するため許容。

### R-057: QuantityEngine の計算量削減・ログ除去

対象ファイル: `logic/QuantityEngine.ts`（+ 既存テスト）

1. `calculateCapacityForDate` 内の console.log 2 箇所を削除
2. capacity キャッシュ導入: `calculateMetrics` / `calculateVolume` / `allocateBackwardsCore` の 1 回の呼び出しチェーン内で `Map<string, number>`（キー: `${dateKey}|${tenantId ?? ''}`）を共有。グローバル static キャッシュは capacityConfig 変更で破綻するため**禁止**。呼び出しスコープで生成し引数で受け渡すこと
3. TDD: 「キャッシュ有無で計算結果が完全一致する」テストを先に追加。既存テスト green 維持

### R-058: 詳細モーダルの再レンダリング隔離・初期描画軽量化

対象ファイル: `DecisionDetailModal.tsx` / `SideCalendarPanel.tsx` / `DetailQuantityCalendar.tsx` / `RyokanCalendar.tsx` / `RyokanGridView.tsx` / `CalendarCell.tsx`

1. `DecisionDetailModal`:
   - console.log 2 箇所削除（128行・136行付近）
   - `allocationDetails` useMemo の deps から `activeDateInput` を除去
   - `SideCalendarPanel` へ渡す `new Date(dueDate)` / `new Date(prepDate)` を useMemo 化
   - `onSelectDate` ハンドラを useCallback 化
   - useMemo 内の `JSON.parse(localStorage.getItem(...))`（160-161行）を useMemo 外で 1 回だけ評価
2. `SideCalendarPanel` / `DetailQuantityCalendar`: `React.memo` でラップ
3. `RyokanCalendar`: `renderItemTitle` を useCallback 化。子ビューへ渡すインラインコールバック（`onExternalEventClick` 等）を useCallback 化
4. `RyokanGridView`: `CalendarCell` へ渡すコールバックを安定化
5. `CalendarCell`: `React.memo` でラップ（props は既に概ねプリミティブ/安定参照。`projects` 配列と `externalEvents` 配列は親側で参照安定を確認）
6. mini レイアウト（詳細モーダル内）時の初期 range を ±2ヶ月 → **±1ヶ月**（約 90 セル）に縮小。`RyokanCalendar.tsx:274-313` の range 初期化で `isMini` を参照。スクロールでの自動拡張（handleScroll）は既存挙動のまま
7. TDD: 「タイトル input への入力で CalendarCell が再レンダリングされない」ことをレンダーカウントで検証するテストを先に書く
