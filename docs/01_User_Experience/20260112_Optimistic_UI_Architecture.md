# アーキテクチャ決定記録: Server Authority & Optimistic UI
作成日: 2026-01-12
同意者: ユーザー、開発AI

## 背景
「バックエンドに知能（ロジック）を持たせる」という方針に対し、
「ブラウザを単なるビューアにすると、通信タイムラグで操作感が損なわれるのではないか」
という懸念が提起された。

「30秒で終わる体験」を実現するには、軽快なレスポンスが不可欠である。

## 決定事項: ハイブリッド戦略 (Optimistic UI)

**「ロジックの正解はサーバーが持つ。体験の速度はクライアントが作る。」**

### 1. サーバーの責務 (The Truth)
*   **判定権限の独占**:
    *   Todayへのコミット上限（最大2件）の判定。
    *   RDD（判断すべき日）の計算。
    *   「完了」の事実の確定・永続化。
*   これらはブラウザ側で勝手に判断してはならず、必ずAPI応答をもって正とする。

### 2. クライアントの責務 (The Speed)
*   **楽観的更新 (Optimistic Update)**:
    *   ボタンを押した瞬間、APIの返答を待たずにUI上のステータスを変更する。
    *   （例：Confirmを押したら即座にチェックが付き、カードが移動する）
*   **事後整合**:
    *   裏でAPIリクエストを送信する。
    *   万が一サーバーからエラー（例：「既に2件埋まっています」）が返ってきた場合、UIをロールバックし、ユーザーに静かに伝える。
    *   ※ただし、通常操作でエラーが出ないよう、UI側でも「ボタン非活性化」などの最低限のガードは行う（ロジックではなくガードレールとして）。

## 結論
Phase 2（Frontend Dumb-ification）の実装においては、
**「ロジックをJSから消す」のではなく、**
**「JSのロジックを『判定』から『予測表示』に格下げする」** 方針で進める。

これにより、データの整合性と操作のサクサク感を両立させる。
