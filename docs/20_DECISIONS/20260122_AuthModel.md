# AI会議_権限モデル設計.md

## 0. 会議概要
**テーマ**: JBWOS (Joinery Basic Work OS) におけるデータ権限とプライバシーモデルの再定義
**ゴール**: 「個人のプライバシー」と「会社の管理・生産性」を両立する実装仕様の確定
**トリガー**: ユーザーからの「全データが見えてしまう」というセキュリティ懸念の提示

## 1. 前提・制約（User Request）
1.  **個人タスク/プロジェクト**: 自分だけが見れる（基本）。
2.  **マネージャー権限**:
    - 社員の「担当タスク・プロジェクト」 → 詳細が見れる。
    - 社員の「個人的タスク」 → **量感（カレンダーの埋まり具合）のみ**見れる。詳細は見れない。
3.  **製造業プロジェクト**:
    - 会社（テナント）に属する。
    - 社員全員が見れる（透明性）。
    - 担当者割り当てが可能。
4.  **退職者対応**:
    - 履歴は残るが、アクセス権は剥奪。連絡不可等の表示制御。

## 2. キャスティング
1.  **Arch (技術・実装)**: DBスキーマ、パフォーマンス、クエリ複雑性担当
2.  **Biz (ビジネス・管理)**: 組織管理、生産性可視化、マネージャーの利便性担当
3.  **Priv (UX・プライバシー)**: ユーザー心理、入力の心理的安全、使い分けの明確さ担当
4.  **Legal (リスク・法務)**: 労務管理、情報漏洩、データ所有権担当

---

## 3. ラウンド1：初期提案 (Divergence)

### 巡目1: データ所有とスコープの基本定義

-   **Priv**: 「個人タスク」と「仕事タスク」の境界線がUI上で曖昧だと、ユーザーは恐怖で何も書けなくなる。入力時に明確に「これは会社に見える」「これは見えない」がわかるスイッチか、デフォルト設定が必要だ。個人的には **「Life」ログはデフォルトPrivate**、**「Dev/Work」ログはデフォルトPublic(Company)** という自動振り分けを提案する。

-   **Arch**: 実装観点では、`items` テーブルに `visibility` カラム（`private`, `public`, `limited`）を追加し、さらに `tenant_id` と `owner_id` でフィルタリングするのが定石。ただし「量感だけ見える」機能はクセ者だ。SQLで `COUNT` や `start/end` だけ取得する専用APIが必要になる。既存の `getAll` を使い回すと事故る。

-   **Biz**: マネージャーとしては「隠れ残業」や「過重労働」を把握したい。個人的なタスクの中身は見たくないが、「何時から何時まで埋まっているか」は重要。Googleカレンダーの「予定あり」表示のようなイメージで、**Masked View** を標準機能として提供すべき。これは「監視」ではなく「守る」ための機能として打ち出す。

-   **Legal**: データ所有権の問題がある。「個人の日記」を会社のサーバーに置く場合、会社がそのデータを閲覧権限を持つ法的な正当性が必要。ユーザーが「個人的なタスク」として入力したものが、システム管理者（DB直接閲覧）に見えるリスクも含め、規約上の明記が必要。実装としては、個人タスクは暗号化保存まで検討すべきか？ いや、そこまではオーバーキルか。

### 巡目2: 製造業プロジェクトの特殊性

-   **Arch**: 製造業プロジェクトは `projects` テーブルに `type='manufacturing'` を持たせ、このタイプの場合は強制的に `tenant_id` に紐づく全メンバーに `READ` 権限を与えるロジックが良い。既存のプロジェクトは `membersOnly` などのフラグで制御するハイブリッド構成になる。

-   **Biz**: 製造現場は「誰が何を持っているか」が可視化されていないとラインが止まる。ここはフルオープンが正解。ただし、**「担当者割り当て」は排他制御**に近い意味を持つ場合がある。「俺がやってるから手出すな」的な。権限と同時に「ステータスロック」も必要かもしれない。

-   **Priv**: 会社閲覧可能なプロジェクトでも、「自分用メモ」を付けたいニーズはある。「この建具の納まり、自信ないから後で調べる」みたいな。パブリックなプロジェクトの中に、**プライベートなサイドメモ (SideMemo)** をアタッチできる構造は維持してほしい。

-   **Legal**: 退職者の扱いについて。退職後も「誰が作ったか」の情報は製造物責任（PL法）や品質保証の観点で消してはならない。`users` テーブルから論理削除しても、`items` の `created_by` や `assigned_to` のIDは保持し、表示時に「退職済みユーザー」としてマスク表示または静的表示に切り替える実装が必要。

### 巡目3: マネージャーの「量感」閲覧の実装

-   **Arch**: 「量感だけ」の実装詳細だが、APIを分けるべきだ。`/api/schedule/details`（権限ありのみ）と `/api/schedule/capacity`（全期間のブロック情報のみ、タイトルなし）に分離する。フロントエンドで隠すのはセキュリティ上NG。必ずバックエンドでフィルタリングする。

-   **Biz**: 評価制度との連動も視野に入れたい。個人タスクが多くて仕事が進まないのか、単にキャパオーバーなのか。マネージャービューでは、**「Lifeブロック」と「Workブロック」の色分け** だけは見えてほしい。「遊んでるな」ではなく「休んでるな」を確認したい。

-   **Priv**: 色分けが見えるだけでも「こいつ夜中にLifeタスク入れてるな」とか推測されるのが嫌なユーザーもいる。**「Private」一色**で統一するオプションが欲しい。「量感」とは「空き時間があるかないか」だけで十分なはずだ。

-   **Legal**: 同意。詳細な種別まで見えると「プライバシーの侵害」における精神的苦痛の主張を許す隙になる。労務管理上必要なのは「労働可能時間」の把握のみ。

### 巡目4: デフォルト動作とUI

-   **Arch**: DBマイグレーションで既存データをどうするか。現在はすべて「Public」状態に近い。移行スクリプトで、既存の全てのLog/Itemを一旦どう扱うか？ 安全側に倒すなら、**全て「Private」に倒して、ユーザーに公開設定し直してもらう**のが事故がない。

-   **Biz**: いや、それだと業務が止まる。「本日やる」リストが真っ白になったら現場が大混乱だ。**「プロジェクトに紐付いているものはPublic（Project Scope）」**、**「InboxにあるものはPrivate」** という移行ロジックが妥当ではないか。

-   **Priv**: 新規作成時のUIについて。プロジェクト未選択のタスクを作成するとき、デフォルトは「Private」であるべき。そして「プロジェクトに紐付けた瞬間」に「このプロジェクトのメンバーに公開されます」というトーストかアイコン変化でフィードバックを与えるべき。

-   **Legal**: 同意。「意図せず公開されていた」が一番のリスク。UIでの**「公開範囲アイコン（鍵マーク vs 地球儀マーク）」**の常時表示を必須要件としたい。

### 巡目5: 認証とテナント分離の徹底

-   **Arch**: そもそも現状の `ItemController` が `SELECT * FROM items` しているのが論外。全てのクエリに `WHERE tenant_id = ?` は最低限必須。その上で `AND (visibility = 'public' OR user_id = ?)` を付与するBaseクラスの共通メソッドを作るべき。これ漏れが一番怖い。

-   **Biz**: SaaSとして売るならマルチテナントのデータ分離は生命線。バグで他社の情報が見えたら即サービス終了だ。**RLS (Row Level Security)** のような機構をアプリ層で確実に実装してくれ。

-   **Priv**: ログアウト忘れ対策も。共用PCで使う現場も多い。一定時間操作がない場合の自動ロックや、画面全体を隠すスクリーンセーバー機能もUXとして欲しい。

-   **Legal**: アクセスログの保存。誰が誰の「量感」を見たか、誰がプロジェクトの詳細を見たか。マネージャー権限の濫用を防ぐため、**「マネージャーが詳細ログにアクセスしました」**という通知はやりすぎか？ いや、アクセスログは監査用にバックエンドに残すだけでいいだろう。

## 4. ラウンド2：交差レビュー (Debate)

**Arch**: Bizの言う「RLSをアプリ層で」は賛成だが、SQLiteでやるなら `BaseModel` でクエリスコープを強制する仕組みが必要。PHP側で `Scope` クラスを作って、`Item::withScope($user)->get()` のように書かないと取れないようにしよう。生のSQLをControllerに書くのは禁止にする。

**Biz**: Privの「プロジェクト紐付けで公開」は直感的だが、例外がある。「公開プロジェクト内の自分だけタスク」はあり得るか？
→ **Priv**: あり得るが、仕様が複雑になる。「プロジェクト＝共有の場」と割り切ったほうがユーザーのメンタルモデルは単純になる。「共有したくないならプロジェクトに入れるな（または自分専用プロジェクトを作れ）」というルールが良い。

**Legal**: Archの「既存データ移行」について、Biz案（プロジェクト紐付け=Public、Inbox=Private）は合理的だが、**「プロジェクトに紐付いていないが仕事のタスク」**がInboxにある場合、同僚から見えなくなって困る可能性がある。
→ **Arch**: それは「未割当」状態。やはりInboxはデフォルトPrivateで良い。見せたければ「共有Inbox」プロジェクトに入れるべき。**「Inboxは個人の脳内」**というメタファーを徹底しよう。

**Priv**: 「量感」についてLegalの指摘通り「Private一色」で行こう。Bizの「Life/Workの色分け」は諦めてくれ。ユーザーの信頼が第一だ。
→ **Biz**: 了解。では「Busy」という単一ステータスで表示する。

## 5. ラウンド3：統合・収束 (Convergence)

### ファシリテーターまとめ

1.  **データ構造**:
    -   `visibility` カラムを追加せずとも、以下のロジックで制御可能か？
        -   **Project未割当 (Inbox)**: Private (Owner only)
        -   **Project割当 (Attached)**: Projectの権限に従う (Public/Shared)
    -   いや、万全を期すために `visibility` (`private`, `public_project`, `public_tenant`) は持たせておこう。将来的に「プロジェクト外の共有タスク」も対応できるように。

2.  **マネージャー権限**:
    -   APIレベルで分離。
    -   `/api/schedule/capacity`: 時間枠のみ、タイトル秘匿。
    -   詳細閲覧は、対象ユーザーが「明示的に共有設定したプロジェクト」内のタスクのみ可能。

3.  **製造業プロジェクト**:
    -   `projects` テーブルに `context_type` ('personal', 'business_unit', 'manufacturing') を追加。
    -   `manufacturing`: Automatic Tenant Public.

4.  **移行計画**:
    -   `project_id IS NOT NULL` → Public
    -   `project_id IS NULL` → Private
    -   これで既存データの整合性を保ちつつ、セキュリティを確保する。

---

## 6. 最終結論 (Decision)

### 採用する仕様

1.  **アイテム(Log/Task)の権限ルール**:
    -   **Private (デフォルト)**: `project_id` が NULL のアイテム。作成者のみ閲覧・編集可。
    -   **Project Shared**: `project_id` を持つアイテム。そのプロジェクトの閲覧権限を持つ全員が閲覧可。
    -   **編集権限**: プロジェクト内タスクでも、編集は「作成者」または「Assignee」または「Admin」に限定（行儀の悪い上書き防止）。

2.  **プロジェクトの権限ルール**:
    -   **Standard Project**: 作成者がメンバーを指定（現状は全公開だが、将来的にMember管理を追加）。当面は「テナント内全公開」で運用し、運用でカバー。
    -   **Manufacturing Project**: テナント内全公開（強制）。

3.  **マネージャー「量感」ビュー**:
    -   専用エンドポイント `/api/users/{id}/capacity` を実装。
    -   返却データ: `{ start: timestamp, end: timestamp, is_busy: true }` の配列のみ。タイトル・内容は一切含まない。

4.  **実装アクション**:
    -   `ItemController` の `getAll` を廃止し、`getMyItems` と `getProjectItems` に分離。
    -   `ProjectController` に `type` (manufacturing等) を追加。
    -   全SQLに `WHERE tenant_id = ?` を徹底。

### 次のアクション
1.  **Backend**: `ProjectController`, `ItemController` のクエリ全面書き換え。RLS的なスコープ適用の徹底。
2.  **Migration**: `is_public` フラグの検討（今回は Project 紐付け判定で代用するためスキーマ変更なしか、念のため `scope` カラム足すか。→ **今回はロジックでカバーし、スキーマ変更は最小限にする**）。
3.  **Frontend**: 自分がアサインされていない/所有していないPrivateタスクは表示されないことを確認。

会議終了。
