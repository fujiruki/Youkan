# UI/UX 改善提案ミーティング議事録 (2026/01/12)

## 課題
ユーザーからのフィードバックにより、以下の課題が明確化されました。
1. **用語の難解さ**: 「RDD」「判断ボード」「GDB」などの専門用語（または開発者用語）が、直感的な理解を妨げている。
2. **誘導の不足**: ユーザーが次に何をすべきか、どこを見ればいいかが直感的に分からない。「迷わせない」誘導が必要。

## 改善方針
**「判断を終わらせる」というコア価値を損なさずに、誰でも直感的に使えるUIへ刷新する。**
専門用語を排除し、行動ベースのラベリングに変更します。

## 具体的な改善案

### 1. カラム名の刷新 (Renaming)
開発者目線の用語から、ユーザーの行動に基づいた用語へ変更します。

| 現行 (Current) | **変更案 (Proposed)** | 意図・説明 |
| :--- | :--- | :--- |
| **Inbox (放り込み箱)** | **受信箱 (Inbox)** | そのままですが、より一般的になじみのある言葉に。 |
| **Scheduled (予定)** | **カレンダー (Schedule)** | 「予定」よりも「カレンダー連携」であることを強調。 |
| **Waiting (待ち)** | **連絡待ち (Waiting)** | 単なる「待ち」ではなく、「誰かの反応を待っている」ことを明確に。 |
| **Ready (今日やる)** | **今日やる (Today)** | 「準備完了(Ready)」ではなく、「今日実行する」という意思決定の結果であることを強調。<br/>サブタイトル: **「最大2つまで」** |
| **Execution (実行)** | **実行中 (Doing)** | 今まさにこの瞬間に手を動かしているもの。 |
| **Done (記録)** | **完了ログ (Log)** | 終わったことの記録。実績。 |

### 2. ヘッダーデザインの改良 (Visual Hierarchy)
各カラムのヘッダーに、アイコンと簡単な説明文（マイクロコピー）を常時表示し、その列が何のための場所かを常に伝えます。

- **受信箱**: 📥 *「頭の中を空にする場所」*
- **カレンダー**: 📅 *「時間が決まっている約束」*
- **連絡待ち**: ⏸️ *「返信・承認待ち」*
- **今日やる**: ▶️ *「絶対に終わらせる (MAX 2)」* - **最重要強調**
- **実行中**: 🔥 *「今、この瞬間やること」*
- **完了ログ**: ✅ *「今日の成果」*

### 3. 空の状態（Empty State）の活用
アイテムがない時の表示を、「データがありません」ではなく、「次に何をすべきか」のガイドにします。

- **受信箱が空**: "素晴らしい！頭の中は空っぽです。何か思いついたら即座に入力しましょう。"
- **今日やるが空**: "まずは「受信箱」から、今日絶対にやることを1つ選んでドロップしてください。"
- **実行中が空**: "さあ、「今日やる」から1つ選んで開始しましょう。"

### 4. 操作誘導（チュートリアル）
- **初回起動時**: `FirstExperienceModal` (既存) で概念を伝える。
- **常時**: 画面右上に「？」ボタン（実装済み）に加え、各セクションにホバー時に表示される「？」チップを追加検討。
- **ドラッグ操作**: ドラッグ開始時に、ドロップ可能なエリアをハイライトし、「ここへ移動」等のラベルを表示する。

## 次のアクション (Implementation Plan)
1. **カラムヘッダーのUIコンポーネント改修**: タイトル、アイコン、説明文を含むリッチなヘッダーに変更。
2. **文言の修正**: `i18n/labels.ts` を修正し、全体的な用語を上記提案に合わせて統一。
3. **Empty Stateの実装**: 各カラムの `Placeholder` 表示をリッチにする。
