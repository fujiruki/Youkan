# アイテム統合データモデル設計

> [!NOTE]
> ✅ **実装完了** (2026-01-28): v21 マイグレーション完了。`projects` テーブル廃止済み。

## 概要

タスクとプロジェクトを **単一の `items` テーブル** で管理する設計方針。

## 核心的な考え方

- **アイテム (Item)** = 「意識したいこと」を表す最小単位
- アイテムは「タスク」と「プロジェクト」という **状態** を持てる（`is_project` フラグ）
- プロジェクトは「細分化が必要なアイテム」または「後で細分化するつもりのアイテム」

## データ構造

```
┌───────────────────────────────────────────────────────────────┐
│                       items テーブル                           │
│  - id, title, tenant_id, created_by                          │
│  - parent_id (親アイテムへの参照)                              │
│  - is_project (プロジェクトフラグ)                             │
│  - assignee_id (担当者)                                       │
│  - due_date, my_deadline, estimated_hours                    │
│  - plugin_data (JSON: プラグイン拡張用)                        │
└───────────────────────────────────────────────────────────────┘
              │
              │ is_project = true かつ製造業プラグイン有効時
              ▼
┌───────────────────────────────────────────────────────────────┐
│           manufacturing_projects テーブル                      │
│  - item_id (FK → items.id)                                    │
│  - client_name, gross_profit_target, etc.                    │
└───────────────────────────────────────────────────────────────┘
```

## 設計決定事項

| 項目 | 決定 |
|:---|:---|
| テーブル構造 | `items` に統合、`projects` テーブルは廃止 |
| プロジェクトの定義 | `is_project = true` のアイテム |
| 親子関係 | `parent_id` で表現。深さは未定（専門家会議で決定） |
| ダッシュボード表示 | プロジェクトも子タスクも表示する（表示方法は専門家会議で決定） |
| プラグイン拡張 | 別テーブル（例: `manufacturing_projects`）で紐づけ |

## 未決定事項（専門家会議で議論）

1. **親子関係の深さ**: 無制限 or 2階層まで など
2. **ダッシュボードでの子タスク表示方法**: インデント、グループ化、展開UI など

## 保留事項（アカウント種別）

- ログイン画面でユーザー/会社のタブ分け
- 会社アカウントと個人アカウントの明確な UI 分離
