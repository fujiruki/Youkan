# Judgment OS 思想違反防止チェックリスト
（Global Decision Board 実装用）

この文書は、本システムが **「判断を終わらせる道具」** であり続けるための
最低限かつ必須のガードレールを定義するものである。

実装・UI設計・仕様追加の際は、必ず本チェックリストを通過させること。

---

## 0. このチェックリストの位置づけ

- これは **UXガイド** ではない  
- これは **機能要件** でもない  
- これは **思想（Constitution）を破らないための安全装置** である  

一項目でも違反した場合、
その機能・UI・挙動は **実装不可** とする。

---

## 1. Global Decision Board 全体チェック

### G-1. 起動時に「考えさせない」か？
- [ ] アプリ起動直後に **プロジェクト一覧が表示されない**
- [ ] 起動直後に **期限・日付・ガントが一切見えない**
- [ ] 起動直後に表示されるのは **判断結果（Inbox / Waiting / Ready / Pending）のみ**

❌ NG例  
- 「どのプロジェクトを開きますか？」  
- 「今日のスケジュールはこちらです」

---

### G-2. 今日やることは“一覧から選ばせて”いないか？
- [ ] ユーザーは **一覧を比較して今日の作業を決める必要がない**
- [ ] 「今日はこれをやる」が **既に決まった状態**で表示される
- [ ] Ready に1件でも入ったら、それ以外を探させない

❌ NG例  
- Inbox を眺めて「どれにしようかな」と考えさせる構造

---

## 2. Inbox（未判断）チェック

### I-1. Inboxは「比較の場」になっていないか？
- [ ] Inbox 内で **並び替え・優先度付けをしない**
- [ ] 重さ・時期感は **小さく表示され、判断材料に使えない**
- [ ] Inbox を見ても「どれが重要か」が分からない設計になっている

❌ NG例  
- 期限順ソート  
- 重い順表示  
- 色で重要度が分かるUI

---

### I-2. Inboxの件数が心理的負荷にならないか？
- [ ] 10件を超えた場合、警告ではなく**肯定メッセージ**が出る
- [ ] 表示メッセージは以下の思想に合致している

**必須メッセージ例**
>「今日は、Inboxから1件出せば十分です」

❌ NG例  
- 「Inboxが溜まっています」  
- 「未処理タスクが○件あります」

---

## 3. Ready（着手可）チェック【最重要】

### R-1. Readyは「作業中リスト」になっていないか？
- [ ] Ready の意味が UI 上で明確に定義されている
- [ ] 定義は以下と一致している

**Ready 定義（必須）**
>「今日は、これと向き合うと決めたもの」

❌ NG例  
- 「今すぐできるタスク」  
- 「進行中タスク」

---

### R-2. Ready件数制限は絶対か？
- [ ] Ready は **全プロジェクト合計で最大2件**
- [ ] 3件目を入れようとすると **必ずブロックされる**
- [ ] ブロック時、叱責や警告は一切出ない

**必須メッセージ例**
>「今日は、これ以上決めなくて大丈夫です」

❌ NG例  
- 設定で上限変更可能  
- 上限超過を許可する例外処理

---

### R-3. Readyに入れた瞬間、判断終了が宣言されるか？
- [ ] Ready に入れた瞬間、以下のメッセージが表示される

**必須メッセージ**
>「今日は、これで十分です」

- [ ] Inbox / Waiting は視覚的に弱くなる（薄くなる・折りたたまれる）
- [ ] 次の判断を促すUIが存在しない

---

### R-4. Readyが空になったとき「完全停止」を肯定しているか？
- [ ] Ready が空になった瞬間、中央にメッセージが出る

**必須メッセージ**
>「今日はもう、やるものはありません」

- [ ] Inboxへ誘導しない
- [ ] 次のタスク提案をしない

❌ NG例  
- 「次にやることを選びましょう」

---

## 4. Waiting / Pending チェック

### W-1. Waitingは「自責」を生まないか？
- [ ] Waiting の理由が **明示される**
- [ ] 理由は「人・材料・承認」など外部要因である

❌ NG例  
- 「自分の準備不足」的な表現

---

### P-1. Pendingは視界ノイズになっていないか？
- [ ] Pending はデフォルトで折りたたまれている
- [ ] 件数のみ表示され、内容は見えない
- [ ] 自動で Ready / Inbox に戻らない

---

## 5. External View（対外説明モード）チェック

### E-1. Internal View に日付を逆流させていないか？
- [ ] External View の日付・ガントは Internal View に一切影響しない
- [ ] judgmentStatus は External View では無視される

---

### E-2. 説明文は「誠意ある適当さ」になっているか？
- [ ] 正確な日付を約束しない
- [ ] 謝罪テンプレになっていない
- [ ] 状態説明のみを行う

❌ NG例  
- 「必ず◯日までに」  
- 「遅れており申し訳ありません」

---

## 6. 最終判定ルール

- このチェックリストは **常に最優先**
- 新機能・改善案・要望が出た場合：

**必ず問うこと**
>「これは判断を終わらせるか？  
>それとも、判断を増やすか？」

後者なら **即却下**。

---

## 結語

このツールは、
**効率化のための道具ではない。**
**管理のための道具でもない。**

これは、  
**「もう考えなくていい」と言ってくれる装置**である。

この思想を壊さない限り、
システムは正しい。
