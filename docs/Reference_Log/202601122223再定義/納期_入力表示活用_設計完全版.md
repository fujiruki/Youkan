# 納期（取付日）入力・表示・活用 設計完全版.md
最終更新: 2026-01-12

本ドキュメントは、JBWOSにおける **納期（取付日）** を、
現実に即しつつ思想を壊さず扱うための
**思想・UI・ワイヤー・画面遷移・活用ロジック**を完全にまとめたものである。

本書は実装AIにそのまま渡すことを前提とする。

---

## 1. 納期の思想（最重要）

### 1.1 納期の定義
- 納期は **判断材料となる事実**
- 行動を縛るための装置ではない
- 焦り・恐怖・罪悪感を発生させてはならない

### 1.2 絶対にやらないこと
- 残日数カウント表示
- 赤字・警告色・期限切れ表現
- Today / GDB 一覧への露出
- 納期＝優先度の自動決定

---

## 2. データ仕様

### 2.1 フィールド定義
- due_date : Date | null
- due_status : enum
  - confirmed（日付確定）
  - waiting_external（相手都合で未確定）

### 2.2 必須性
- 入力必須ではない
- 未入力・未確定は正常状態

---

## 3. 納期入力UI設計

### 3.1 入力可能な画面
- **判断詳細ビューのみ**

他画面での入力は禁止。

### 3.2 表示例

#### 確定時
```
【判断材料】
取付日：2026-02-15  [編集]
```

#### 未確定時
```
【判断材料】
取付日：未確定（相手都合） [変更]
```

### 3.3 入力操作
- 日付ピッカー or 直接入力
- クリア（未入力へ戻す）可能
- 入力時の警告・確認ダイアログなし

---

## 4. ワイヤーフレーム（固定）

```
┌ 判断詳細ビュー ─────────────────────┐
│ 判断タイトル                          │
├───────────────────────────────────┤
│ 【判断材料】                         │
│  取付日：2026-02-15 [編集]           │
│  想定製作日数：5日                   │
│  材料リードタイム：3日               │
├───────────────────────────────────┤
│ 【判断】  [Yes]  [Hold]  [No]        │
└───────────────────────────────────┘
```

---

## 5. 画面遷移

### 5.1 基本遷移
- GDB → 判断詳細ビュー
- 納期編集 → 同画面内完結
- 編集による画面遷移なし

### 5.2 Today / History との関係
- Today：表示しない
- History：判断確定時のみ結果を記録

---

## 6. イベントと副作用

### 6.1 DueDateUpdated
**発火条件**
- 納期入力・変更・クリア時

**副作用**
1. due_date / due_status 更新
2. RDD 再計算（内部）
3. Today / GDB 表示は即時変化しない

---

## 7. RDDとの関係

### 7.1 confirmed の場合
- RDD = due_date −（製作日数＋材料LT）

### 7.2 waiting_external の場合
- RDD = null
- 自動判断発生なし

---

## 8. 納期が近い場合の扱い

- 自動警告なし
- 強制Today表示なし
- GDBに「判断」としてのみ出現可能

```
◯◯案件｜納期対応可否を判断する
```

---

## 9. 現実対応テンプレ

### 9.1 相手都合で未確定
- due_status = waiting_external
- 明示的状態として表示
- 仮日入力を防止

### 9.2 長期未確定
- 21日経過で再判断候補としてGDB再出現
- 通知・催促なし

---

## 10. 思想テスト（納期編）

以下が一つでも成立したら設計違反。

- 納期がプレッシャーとして感じられる
- 入力しないと不安になる
- Todayで追われている感覚が出る
- 「遅れている」という言葉が出る

---

## 11. 最終固定宣言

> 納期は  
> 行動を急かすためにあるのではない。  
>  
> **判断を誠実に行うための、静かな事実である。**
