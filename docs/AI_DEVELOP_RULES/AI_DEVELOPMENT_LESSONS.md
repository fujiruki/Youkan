# AI開発における再発防止と品質保証ガイドライン

## 背景
「解決したと思ったのにできていない」という事象の繰り返しを防ぐため、過去の失敗事例から得られた教訓をルール化する。
本ドキュメントは、機能修正やバグ対応を行うすべてのAIが参照すべき行動指針である。

## 1. 共通コンポーネント修正時の「影響範囲」の網羅
**教訓**: 
共通コンポーネント（Modal、Button、Utilitiesなど）を修正した際、特定の画面（報告があった画面）だけの確認で完了としてしまい、他の呼び出し元での不具合（Propの渡し漏れなど）を見落とす事例があった。

**アクション**:
- **全検索の徹底**: 共通コンポーネントを変更する際は、必ず `grep` や「参照の検索」を行い、**すべての呼び出し元**を特定する。
- **インターフェース整合性の確認**: Propを追加・変更した場合、すべての親コンポーネントが新しいI/Fに適合しているかコードレベルで確認する。

## 2. モック(Mock)テストの限界認識と実動作検証
**教訓**:
`vitest` 等の単体テストにおいて、子コンポーネントや外部依存をモック化しすぎた結果、「接続部分（配線）の不備」を検出できず、テストはPassするが動かないという事態が発生した。

**アクション**:
- **ブラウザ操作テストの活用**: UIの連携（親から子へのデータ渡し、イベント発火）に関わる修正は、コード上のテストだけでなく、**ブラウザサブエージェントによる実際のクリック/入力操作**を行って検証する。
- **Integrationテストの重視**: 可能な限り、実際のコンポーネント階層をごく一部でも結合した状態でテストする（Shallow Renderに頼りすぎない）。

## 3. データフローの全体俯瞰（対症療法の防止）
**教訓**:
「保存されない」等の事象に対し、ボタンのハンドラなど局所的な修正を繰り返したが、根本原因は画面ごとのデータ更新戦略（Optimistic Updateの有無など）の不統一にあった。

**アクション**:
- **ライフサイクルの整理**: 修正着手前に、対象データが「どこで生まれ（Fetch）、どこで保持され（State/Store）、どう保存され（API）、どう再取得されるか」というフローを整理する。
- **根本原因の仮説**: 「なぜ動かないか」だけでなく「なぜその修正で直るのか（他の画面も同じ仕組みか）」を自問し、システム全体の一貫性を保つ修正を行う。
