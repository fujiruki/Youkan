# ドキュメント整理ルール（Documentation Structure Rules）

本プロジェクトの `docs` フォルダは、以下の構造で管理します。
AIおよび人間がドキュメントを作成・配置する際は、このルールに従ってください。

---

## 1. フォルダ構造

```
docs/
├── 00_Vision/                     # ビジョン・企画・対象ユーザー
├── 01_User_Experience/            # 操作フロー・画面遷移・ユーザーストーリー
├── 02_Architecture/               # 技術設計・API・データ構造
├── 03_UI_Components/              # UIコンポーネント・日本語文言
├── 04_Meeting_Logs/               # AI会議ログ・議論記録
├── 05_Roadmap_and_Backlog/        # ロードマップ・今後やることリスト
├── 99_Archive/                    # 古いバージョン・凍結ドキュメント
└── AI_DEVELOP_RULES/              # 開発ルール（このファイルがある場所）
```

### 各フォルダの役割

| フォルダ名 | 役割 | 含まれるもの |
|---|---|---|
| `00_Vision` | **なぜ作るか、誰のためか** | 企画書、対象ユーザー像、ユーザーの生の声、ビジョン物語、INDEX.md（ナビゲーション） |
| `01_User_Experience` | **どう使うか** | 操作フロー、画面遷移図、ユーザーストーリー（シナリオ形式）、ステータス定義 |
| `02_Architecture` | **どう作るか（技術）** | システム構成図、レイヤー設計、API仕様、データスキーマ、プラグイン設計 |
| `03_UI_Components` | **見た目の詳細** | 日本語文言対応表、ボタン配置議論、UIコンポーネント仕様 |
| `04_Meeting_Logs` | **議論の記録** | AI会議ログ、意思決定の経緯 |
| `05_Roadmap_and_Backlog` | **今後の予定** | やりたいことリスト、バックログ、優先順位議論 |
| `99_Archive` | **歴史の保管庫** | 過去バージョン、凍結スナップショット、Handoverフォルダ |
| `AI_DEVELOP_RULES` | **開発ルール** | AI開発憲法、運用ルール、命名規約（このファイル） |

## 2. ファイル命名規約

### 2.1 基本ルール

| ルール | 説明 | 例 |
|---|---|---|
| **英語 + アンダースコア** | ファイル名は英語、単語はアンダースコアで区切る | `Screen_Flow.md`, `Target_User_Profile.md` |
| **日付プレフィックス（任意）** | 時系列が重要な場合は `YYYYMMDD_` で始める | `20260118_Meeting_Log.md` |
| **カテゴリプレフィックス** | 同一フォルダ内での分類に使う | `Story_`, `Spec_`, `Design_` |
| **日本語ファイル名は避ける** | 互換性のため英語名を推奨。日本語はフォルダ名のみ許容 | ❌ `企画書.md` → ⭕ `Planning_Document.md` |

### 2.2 推奨プレフィックス

| プレフィックス | 用途 | 例 |
|---|---|---|
| `Story_` | ユーザーストーリー、利用シナリオ | `Story_TomorrowPlanning.md` |
| `Spec_` | 仕様書、定義書 | `Spec_Task_Status.md` |
| `Design_` | 設計書 | `Design_Plugin_Architecture.md` |
| `Flow_` | フロー図、遷移図 | `Flow_Screen_Transition.md` |
| `Meeting_` | 会議ログ | `Meeting_20260118_FutureBoard.md` |
| `Raw_` | ユーザーの生の入力（編集しない） | `Raw_Voice_20260114.md` |

### 2.3 特殊ファイル

| ファイル名 | 役割 |
|---|---|
| `INDEX.md` | そのフォルダの目次・ナビゲーション |
| `README.md` | フォルダの概要説明（GitHubなどで自動表示される） |

## 3. ファイル内メタデータ（フロントマター）

すべてのMarkdownファイルの冒頭に、以下のYAMLフロントマターを記載します。

```yaml
---
created: YYYY-MM-DD       # 作成日
updated: YYYY-MM-DD       # 最終更新日
author: USER | AI         # 作成者（ユーザーの生の声は USER、AI生成は AI）
status: active | deprecated | archived  # ステータス
---
```

### ステータスの意味

| ステータス | 意味 |
|---|---|
| `active` | 現在有効。最新の情報。 |
| `deprecated` | 非推奨。新しいドキュメントに置き換わっているが、参照価値あり。 |
| `archived` | 歴史的資料。`99_Archive` に移動すべき。 |

---

## 4. ドキュメント作成時のチェックリスト

- [ ] 正しいフォルダに配置したか？
- [ ] ファイル名は英語 + アンダースコアか？
- [ ] フロントマター（created, updated, author, status）を記載したか？
- [ ] 関連する `INDEX.md` を更新したか？

---

## 5. 移行時の注意

既存のファイルを移動する際は、以下を確認してください。

1. **フロントマターを追加**する（なければ追加）。
2. **日本語ファイル名は英語にリネーム**する。
3. **古いドキュメントは `99_Archive` に移動**し、`status: archived` を設定する。
4. **重複がある場合は統合を検討**する。
